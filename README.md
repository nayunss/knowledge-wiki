# 📚 지식 위키 (LLM-Wiki + RAG)

AI가 작성한 마크다운을 **한 곳에 모아** 위키로 보고, RAG로도 쓰는 개인 지식 베이스 템플릿.

- **단일 원본**: `content/` 폴더의 `.md` 파일들 (`[[위키링크]]`로 연결)
- **웹 위키**: [Quartz](https://quartz.jzhao.xyz/)가 GitHub Pages로 자동 배포
- **RAG**: 같은 레포를 claude.ai Project에 연결하면 질의응답 가능
- **작성**: Claude Code 스킬로 "위키에 올려줘" 한마디 → 자동 커밋·배포 (`claude-skill/` 참고)

## 전체 구조

```
                        ┌─→ Quartz → <USER>.github.io/knowledge-wiki   (위키 뷰)
 GitHub 레포 (원본만) ──┤
   content/*.md [[링크]] └─→ claude.ai Project 연결                    (RAG 질의응답)
        ▲
        │ 작성/편집할 때만: 임시 clone → 쓰기 → push → 삭제
    Claude Code (wiki-note 스킬)
```

로컬 컴퓨터엔 상시 사본을 두지 않는다 — 원본은 GitHub 레포 하나, 작성하는 순간에만 임시 clone → push → 삭제.

## 시작하기

👉 **[SETUP.md](SETUP.md)** 를 따라 하세요. (비개발자: 클릭 방식 / 개발자: `setup.sh` 한 방)

## 노트 형식 — OKF 규약

노트는 [OKF(Open Knowledge Format)](https://github.com/GoogleCloudPlatform/knowledge-catalog/tree/main/okf)의 가벼운 규약을 따른다. OKF는 "md + YAML frontmatter를 벤더 중립 지식 포맷으로" 삼는 스펙으로, 이 위키의 설계 철학과 같다. 그중 가치 있는 규약만 차용했다.

**frontmatter**
```yaml
---
title: 표시 제목
type: 개념            # 개념 / 도구 / 설계결정 / 레퍼런스 / 플레이북 … (자유값, 서술적으로)
description: 한 줄 요약   # index·검색 스니펫·미리보기·RAG 근거에 쓰임
tags: [선택]
resource: https://…   # (선택) 특정 도구·레포·API를 다루는 노트면 정식 URL
---
```

**본문·구조**
- **구조적 마크다운 우선** — 제목·목록·표·코드블록이 프로즈보다 검색·에이전트 이해에 유리.
- 관련 노트는 `[[위키링크]]`로 연결(없는 노트 링크 = 아직 안 쓴 지식 stub, 허용).
- 외부 출처는 맨 아래 `## 출처` 섹션에 목록으로.
- **점진적 탐색**: 주제가 뭉치면 `content/<주제>/` 폴더로 묶고, 폴더에 `index.md`(설명 포함 목록)를 둬 랜딩으로 삼는다. 예: `content/ai-엔지니어링/`.

**의도적으로 안 쓴 OKF 요소** (우리가 이미 더 낫게 해결): `timestamp`·`log.md`(→ git 이력), 절대링크(→ `[[위키링크]]`가 파일명으로 해석돼 이동에 안전), 번들 `viz.html`(→ Quartz 그래프뷰).

> 이 규약은 `claude-skill/`의 wiki-note 스킬에 반영돼 있어, "위키에 올려줘"로 올리는 노트는 자동으로 이 형식을 갖춘다.

## 노트 작성 시 기존 내용 참고

로컬에 상시 사본이 없어도, "위키에 올려줘"를 요청하면 매번 레포를 clone하므로 **현재 위키의 모든 노트를 읽어** 참고한다.

- 매 작성 = `git clone`으로 **현재 `content/` 전체**를 임시로 받음 → 기존 노트를 읽고 `[[링크]]`로 연결하거나, 기존 노트를 편집하거나, 중복을 피한다.
- 즉 **작성 순간에 clone된 최신본이 곧 참고 자료**다. 끝나면 임시 폴더는 삭제(로컬에 사본 안 남음).

단, 참고하는 것은 **과거 이력(스냅샷)이 아니라 현재 최신 상태**다:

| 참고 대상 | 자동 |
|---|---|
| 지금 `content/`에 있는 모든 노트 | ✅ (매번 clone) |
| 그걸로 링크·편집·중복 회피 | ✅ |
| 삭제된 노트 / 특정 노트의 과거 버전 | ⚠️ 기본 X — 얕은 clone(`--depth 1`)은 커밋 이력 제외. 필요 시 전체 이력 clone으로 꺼냄 |

> 옛 버전이나 지운 노트를 되살려 참고해야 하면 "이력에서 꺼내줘"라고 요청하면 된다(평소엔 최신본만 받아 빠르게 처리).

## RAG로 쓰기 — claude.ai Project에 연결

내 노트를 근거로 Claude에게 질문하려면 레포를 claude.ai Project의 knowledge에 연결한다.

### 방법 A — GitHub 레포 연결 (자동 동기화, 가능하면 추천)
1. claude.ai → 왼쪽 **Projects** → 기존 프로젝트 열거나 **+ 새 프로젝트**
2. 프로젝트의 **Knowledge(지식)** 영역 → **Add content / Add from GitHub** 버튼
3. 처음이면 **GitHub 인증(Authorize)** 창 → 승인 → 레포 목록에서 본인 `knowledge-wiki` 선택
4. 연결되면 `content/`의 md가 knowledge로 들어오고, 이후 push하면 갱신됨

> GitHub 연결 메뉴는 플랜(Pro/Team)·계정 설정에 따라 **Settings → Connectors**에서 GitHub를 먼저 켜야 보이기도 한다. 메뉴가 아예 없으면 방법 B로.

### 방법 B — 수동 업로드 (연결 메뉴가 없을 때, 항상 됨)
1. 프로젝트 **Knowledge → "파일" 옆 `+`** → 파일 업로드
2. `content/`의 md 파일들을 선택해 올림
3. 단, 자동 동기화가 아니라 **노트를 추가/수정하면 그 파일을 다시 업로드**해야 반영됨

## 비용(토큰)·시간 감각

이 위키에는 두 개의 발행 경로가 있고 비용 차이가 크다. 검증이 필요 없는 짧은 메모까지 하네스로 돌리면 낭비다 — 경로를 골라 쓰자.

### 경로 1 — wiki-note (단순 업로드: "이거 위키에 올려줘")

clone → 작성 → index 링크 → push. **순수 증분 약 3–5k 토큰, 짧은 세션 기준 노트 1개 ≈ 5–15k.** 소요 1–2분 + 배포 1–2분.

### 경로 2 — wiki-post 하네스 (작성+검증+발행: "~에 대해 글 써서 위키에 올려줘")

에이전트 4개(작성 1 + 병렬 검증 2 + 최종 가독성 1, 전부 최상위 모델)가 돌아 품질을 사는 대신 비용이 크다. **실측 기준** (1,200–1,600단어 글):

| 단계 | 토큰(실측) | 시간(실측) |
|---|---|---|
| tech-writer 초안 작성 | 45–95k | 2–4분 |
| fact-checker 팩트·최신성 (웹 검색 18–20회) | 57–105k | 3–23분† |
| copy-editor 오탈자·윤문 | 52–70k | 1–27분† |
| readability-reviewer 최종 가독성 게이트 | 45–60k | 1–2분 |
| 오케스트레이션(인벤토리·병합·게이트·발행) | 10–25k | 3–5분 (배포 포함) |
| **소계 (수정 루프 없이)** | **약 210–350k** | **약 12–35분** |

† 원문 전문을 내려받아 전수 대조하거나 렌더 검증을 돌리면 길어진다. 병렬이라 벽시계 시간은 둘 중 긴 쪽. 가독성 게이트는 병합 뒤 순차 실행이라 소요가 더해진다.

**수정 루프(FAIL → 재작성 → 재검증)는 생각보다 비싸다.** 실측 **회당 +120–165k, +5–12분** (tech-writer 재호출 52–90k + fact-checker 재검증 71–73k). 초판 README의 "+50–80k"는 과소평가였다.

**실측 3편 모두 FAIL을 한 번씩 맞았다.** 즉 수정 루프는 예외가 아니라 **기본값으로 잡아야 한다**(2026-07-17에 이 비용을 줄이려는 시도를 넣었다 — 아래 참고):

| 글 | 소계 | 수정 루프 | 실제 합계 |
|---|---|---|---|
| SkillOpt 비판적 리뷰 | 211k | 124k (1회) | **약 335k** |
| 하네스 엔지니어링의 미래 | 216k | 163k (1회) | **약 379k** |
| AIDE² 비판적 리뷰 † | 327k | 126k (1회) | **약 475k** |
| TurboQuant/TurboVec 분석 ‡ | 약 300k | 0회 | **약 300k** |

† 최종 가독성 게이트를 포함한 첫 실측(서브에이전트 실측 합 530k = 위 453k + 별도 배경 리서치 77k, 오케스트레이션 추정 약 25k). 앞의 두 예시는 게이트 도입 전 수치라 각각 +약 45k로 보정해 읽어야 한다.

‡ **수정 루프 0회 첫 사례(2026-07-19).** 서브에이전트 실측 합 약 240k(작성 55k + 팩트 70k + 윤문 60k + 가독성 55k) + 리서치·오케스트레이션 약 60k. FAIL이 안 나 재작성·재검증(회당 약 126k)이 통째로 빠졌다 — 규칙 ⑥이 겨냥한 바로 그 비용이다.

- 요약: **하네스 1편 ≈ 375–475k 토큰, 20–40분.** 단순 업로드의 **약 30–95배 토큰**.
- **긴 1차 소스를 전수 대조하면 상단으로 간다.** AIDE² 편이 비싼 건 글이 길어서가 아니라, 원문 verbatim(약 30k)을 작성·검증·재검증이 각각 다시 읽었기 때문이다. 요약본 대신 원문을 넣는 선택의 값이자 대가다 — 그 덕에 요약본이 지웠던 비단조성 대목을 초안이 살렸다.
- 그 대가로 검증된 글을 얻는다. 4편 모두 실질적 오류를 걸러냈고, 그중 **셋은 틀린 수치가 아니라 맞는 수치 위의 틀린 해석**이었다(원문에 없는 인과, 좁은 사실의 과잉 일반화, 척도가 다른 수치의 비교). 이런 건 사람이 읽어서 잡기 가장 어려운 종류다.
- **수정 루프 비용을 줄이려는 시도 (2026-07-17) → 유효로 확정 (2026-07-19).** 해석 FAIL 3연속의 공통 구조는 "엣지"였다 — 사실 하나하나는 맞는데(AIDE² 편은 수치 22/22) 사실 **둘을 잇는 순간** 무너진다. 검증 쪽엔 이걸 겨냥한 규칙(⑧ 해석 검증)이 있었지만 작성 쪽엔 없어서, 같은 오류를 매번 만들고 매번 126k를 내고 잡았다. tech-writing에 대응 규칙("두 사실을 잇는 문장은 그 연결의 근거를 대라")을 미러링했다. 새 에이전트도 새 단계도 아니라 **규칙 한 줄**이라 비용 증가는 없다. **결과: 도입 후 6편 창에서 해석 FAIL 1편뿐(그마저 규칙 도입 직전 편), 수정 루프 0회.** 확정을 걸어둔 "비토론형·해석 고밀도 편"이 turbovec-벡터압축(단일 tech-writer, 개체 혼동·척도 다른 비교·인과 비약이 다 실재)으로 도착해 팩트 FAIL 0으로 버텼다 — ⑥ 유효 확정. 규칙 한 줄이 회당 약 126k의 해석 수정 루프를 없앤 셈. 회귀 계측은 계속(위키-하네스 창 상태 6편).
- 위키가 커져도 사이클 비용은 거의 일정 — 인벤토리는 frontmatter만 수집하고, 본문은 관련 노트만 읽는다.

### 경로 3 — deep-research 선행 리서치 (선택)

논문 한 편이 아니라 **지형 전체**를 다루는 글(예: "하네스 엔지니어링의 전망")은 앞단에 리서치 하네스를 붙일 수 있다. 팬아웃 검색 → 소스 fetch → 주장 추출 → 적대적 검증(3표 중 2표 반증 시 기각) → 종합.

**실측 1회: 에이전트 111개, 소스 28편, 주장 134개 추출 → 25개 검증 → 18확정·7기각. 약 3.2M 토큰, 약 40분.**

- **경로 2의 약 10배다.** 함부로 붙이지 말 것 — 주제가 "여러 출처를 교차 검증해야만 답이 나오는 것"일 때만 값을 한다.
- 산출된 사실 파일은 **1차 소스가 아니다.** 리서치도 틀린다(실측: 좁은 사실을 벤더 전반으로 일반화한 오류가 초안까지 갔고, 다른 편에선 척도가 다른 수치를 비교한 framing을 작가가 물려받아 FAIL이 났다). wiki-post의 fact-checker가 원문으로 되짚는 이유다.
- **그래서 리서치 프롬프트가 두 가지를 강제한다 (2026-07-17).** ⑴ **엣지 금지** — 리서치는 사실만 모으고, 사실 사이의 인과·비교·일반화는 만들지 않는다(출처가 다른 수치끼리 비교 금지). ⑵ **산출 파일 맨 위에 자기 경고 헤더**(`⚠️ 1차 소스 아님 · 여기 적힌 비교·인과·일반화는 가설`). 실측 대조군이 있다 — 경고를 단 근거 파일은 작가가 지켰고, 같은 경고가 없던 파일의 오류는 그대로 초안에 실렸다. 규칙을 오케스트레이터의 기억에 맡기는 것보다 **파일이 경고를 들고 다니게** 하는 쪽이 싸다.
- 기각된 주장이 남긴 공백은 글에 **"미측정"으로 정직하게 남긴다.** 검증 못 한 걸 아는 척하지 않는 게 이 경로의 값이다.


### 경로 4 — 최신성 스윕 (발행 뒤 유지보수)

글은 발행하고 끝이 아니다. 2주쯤 지나면 버전·개수·"현재/최신" 같은 시점 표기가 썩는다. **실측 1회 (2026-07-17, 34편 위키 기준):**

| 단계 | 비용 | 잡은 것 |
|---|---|---|
| 기계 스캔 (에이전트 0) | 약 40k, 5분 | 외부 링크 168개 생존, arXiv 32/32 실존, **제목↔URL 불일치 1건** |
| 위험군 10편 §1 최신성 (에이전트 5개 병렬) | 약 440k, 60분 | **6편 갱신** (버전 4건·개수 2건), 4편 변경 없음 |

- **먼저 기계로 훑어라.** 링크 생존·arXiv 실존·제목 대조·버전 문자열 추출은 에이전트 없이 된다. 40k로 오류 1건을 잡았고, 그게 어디에 에이전트를 붙일지도 알려줬다.
- **에이전트는 도구 노트에만 붙여라.** 개념·메타 노트는 외부 사실이 없어 검증 대상이 아니고, 자체 실험 기록은 웹으로 검증할 수 없다(UNVERIFIED가 정답). 34편 중 실제 위험군은 10편이었다.
- **썩는 건 판단이 아니라 시점 표기다.** 6편 중 논지가 무너진 건 0건. 사이트가 99 → 117개로 개편된 노트조차 논지 8개가 전부 재현됐다. 반대로 **"예고"가 가장 빨리 썩는다** — "다음 버전에 온다"고 쓴 건 이미 나와 있었다.
- **부정 주장은 의외로 오래 간다.** "이 옵션은 없다"·"지원하지 않는다"류가 2주 뒤에도 전부 참이었다. 발행 전 검증이 제대로 된 주장은 견딘다.
- **검증 리포트도 1차 소스가 아니다.** 세 번 갈렸다 — 공식 문서는 리포트의 권고와 다른 구조를 택했고, `breaking` 4건 중 2건은 "non-**breaking** spaces" 오탐이었으며(그대로 반영했으면 멀쩡한 판단을 틀리게 고칠 뻔했다), 수치는 노트가 적어둔 집계 방법으로 재현해야 리포트와 일치했다.

> 경로 선택: 짧은 메모 → **경로 1**. 출처가 분명한 주제(논문 1편, 도구 1개) → **경로 2**. 지형·전망·논쟁 → **경로 3 + 2**.
