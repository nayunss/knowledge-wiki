# 프로덕션 AI 에이전트의 기본 개념

O'Reilly 책 *AI Agents: The Definitive Guide*의 공개 코드 레포(챕터별 Jupyter 노트북 35개)를 읽고, 거기서 다루는 개념 지도를 **직접 재서술**한 노트다. 책 본문이 아니라 코드 노트북의 개념·사용 도구만 뽑아 정리했고, 등장하는 프레임워크·주장·수치는 문서 끝의 독립 출처로 따로 검증했다. 표현이 아니라 뼈대만 가져왔다.

한 문장으로 요약하면, 이 책의 12장은 **"LLM 한 번 호출"에서 시작해 "무너지지 않는 프로덕션 에이전트 시스템"까지 가는 길**을 순서대로 깐다. 그 길에서 반복해 나오는 교훈이 하나 있다.

> 에이전트의 신뢰성은 모델 품질이 아니라 **오케스트레이션과 제약**에서 나온다.

관련 개념은 [[ai-엔지니어링-4계층]]과 [[에이전틱-엔지니어링]], 그리고 [[agents-cli-lifecycle-sdlc]]에서 다른 각도로 다룬다.

## 큰 그림 — 네 국면

12장을 하나씩 보면 흩어져 보이지만, 실은 네 국면으로 묶인다.

| 국면 | 장 | 질문 |
|---|---|---|
| A. 기초와 패턴 | 1–4 | 에이전트란 무엇이고, 어떻게 추론·협업하게 만드나 |
| B. 계약과 거버넌스 | 5–6 | 프로토타입을 어떻게 신뢰할 수 있게, 안전하게 만드나 |
| C. 배포와 하드닝 | 7 | 실제 제품에서 어떻게 살아남게 하나 |
| D. 평가·메모리·비용·위협 | 8–12 | 어떻게 측정·기억·절약하고, 공격을 막나 |

## 국면 A — 기초와 패턴 (1–4장)

### 1장. LLM에서 에이전트로

출발점은 상태 없는 LLM 호출을 **도구를 쓰고 기억을 갖는 최소 루프**로 바꾸는 일이다. 모델에 도구를 바인딩하고, 모델이 "도구를 부를지 / 끝낼지"를 스스로 정하게 한 뒤, 라우터가 그 결정을 보고 도구 노드를 돌리거나 종료한다. 대화 상태는 메시지 누적으로 쌓고, 체크포인트와 스레드 ID로 재현 가능한 기억을 만든다. 손으로 짠 루프와 그래프 프레임워크(LangGraph) 버전을 나란히 놓아 "왜 프레임워크가 필요한가"를 보여준다.

### 2장. 아키텍처와 패턴

한 장에서 에이전트 설계의 표준 어휘를 훑는다.

- **CoT (Chain-of-Thought)** — 추론을 "설명 → 패턴 → 요약"으로 구조화하고, 실행은 샌드박스 REPL에 분리한다.
- **ReAct** — 행동 → 관측 → 답변을 루프로 돌리되, 원시 사고사슬을 노출하지 않는 읽기 좋은 트레이스를 남긴다.
- **ToT (Tree-of-Thought)** — 여러 후보를 생성하고, 더 강한 심판 모델이 고른 뒤, 조사→초안으로 넘어가는 다단계 파이프라인. 역할마다 다른 모델·온도를 쓴다.
- **HITL (Human-in-the-Loop)** — 인터럽트로 에이전트를 멈추고 사람이 승인·수정·피드백한 뒤 결정적으로 재개하는 여섯 가지 통제 패턴.
- **Swarm** — 동료 에이전트들이 서로에게 제어권을 넘기는(handoff) 협업, 기본 활성 에이전트 하나가 있다.
- **계층 팀 (Hierarchical teams)** — 감독자(supervisor)가 작업자에게 라우팅하고, 팀 자체를 서브그래프로 중첩한다.

여기서 두 축이 갈린다. **계획형**(ToT처럼 먼저 설계)과 **반응형**(ReAct처럼 하면서 판단), 그리고 **단일 에이전트**와 **멀티 에이전트**.

### 3장. 고급 추론과 확장 실행

품질을 "손으로 짠 보상"이 아니라 학습·탐색으로 끌어올리는 기법들.

- **RULER** — 여러 후보 궤적을 상대적으로 순위 매기는 LLM 심판(OpenPipe ART의 일부). 정답 라벨이나 수제 보상 함수 없이 "이 중 뭐가 더 나은가"만 물어 강화학습·선호 워크플로에 쓴다.
- **AB-MCTS / TreeQuest** — 몬테카를로 트리 탐색으로 답을 반복 개선한다. "더 깊이 파기(refine)"와 "더 넓게 다시 뽑기(regenerate)" 사이를 적응적으로 저울질하는 추론시(inference-time) 스케일링. (Sakana AI)
- **ART + RULER 강화학습** — 실제로 로컬 모델을 강화튜닝한다. 딱딱한 정답 검사와 부드러운 RULER 품질 신호를 섞어 보상으로 쓴다.

### 4장. 모델 능력과 최적화

감독자-작업자 계층 팀으로 리서치+문서작성 파이프라인을 캡스톤으로 조립한다. 2장 계층 팀에 컨텍스트 관리(토크나이저 기반 메시지 트리밍)를 더해, "능력 있는 모델을 어떻게 역할별로 배치하고 컨텍스트를 아끼나"를 보여준다.

## 국면 B — 계약과 거버넌스 (5–6장)

### 5장. 프로토타입에서 프로덕션으로

여기서 무게중심이 "잘 되는 데모"에서 "안 무너지는 시스템"으로 넘어간다.

- **Pydantic 계약** — 시스템 경계마다 타입을 계약으로 박아, 프로바이더별 JSON 차이나 타입 불일치를 하류에서 조용히 터지기 전에 경계에서 잡는다. 실패 시 자가 치유(그래프 refine 노드 또는 Instructor 자동 재시도)로 복구한다.
- **Lusser의 법칙 (곱셈 신뢰성 법칙)** — 직렬 시스템의 성공률은 각 단계 성공률의 **곱**이다. 그래서 에이전트를 길게 이으면 정확도가 빠르게 무너진다. 해법은 경계마다 검증을 넣는 것. 이건 로버트 루서가 1950년대 로켓 신뢰성에서 정립한 실제 공학 원리다.
- **MCP (Model Context Protocol)** — 도구를 에이전트 코드에서 분리하는 범용 어댑터. 서버가 도구를 노출하고 클라이언트가 소비하며, 거버넌스는 모델이 아니라 서버 경계에서 강제한다. (Anthropic, 2024)
- **deep agents** — 계획·서브에이전트·가상 파일시스템을 갖춘 장기 실행 에이전트 하네스(LangChain, Claude Code에서 영감).

### 6장. 안전한 실행과 도구 거버넌스

"어떤 도구가 도는가"와 "어떻게 도는가"를 통제한다.

- **A2A 거버넌스 층** — 에이전트와 도구 사이에 강제 층을 끼운다. 모든 외부 접근이 하나의 `governed_call()`(기본 거부, fail-closed)을 통과한다. 허용목록·예산 한도·인자 게이트·연결/실행 분리·사람 승인·감사 로그의 여섯 장치. (A2A는 Google이 만든 에이전트 간 프로토콜)
- **코드 모드 (programmatic tool calling)** — 도구를 한 번에 하나씩 부르는 대신, 모델이 여러 도구 호출을 오케스트레이션하는 파이썬 프로그램 하나를 짠다. 반복·정렬·필터 같은 제어 흐름이 코드에 살고, 중간 결과는 샌드박스에 남아 왕복이 줄고 싸진다.
- **Monty** — 그 코드를 가두는 경계. 파일시스템·네트워크·임포트·호스트 전역을 기본 거부하는 최소 파이썬 인터프리터(Rust 작성, Pydantic). 샌드박스가 없으면 프롬프트 인젝션이 곧 원격 코드 실행이 된다는 걸 보여준다.
- **E2B** — 에이전트가 생성한 코드를 격리된 클라우드 샌드박스(마이크로 VM)에서 돌린다. 세션마다 샌드박스 하나씩, 병렬 격리가 프로덕션 패턴.

## 국면 C — 배포와 하드닝 (7장)

"에이전트 감(感)"에서 "시스템 물리학"으로 넘어가는 장.

- **추론 물리학** — 콜드 스타트는 30–120초의 배포 세금(다운로드→VRAM 적재→첫 요청 워밍업)이고, 최소 레플리카를 0보다 크게 두어 완화한다. KV 캐시는 비싼 작업 메모리이고, **프리픽스 캐싱**(고정 시스템 프리픽스 + 가변 꼬리)으로 첫 토큰 지연(TTFT)을 2–5배 줄이지만 프리픽스가 바뀌면 무효화된다. 서빙 백엔드는 vLLM(PagedAttention)·SGLang(RadixAttention)·TGI(연속 배칭 구현 — 연속 배칭 자체는 Orca 논문이 원조)를 비교하되, 에이전트 코드는 OpenAI SDK로 백엔드 중립을 유지한다.
- **3단 폴백 (fallback)** — 모델 실패·드리프트를 견디는 구조화 출력 전략. ① 엄격 JSON 스키마 모드(모양은 강제하지만 판단은 아님) → ② Instructor + Pydantic 검증기(검증 실패 시 오류를 프롬프트에 주입해 재시도) → ③ 프롬프트 전용 + 정규화 함수. 정규화 맵을 검증기에 두어 모든 단계가 "High"를 "p1"로 바꾸게 한다. 그래도 모델들은 우선순위·감정 같은 주관적 필드에서 서로 다르게 판단하므로, 폴백은 파이프라인 전체로 시험해야 한다.
- **하드닝된 루프의 세 기둥** — <strong>체크포인팅</strong>(직렬화 가능한 세션을 Redis/Postgres에 저장, 크래시 복구), <strong>컨텍스트 프루닝</strong>(토큰 인식 "청소부"가 시스템 프롬프트와 최근 턴은 지키고 나머지는 요약), <strong>서킷 브레이커</strong>(예산·턴 퓨즈로 폭주 루프 차단). 매 턴 이 셋이 합쳐진다: 프루닝 → 호출 → 지표 갱신 → 브레이커 점검 → 체크포인트. 긴 작업은 블로킹 REST 대신 WebSocket/SSE로 스트리밍한다.

## 국면 D — 평가·메모리·비용·위협 (8–12장)

### 8장. 기초 평가와 관측

- **평가 하네스** — 시나리오 구성 → 제한 동시성 비동기 생성 → 배치 채점(LLM 심판 + 결정적 검사) → 위반 우선 리뷰 → CSV 내보내기. 좋은 시나리오는 페르소나·감정 상태·다중 턴 흐름·명시적 스트레스 차원(긴급·모호·좌절·지식 불일치·적대적 의도)을 담는다. CI/CD에서 임계값 게이트로 승격을 막는다.
- **적대적 평가** — 도구 예산·승인 검사·명시적 차단 같은 하드 제약을 런타임에 건 에이전트를, <strong>OWASP Top 10 for Agentic Applications (2026)</strong>의 **ASI02: 도구 오용·악용** 항목으로 시험한다(DeepTeam 사용). 핵심 명제는 다시 이것 — 신뢰성은 모델이 아니라 오케스트레이션·제약에서 온다.

### 9장. 맞춤·고급 평가

- **관측 ≠ 평가** — LangSmith·Langfuse에 궤적을 남기는 건 시작일 뿐, 그 위에 외부 평가 파이프라인을 따로 얹는다. 대표 트레이스를 뽑아 하드 검사·루브릭·트레이스 심판 세 신호로 채점하고, 치명적 실패는 재생 가능한 벤치마크(JSONL)로 승격해 모델 교체 전에 비교한다.
- **RULER의 상대 순위** — 트레이스별 점수를 보완해, 같은 시나리오의 여러 후보 답을 서로 견주어 약한 답을 골라낸다.
- **AgentVista** — 멀티모달·장기·도구 상호작용 209과제의 초난도 벤치마크(HKUST-NLP). 최상위 모델도 30%를 넘기기 어렵다는 점이 "에이전트는 아직 멀었다"를 정직하게 보여준다.

### 10장. 에이전트 메모리

- **메모리 분류학** — 단기(메시지 + 체크포인터 + 스레드 ID)와 장기(의미 기억=사실+임베딩, 일화 기억=작업/결과, 절차 기억=지침)로 나눈다. 체크포인터가 없으면 매 턴이 독립이라 이름조차 잊는다 — 지속성은 **명시적으로 설정**해야 한다.
- **거버넌스** — 메모리 네임스페이스는 모델이 아니라 **인증**에서 오고(조직·사용자·종류), 관리자 쓰기는 RBAC로, 저장은 암호화하며, 사용자가 자기 메모리를 보고 지울 수 있어야 한다(GDPR). 메모리 오염(poisoning) 방어도 포함.
- **지속성 세금** — 메모리 갱신을 매 턴이 아니라 간격·작별 정책으로 돌려 비용을 통제한다. 프로덕션에선 SQLite/Postgres 저장소로 바꾼다.

### 11장. 연산에서 비용으로

- **서빙 비용의 물리학** — 프리필 대 자기회귀 디코딩, 토큰당 KV 캐시 증가를 계산해 GPU 메모리 예산 안에서 최대 동시 요청 수와 $/토큰을 추정한다. LLM은 메모리 대역폭에 묶인다는 점, 하이브리드 어텐션과 MoE(총 파라미터=메모리 바닥, 활성 파라미터=연산 비용)를 함께 본다.
- **의미 캐싱 스웜** — 새 사실을 못 더하는 도구를 벌점 주고, 진전 없으면 강제로 종합하게 하는 진전 가드. 계획자의 전체 조건 맥락을 키로 하는 계층 캐시. 단, 선형 스캔은 안 커지니 벡터 인덱스(FAISS/pgvector)로, TTL 없으면 낡고, 테넌트 간 유출은 샤딩으로 막는다.
- **토폴로지가 비용을 정한다** — 비용을 모델 ID가 아니라 **아키텍처 행동**(단발/ReAct/감독자 멀티에이전트)의 함수로 모델링한다. 요청당 호출 수가 분산의 가장 큰 동인이고, 프리픽스 캐시 할인이 입력 토큰을 깎는다.

### 12장. 위협 모델링

- **LlamaFirewall** — 신뢰 맥락을 아는 계층형 프롬프트 보안 방화벽(Meta). 규칙 스캐너가 프롬프트 인젝션·유출·보이지 않는 유니코드·역할 조작을 잡고, 출처(신뢰 못 할 사용자 vs 개발자 vs 도구 출력)에 따라 공격성을 조절한다. 규칙을 피해가는 의역 공격은 의미 기반 LLM 심판 층이 잡되, 명백히 양성·악성인 입력은 규칙이 처리하고 **애매한 것만** 모델로 에스컬레이션해 호출을 아낀다.

## 관통하는 원칙

12장을 꿰뚫는 규칙들 — 도메인이 무엇이든 차용할 수 있다.

1. **신뢰성은 오케스트레이션과 제약에서 온다.** 더 좋은 모델이 아니라 더 좋은 경계·게이트·검증이 시스템을 세운다.
2. **계약을 경계마다.** Pydantic 스키마를 시스템 이음매마다 박아 실패를 하류가 아니라 그 자리에서 잡는다.
3. **거버넌스는 모델이 아니라 서버·게이트에서.** 허용목록·예산·승인·감사는 프롬프트가 아니라 코드로 강제한다.
4. **직렬로 이으면 곱으로 무너진다 (Lusser).** 그래서 단계마다 검증을 끼운다.
5. **비용은 토폴로지의 함수.** 아키텍처가 정해지면 비용의 형태도 정해진다.
6. **평가는 위반 우선.** 잘 된 걸 세지 말고 깨진 걸 먼저 본다.
7. **애매한 것만 비싼 판단으로.** 명백한 건 규칙으로 싸게, 불확실한 것만 LLM 심판으로.
