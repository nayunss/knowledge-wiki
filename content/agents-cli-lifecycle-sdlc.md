# agents-cli 라이프사이클과 AI 시대 SDLC

[[google-agents-cli]]의 8단계 개발 라이프사이클을 출발점으로, 이 구조가 사실은 **2026년 AI 소프트웨어 개발이 수렴하고 있는 새 SDLC의 참조 구현**임을 정리한다. 즉 "에이전트 배포 도구"의 라이프사이클이지만, 도메인이 에이전트가 아니어도 그 뼈대(스펙 앵커 → 자동 생성 → 평가 게이트 → 배포 → 관측 되먹임)는 그대로 차용할 수 있다.

여러 소스를 교차 검증했고, 프레임워크의 상당 부분은 구글 백서 *The New SDLC with Vibe Coding*(Addy Osmani·Shubham Saboo·Sokratis Kartakis, 2026-05)을 뼈대로 하되 표현이 아니라 개념만 가져와 자체 서술했으며, 사실 주장은 문서 끝의 독립 출처로 재검증했다.

## 한눈에

- **무게중심이 이동했다**: "코드를 쓰는 일"에서 "의도를 명세하고 결과를 검증하는 일"로. 생성은 대체로 풀렸고, 검증·판단·방향 설정이 새 craft다.
- **에이전트 = 모델 + 하네스**: 에이전트 실패의 대부분은 모델이 아니라 하네스(설정) 문제다.
- **개발자의 산출물은 코드가 아니라 코드를 만드는 시스템**(팩토리 모델)이다.
- **검증이 관문**: 결정론적 테스트 + 비결정론적 평가(eval)가 없으면, 프롬프트가 아무리 정교해도 그냥 바이브 코딩이다.
- agents-cli의 `scaffold → eval → deploy → observe` 무한 루프가 이 원리를 벤더 툴로 물리 구현한 것이다.

## agents-cli의 8단계 라이프사이클

핵심 아이디어: **"노트북에서 잘 돈다"에서 "프로덕션에서 살아있다"까지를 `scaffold → eval → deploy → observe` 네 동사의 무한 루프로 잇는다.** Phase 0(Spec)이 루프 전체의 앵커다 — eval 루브릭, scaffold 플래그, 안전 가드레일, 트레이스 속성이 전부 스펙에서 파생된다.

| # | 단계 | 하는 일 | CLI / 스킬 | 입·출력 |
|---|------|---------|-----------|---------|
| **0** | **Spec** | `.agents-cli-spec.md`에 도구·제약·성공 기준을 한 화면 분량으로 명세 | `workflow` 스킬 | 입력: Agent Garden 템플릿 → 출력: 마크다운 스펙 |
| **1** | **Scaffold** | 프로덕션 형태의 프로젝트(약 72개 파일: 에이전트 코드·테스트·eval 보일러플레이트·Terraform·CI/CD·매니페스트) 생성 | `scaffold create` | 입력: 스펙+플래그 → 출력: 완성된 프로젝트 디렉터리 |
| **2** | **Build** | 에이전트 본체 작성 — 모델·instruction·tools·`App` 래퍼(약 30줄) | `adk-code` 스킬 | 입력: ADK 프레임워크 → 출력: 동작하는 에이전트 코드 |
| **3** | **Orchestrate** | 한 에이전트가 팀으로 커지면 전문 에이전트들을 조합. 프로세스 간 호출은 A2A 프로토콜 | `adk-code` 스킬 | 입력: 에이전트 코드 → 출력: 멀티에이전트 아키텍처 |
| **4** | **Evaluate** | 배포 전에 데이터셋 대비 점수화. LLM 심판 + 루브릭 | `eval generate/grade/synthesize/compare/analyze/optimize` | 입력: 데이터셋 → 출력: eval 점수 · **5–10회+ 반복** |
| **5** | **Deploy** | Agent Runtime / Cloud Run / GKE로 배포 | `deploy` | 플래그: `--dry-run`, `--agent-identity`, `--iap` → 출력: 라이브 엔드포인트 |
| **6** | **Publish** | Gemini Enterprise 카탈로그에 등록(ADK/A2A 모드)해 발견 가능하게 | `publish` | 입력: 배포된 엔드포인트 → 출력: 카탈로그 항목 |
| **7** | **Observe** | Cloud Trace 스팬 + BigQuery 분석. **프로덕션 데이터가 다음 반복의 데이터셋으로 되먹임** | `observability` 스킬 | 입력: 실제 트래픽 → 출력: 트레이스·비용·회귀 |

**되먹임 루프가 핵심이다.** Phase 7(관측)의 프로덕션 관찰이 → Phase 4(평가)의 내일 데이터셋이 되고 → 수정·재평가·재배포로 이어진다. 선형 파이프라인이 아니라 순환이다.

권장 실천:
- 빈 화면에서 시작하지 말고 Agent Garden 템플릿부터.
- eval은 임계값 넘기까지 5–10회+ 반복을 기대할 것.
- 프로덕션 보안은 에이전트별 서비스 계정(`--agent-identity`) + IAP.
- scaffold 시점에 `--bq-analytics`를 켜서 관측 루프를 처음부터 닫아둘 것.

## 왜 이게 '새 SDLC'의 참조 구현인가

### 무게중심의 이동 — "생성은 풀렸고, 검증이 새 기술이다"

전통 SDLC와 AI 시대 SDLC의 결정적 차이는 속도가 아니라 **무게중심의 이동**이다.

> "Generation is largely solved. Verification, judgment, and direction are the new craft."
> (생성은 대체로 해결됐다. 검증·판단·방향 설정이 새로운 기술이다.)

인간의 역할이 **구현자**(implementer)에서 **오케스트레이터**(orchestrator)로 이동한다. 코드를 쓰는 대신, 스펙을 정밀화하고·에이전트 출력을 리뷰하고·아키텍처 트레이드오프를 판단한다. agents-cli가 Spec(0)을 루프의 앵커로 두고 Evaluate(4)를 배포 앞에 강제하는 설계가 바로 이 이동을 코드로 옮겨 놓았다.

### 바이브 코딩 ↔ 에이전틱 엔지니어링 스펙트럼

이분법이 아니라 **검증 엄격도의 스펙트럼**이다. AI를 쓰느냐가 아니라, 그 출력을 얼마나 구조·검증·인간 판단으로 감싸느냐가 위치를 정한다.

| 수준 | 특징 | 적합 |
|------|------|------|
| **Vibe Coding** | 즉흥 프롬프트, 최소 테스트 | 프로토타입·버리는 코드 |
| **Structured AI-Assisted** | 제약 있는 상세 프롬프트, 수동 테스트 | 기존 코드베이스 |
| **Agentic Engineering** | 정식 스펙·아키텍처 문서·메모리 파일·자동 테스트·CI/CD 게이트 | 프로덕션 |

> 결정적 통찰: **결정론적 테스트(tests)와 비결정론적 평가(evals) 둘 다 없으면, 프롬프트가 아무리 정교해도 그냥 바이브 코딩이다.** 테스트는 "이 입력에 이 출력"이라는 결정론적 부분을, eval은 "에이전트가 옳은 궤적을 밟고 옳은 도구를 골라 품질 기준을 넘겼나"라는 비결정론적 부분을 검증한다.

주말 프로토타입은 순수 바이브 코딩으로 충분하고, 금융 트랜잭션을 다루는 프로덕션 API에는 에이전틱 엔지니어링이 필요하다. 어디에 선을 그을지 아는 것이 곧 실력이다.

## 이 SDLC를 떠받치는 네 기둥

### 1) 하네스 — Agent = Model + Harness

새 모델이 나올 때마다 에이전트는 똑똑해지고, 문제가 터지면 모델부터 탓하기 쉽다. 반론은 **모델은 하나의 입력일 뿐이고, 실제 행동은 그 주위를 감싼 하네스(harness)가 지배한다**는 것이다. 하네스 = 모델을 둘러싼 발판이며 구성요소는:

- **지시·룰 파일**: `AGENTS.md`/`CLAUDE.md`/`GEMINI.md`, 스킬 파일, 서브에이전트 프롬프트 — 에이전트가 누구이고 무엇을 금지당하는지.
- **도구(Tools)**: 함수·MCP 서버·API + "언제 어떻게 호출하는지"를 알려주는 산문.
- **샌드박스/실행환경**: 코드가 실제로 도는 곳, 접근 가능/불가 범위.
- **오케스트레이션**: 서브에이전트 스폰, 모델 라우팅, 전문가 간 핸드오프 규칙.
- **가드레일/훅(Hooks)**: 특정 라이프사이클 지점(도구 호출 전, 파일 수정 후, 커밋 전)에 도는 결정론적 코드. "잊으면 안 되는데 자꾸 잊는 것"을 강제.
- **관측(Observability)**: 로그·트레이스·평가·비용/지연 계측. 없으면 에이전트가 잘하는지 조용히 표류하는지 알 길이 없다.

**핵심 명제: "대부분의 에이전트 실패는, 정직하게 뜯어보면 설정 실패다."** 빠진 도구, 모호한 규칙, 없는 가드레일, 노이즈로 꽉 찬 컨텍스트 창이 원인이지 모델이 아니다. 이건 마케팅이 아니라 벤치마크로 확인된다:

- LangChain 보고: **모델은 고정한 채 시스템 프롬프트·도구·미들웨어만 손봐서** Terminal Bench 2.0 점수를 52.8에서 66.5로(**+13.7점**) 끌어올림. ([LangChain — harness engineering](https://www.langchain.com/blog/improving-deep-agents-with-harness-engineering))
- 실패 사례를 뜯어보면 모델 생성이 아니라 하네스 결함(레포 컨텍스트 누락·취약한 도구·약한 검증기)에서 비롯된 게 많다는 진단도 있다 — "하네스 격차(harness gap)". ([arXiv 2606.06324 — Diagnosing and Repairing Harness Flaws](https://arxiv.org/pdf/2606.06324))

바이브 코딩과 에이전틱 엔지니어링의 차이도 결국 **하네스를 얼마나 의도적으로 구성하느냐**다 — 같은 에이전트로도 둘 다 가능하다. 우리 위키의 [[하네스-엔지니어링]] 관점과도 정확히 맞물린다.

### 2) 팩토리 모델 — 산출물은 "코드"가 아니라 "코드를 만드는 시스템"

공장장은 위젯을 손으로 조립하지 않는다. 조립 라인을 설계하고 품질관리를 건다. AI 시대 개발자도 같다 — 산출물은 코드가 아니라 **코드를 생산하는 시스템**이다:

- 무엇을 만들지 정의하는 스펙·컨텍스트
- 스펙을 구현으로 번역하는 에이전트
- 정확성을 검증하는 테스트·품질 게이트
- 실패를 에이전트로 되돌리는 피드백 루프
- 에이전트를 안전한 범위에 묶는 가드레일

성공은 단계별 지시가 아니라 **성공 기준을 주고 반복하게 두는 것**에서 나온다. 이게 agents-cli의 `scaffold → eval → deploy → observe` 루프가 물리적으로 구현하는 바다.

### 3) 컨텍스트 엔지니어링과 그 경제학

출력 품질은 영리한 프롬프트보다 **제공한 컨텍스트의 품질**에 달렸다. 컨텍스트는 6종 — **지시·지식·메모리·예시·도구·가드레일** — 이며, 규모에서는 이를 둘로 나누는 게 1급 아키텍처 결정이 된다:

- **정적 컨텍스트**: 항상 로드(시스템 지시, 룰 파일, 전역 메모리, 페르소나). 관련성과 무관하게 매 상호작용에 실리므로 **토큰이 비싸다**.
- **동적 컨텍스트**: 필요할 때만 로드(작업 매칭으로 트리거되는 스킬, 실행 중 도구 결과, RAG 문서). 필요할 때만 비용을 지불해 효율적.

이 트레이드오프를 규모에서 관리하는 핵심 패턴이 **Agent Skills** — 절차적 지식을 이식 가능한 패키지로 묶고, **점진적 공개**(progressive disclosure)로 시작 시엔 가벼운 메타데이터만, 작업이 매칭되면 전체 지시를, 필요할 때만 깊은 참조를 당겨온다. 덕분에 에이전트는 수십 개 전문능력을 지니면서도 지금 쓰는 하나의 토큰 비용만 낸다. (agents-cli가 SKILL.md 7개를 심는 방식이 바로 이것 → [[google-agents-cli]])

**이건 기술이자 재무 전략이다.** 리더의 관심사는 개발 속도가 아니라 **총소유비용**(TCO)이고, 두 방식의 비용 구조는 정반대다:

| | 바이브 코딩 | 에이전틱 엔지니어링 |
|---|---|---|
| **CapEx**(초기) | 거의 0 (구독료 + 즉흥 프롬프트) | 높음 (API 스키마·테스트 스위트·컨텍스트 구조 설계) |
| **OpEx**(운영) | **높고 복리로 누적** | 낮음 |

바이브 코딩의 숨은 OpEx:

- **토큰 소각률**: 비구조화 파일을 통째로 컨텍스트에 붓고, 검증 안 된 실수를 반복 수정시키는 "프롬프트 루프" — 낮은 1차 성공률로 토큰을 태운다.
- **유지보수세**: 구조 없는 AI 스파게티 코드를 6개월 뒤 역공학하느라 며칠.
- **보안 교정**: 평가 하네스 없이 코드를 빨리 뽑으면 취약점도 빨리 뽑힌다. 프로덕션에서 고치는 비용은 설계 단계의 지수배.

밀도 높은 고신호 페이로드(정밀한 `AGENTS.md` + 가드레일)를 주면 1차 성공률이 올라 시행착오 루프 비용이 사라진다. 한 걸음 더 나아간 것이 **지능형 모델 라우팅**이다 — 오타 수정·기본 유닛테스트까지 최대 프런티어 모델에 맡기지 말 것. 복잡한 작업(요구·아키텍처·초기 구현)엔 큰 모델, 결정론적 저복잡도 작업(테스트 생성·코드 리뷰·CI/CD 모니터링)엔 작고 빠르고 싼 모델로 라우팅해 품질을 유지하며 토큰 비용을 체계적으로 낮춘다.

### 4) 인간의 새 역할 — 컨덕터/오케스트레이터, 그리고 80% 문제

개발자는 두 모드를 유연하게 오간다:

- **컨덕터(Conductor)**: IDE에서 실시간으로, 코드가 나타나는 걸 보며 매 순간 지휘. 복잡한 로직·까다로운 디버깅·낯선 코드베이스에 적합. 위험: 개발자가 병목이 되어 AI 처리량 이득이 제한됨.
- **오케스트레이터(Orchestrator)**: 목표를 정의해 에이전트에 위임하고 결과를 리뷰. 백그라운드·병렬 실행. 잘 정의된 작업(버그 수정·패턴 기반 기능·마이그레이션·테스트 생성)에 적합. 요구 역량이 구문 숙련이 아니라 **명세·분해**(decomposition)·평가·시스템 설계로 바뀐다.

여기서 걸리는 벽이 **80% 문제**다. AI는 기능 코드의 약 80%를 빠르게 생성하지만, 남은 20%(엣지 케이스·에러 핸들링·통합 지점·미묘한 정확성 요구)에는 깊은 맥락 지식이 필요하다. 더 위험한 건 **에러의 성격 변화**다 — 단순 구문 오류에서 개념적 실패(잘못된 비즈니스 로직 가정, 모호함을 되묻지 않음, 놓친 엣지 케이스, 장기 유지보수 부담을 낳는 설계)로. "코드가 그럴듯해 보이고 기본 테스트도 통과"하기 때문에 더 잡기 어렵다.

정직한 데이터도 이를 뒷받침한다 — 초기 2025년 AI 도구 연구(METR RCT): 숙련 개발자가 익숙한 대형 저장소에서 AI를 썼을 때 스스로는 20% 빨라졌다고 느꼈지만 실제로는 **19% 느려졌다**(검증·디버깅·수정에 든 시간 때문). 단, METR는 2026년 2월 방법론적 한계를 인정했으므로 절대치보다 **체감 속도와 실제 속도가 다르다**는 방향성으로 읽는 게 안전하다. ([arXiv 2507.09089](https://arxiv.org/pdf/2507.09089)) 처방은 명확하다: AI는 잘하는 것(잘 명세된 작업의 빠른 구현)에 쓰고, 인간의 주의력은 AI가 약한 것(모호한 요구·아키텍처 트레이드오프·정확성 검증)에 아껴 둔다.

이 역할 이동은 자율성 스펙트럼(Assistance → Autonomy) 위에서 움직인다. **Assistance**(제안, 개발자 승인 필요) → **Semi-Autonomous**(정의된 작업을 인간 감독 하에 실행) → **Autonomous**(복잡한 SDLC 활동을 최소 개입으로 수행). 현재 대부분 도구는 assistance–semi 사이에 몰려 있고, 분류 축은 **의사결정 권한·작업 복잡도·감독 요구·실패 결과**다. 어느 수준이든 프로덕션 임계 결정에는 인간 감독이 필수이며, 핵심 리스크는 검증 없이 수용하는 **overtrust**(맹신)다.

---

## 페이즈별로 무엇이 바뀌나

업계 소스가 공통으로 그리는 전통 → AI-First 재편:

- **요구사항/기획**: 2–6주 → 1–2일. AI가 스펙·유저스토리·데이터모델·API 스키마 초안을 몇 시간 안에. 인간은 생성이 아니라 **검토·정련**. 요구사항이 문서 핸드오프가 아니라 인간-AI 대화가 되어 명세와 초기 구현을 동시에 낳는다.
- **설계**: AI가 와이어프레임·컴포넌트 생성. 단, **아키텍처 트레이드오프의 주인은 여전히 인간**. AI는 결정된 아키텍처를 구현하는 데 뛰어나다.
- **구현**: 코드의 60–80%를 AI가 생성(에러 핸들링·유닛테스트 포함). 인간은 쓰기에서 **리뷰·유도**로 전환.
- **테스트/QA**: **스펙에서 테스트를 코드보다 먼저 생성**(test-first 역전). 결과물 품질뿐 아니라 **궤적**(trajectory, 에이전트가 어떻게 도달했나)도 평가. 잘 쓴 eval 스위트가 AI에게 "옳음"을 정의해 주는 계약이 된다.
- **배포**: IaC 템플릿화, 환경 패리티 자동 강제, 롤백 트리거 처음부터 내장. 1–3주 → 1–3일.
- **유지보수/관측**: 반응형 → 능동형. **관측을 배포 스펙에 처음부터 넣는다**(observability as architecture). 손댈 수 없던 레거시가 탐색 가능해지는 게 가장 큰 이득.

> velocity 주장(전체 SDLC 6–12개월 → 6–12주, 10–20배 등)은 벤더·블로그성 수치라 감안해 읽을 것. 방향은 맞지만 절대치는 과장일 수 있다.

이 재편을 agents-cli 8단계에 다시 포개면 이렇게 대응한다:

| agents-cli 단계 | 일반 AI SDLC 대응 |
|---|---|
| Spec (0) | 스펙 주도 개발 (기계 판독 스펙) |
| Scaffold (1) | 프로젝트/IaC/CI-CD 템플릿 생성 |
| Build (2) / Orchestrate (3) | AI 구현 + 인간 리뷰·유도, 멀티에이전트 조합 |
| Evaluate (4) | Eval 게이트 + test-first (검증이 새 craft) |
| Deploy (5) / Publish (6) | 자동화 배포·롤백 내장·레지스트리 |
| Observe (7) → 4 | 관측을 아키텍처로, 프로덕션→데이터셋 되먹임 루프 |

## 그래서 무엇을 할까

### 바로 적용할 실천 원칙

1. **스펙을 앵커로.** 서술형 문서 말고 기계 판독 가능한 스펙(YAML/JSON schema/구조화 마크다운). → agents-cli Phase 0.
2. **에이전트 설정을 코드처럼.** `AGENTS.md`·프롬프트·eval을 버전관리하고 PR에서 리뷰.
3. **데모가 아니라 eval 스위트로 게이트.** 워크플로를 평가 통과 여부로 막는다(명시적 루브릭 필수). → Phase 4.
4. **Test-first를 기본값으로.** 스펙에서 테스트를 코드 전에 생성.
5. **관측을 아키텍처로.** 배포 첫날부터 모니터링·롤백 내장, 프로덕션 데이터를 다음 eval로 되먹임. → Phase 7에서 4로.
6. **하네스를 재사용 자산으로.** 프로젝트마다 다시 만들지 말고 공용 하네스 인프라로. → [[하네스-엔지니어링]]
7. **프로토타이핑(빠르게)과 프로덕션(규율 있게)을 명시적으로 구분.**
8. **컨텍스트 엔지니어링을 1급 실천으로.** 6종 컨텍스트를 정적/동적으로 관리하고 모델 라우팅으로 비용을 통제.

### 역할별 시작점

- **개인 개발자**: ① `AGENTS.md`를 10줄로 시작(스택·컨벤션·하드룰·워크플로), 에이전트가 실수할 때마다 규칙 한 줄 추가 ② 코딩 에이전트에 스킬셋(예: Agents CLI) 설치 ③ 반복 워크플로 하나를 첫 에이전트로 ④ **코드보다 테스트·eval 먼저** ⑤ 배포될 코드는 전 줄 리뷰(가짜 패키지·부실 에러 핸들링 의심) ⑥ 기본기(디버깅·시스템 설계) 유지.
- **엔지니어링 리더**: ① 컨텍스트 엔지니어링을 1급 실천으로(AGENTS.md·프롬프트·eval·스킬을 코드처럼 PR 리뷰·버전관리) ② **데모가 아니라 eval에 바를 건다**(명시적 루브릭 필수) ③ AI 코드용으로 코드리뷰 재설계 ④ 프로토타이핑/프로덕션 경계를 명시 ⑤ 하네스를 공용 자산으로 한 번 만들고 여러 번 개선.
- **조직**: ① AI 개발을 생산성 기능이 아니라 엔지니어링 투자로 ② 규모 전에 프로덕션 기반(CI의 궤적·최종응답 eval, 트레이스, 에이전트별 권한) 먼저 ③ 열린 표준(MCP·A2A) 채택 ④ 인간+에이전트 하이브리드 팀 전제 ⑤ 채용·육성을 구현이 아니라 **판단**(명세·평가·아키텍처) 중심으로.

### 세 가지 불변 원칙

1. **구조는 확장되고, 바이브는 안 된다**(Structure scales, vibes don't).
2. **AI는 기존 문화를 증폭한다** — 강점도 약점도 함께 곱한다.
3. **인간의 역할이 진화한다** — 명세·평가·아키텍처 판단·시스템 설계가 곧 craft.

---

## 한 줄 결론

agents-cli의 라이프사이클은 "에이전트 배포 도구"의 얼굴을 하고 있지만, 실제로는 **2026년 AI 개발이 수렴하는 SDLC(스펙 앵커 + eval 게이트 + 관측 되먹임 루프 + 하네스 위의 오케스트레이터로서의 인간)를 벤더 툴로 구현한 참조 구현**이다. 도메인이 에이전트가 아니어도 이 뼈대는 그대로 차용할 수 있다.
