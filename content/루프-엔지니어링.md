# 루프 엔지니어링 (Loop Engineering)

**같은 목표/프롬프트를 완료 조건에 닿을 때까지 반복 실행시키는 제어 흐름 설계.** 매 프롬프트를 손으로 쓰는 대신, 도구 호출 사이에 무엇을 할지·언제 자기 작업을 점검할지·언제 끝났다고 볼지를 설계한다. AI 엔지니어링 [[ai-엔지니어링-4계층]]에서 "시간" 축을 담당하며, 2026년 급부상한 신조어다.

> "에이전트에게 프롬프트하기를 멈춰라. 대신 당신을 대신해 프롬프트해줄 루프를 설계하라."

## 에이전트의 기본 루프

에이전트를 단발 챗봇과 가르는 단 하나의 메커니즘은 **Reason → Act → Observe → Repeat** 순환이다. 목표를 추론하고, 도구를 호출·실행하고, 결과를 관측하고, 목표와 대조해 자기수정하고, 완료까지 반복한다. 전통 파이프라인이 사람이 단계를 미리 고정하는 정적 흐름이라면, 에이전틱 루프는 모델이 **런타임에 다음 단계를 정하므로** 적응·자기수정이 가능하다.

## 대표 형태 세 가지

### (a) 인터벌/스케줄 루프
일정에 따라 작업을 찾아 처리하는 주기 작업. 폴링·감시·정기 태스크에 쓴다. 예: Claude Code `/loop`(지정 간격 반복, 간격 생략 시 모델이 스스로 페이스 조절), `/schedule`(cron 기반 클라우드 루틴). "배포를 5분마다 확인" 같은 유형.

### (b) 자기참조 루프 = Ralph 기법
가장 단순한 형태는 `while :; do cat PROMPT.md | claude-code ; done` — 무한 bash 루프가 고정된 프롬프트 파일을 매번 다시 읽어 넣고, 디스크의 코드를 고치고, 다시 돈다. Geoffrey Huntley가 2025년 창안·대중화했고, 이름은 심슨의 Ralph Wiggum에서 땄다. 정밀함보다 끈기, 정제보다 반복이다.
- 원리는 **반복을 통한 결과적 일관성(eventual consistency)** — 한 번에 완벽할 필요 없이 완료 조건까지 반복을 강제한다.
- **파일시스템이 메모리**다. 대화 이력 대신 코드베이스·TODO·git 히스토리에 상태를 남기고, 매 반복은 **새 컨텍스트**로 시작해 이전 반복의 아티팩트를 마주친다. "한 반복에 한 태스크만"이 유효 컨텍스트 상한을 지키는 결정적 제약이다.
- Anthropic이 `claude-code`에 `ralph-wiggum` 플러그인으로 공식화했다. bash 무한 루프 대신 **Stop hook**으로 종료를 가로채 같은 프롬프트를 세션 안에 되먹인다(`/ralph-loop "<prompt>" --max-iterations <n> --completion-promise "<text>"`).
- Huntley는 엔지니어 무용론을 명확히 부정한다 — "Engineers are still needed."

### (c) 평가-수정 루프
테스트/eval이 통과할 때까지 반복. 미완료 태스크 선택 → 구현 → 테스트·타입체크·린트·CI·(UI면) 브라우저 검증 → 통과 시 커밋 → 상태·학습 기록 → 컨텍스트 리셋 후 반복. 검증이 실패하면 넘어가지 않고 그 에러 출력을 다시 먹여 자기수정을 유발한다. "루프는 빨간불에서 멈춰야 한다."

## 종료 조건과 리스크

**종료 조건**은 겹쳐 쓴다: 완료 프라미스(정해진 문자열 출력) · 최대 반복(`--max-iterations`, 무한 루프 1차 안전장치) · 결정적 종료(컴파일러·린터·테스트 통과) · 정체 서킷브레이커(N 반복 새 커밋 없으면 중단, 시간 상한) · 예산 상한(Claude Agent SDK `maxTurns`·`maxBudgetUsd`) · 반복 호출 감지(같은 행동 2회 초과 시 중단).

**리스크와 가드**
- **루프맥싱(loopmaxxing)**: "무한 반복이 뭐든 풀어줄 것"이라는 함정. "UX를 개선하라" 같은 주관적 목표는 정지점이 없어 API 비용만 부푼다. → 목표를 이진(pass/fail)으로 검증 가능한 한 문장으로("모든 테스트 통과", "린터 0 에러").
- **비용 폭증·무한 루프·국소 최소값**(과감한 변경 대신 소수점 개선만 반복)·**이해 부채**(생성 속도가 리뷰 역량을 앞질러 유지불가 코드가 쌓임).
- **컴팩션 드리프트**: Huntley — "Compaction is the devil." 긴 실행 중 초기 지시가 자동 요약되며 목표에서 이탈한다. 지속 규칙은 프롬프트가 아니라 `CLAUDE.md`/`AGENTS.md`에 둔다.
- **격리**: 피처 브랜치에서만, 샌드박스(Docker/VM), 최소 권한 키, 쓰기는 승인 게이트.
- Osmani의 성숙 경로: ① 휴먼인더루프(모든 행동 승인) → ② 결정적 종료 → ③ 정체 서킷브레이커 → ④ 반복 LLM 작업을 컴파일된 코드로 전환.

## 언제 맞고 안 맞나

- **맞음**: 반복하면 수렴하고 **기계적으로 검증 가능한** 작업 — 마이그레이션·리팩터·의존성 업데이트·테스트 정리. "done"을 사전에 못박을 수 있고 그린필드(신규) 프로젝트에 강하다. 테스트 커버리지·문서화된 컨벤션·사람 감독이 있을 때.
- **안 맞음**: 레거시 코드베이스, 모호한 요구, 약한 테스트로 논리 오류를 못 잡는 경우, 창을 넘는 대형 스코프, 비즈니스 로직·UX 같은 주관적 목표.

## 컨텍스트·하네스와의 결합

- **컨텍스트 전략 두 갈래**: 누적형(한 세션에 쌓다가 한계 시 자동 compaction)과 리셋형(매 반복 메모리 리셋 — Ralph/자기개선 루프). 리셋형은 외부 메모리 4종(`AGENTS.md`·git 히스토리·`progress.txt`·`tasks.json`)으로 반복 간 학습을 복리로 쌓는다. → [[컨텍스트-엔지니어링]]
- **하네스 안의 루프, 루프 안의 팀**: [[하네스-엔지니어링]]이 도구·스킬·서브에이전트·훅의 구조를 정의하면, 루프가 그 구조를 시간축으로 구동하고, 한 반복 안에서 하네스가 서브에이전트 "팀"을 소집한다. 훅(`Stop`·`PreToolUse`)이 루프의 접합부다.

## 다른 층과의 관계

프롬프트(한 번의 입력) → 컨텍스트(한 반복의 정보) → 하네스(환경) 위에서, 루프는 그 전부를 **반복으로 구동**한다. Boris Cherny(Anthropic)의 표현으로 "루프는 소스코드에서 에이전트로의 도약만큼 크다." 전체 지도는 [[ai-엔지니어링-4계층]].
