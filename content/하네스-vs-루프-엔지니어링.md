에이전트 자동화를 설계하는 두 방식, **하네스(harness) 스킬**과 **루프 엔지니어링**을 비교한다. 이름은 비슷하게 "메타적"이지만, 다루는 축이 다르다. 한쪽은 **공간(누가·무엇을·어떻게 나눌까)**, 다른 쪽은 **시간(같은 일을 언제까지 반복할까)**을 설계한다.

## 한 줄 요약

- **하네스** = 일을 **여러 전문 에이전트와 스킬로 쪼개** 팀으로 조율하는 **구조 설계**.
- **루프 엔지니어링** = 하나의 작업을 **완료 조건까지 반복 실행**시키는 **제어 흐름 설계**.

> 하네스는 "팀을 짜는 일", 루프는 "될 때까지 돌리는 일"이다.

## 각각 무엇인가

### 하네스 (goal / harness 스킬)

도메인에 맞는 에이전트 팀과 스킬을 설계하는 **메타 스킬**. 핵심은 **분리**다.

- **에이전트(누가)** 정의는 `.claude/agents/`, **스킬(어떻게)** 정의는 `.claude/skills/`로 분리한다.
- 작업을 전문 영역으로 분해하고(분석·생성·검증·QA 등), 팀 토폴로지(파이프라인·팬아웃/팬인·생성-검증·감독자 등)를 고른다.
- **에이전트 팀**을 기본 실행 모드로 삼아, 팀원끼리 `SendMessage`·`TaskCreate`로 자체 조율한다.
- 오케스트레이터 스킬이 "누가 언제 어떤 순서로" 협업하는지 엮고, `CLAUDE.md`에 포인터를 남겨 다음 세션에서 재사용한다.
- **진화하는 시스템**: 실행 후 피드백을 스킬/에이전트/오케스트레이터에 반영하고 변경 이력을 기록한다.

한마디로 **빌드타임 아키텍처**다. 병렬성과 전문성으로 품질을 끌어올린다.

### 루프 엔지니어링

같은 작업을 **반복 실행**시키는 설계 개념. 대표 형태 두 가지:

- **인터벌/셀프페이스 루프** (`/loop`, `schedule`): 프롬프트나 슬래시 커맨드를 주기적으로 돌린다. 배포 감시, 상태 폴링, 반복 점검 같은 **지속·주기 작업**에 쓴다.
- **자기참조 루프** (Ralph Loop): Stop 훅으로 세션 종료를 가로채 **같은 프롬프트를 완료 조건까지 계속 되먹인다**. 매 반복이 이전 반복이 남긴 파일·git 이력을 보고 스스로 개선한다. `while true`를 세션 안에서 구현한 셈.

한마디로 **런타임 제어 흐름**이다. 지속성과 반복적 자기교정으로 완성도를 끌어올린다.

## 핵심 차이

| 축 | 하네스 | 루프 엔지니어링 |
|----|--------|----------------|
| **설계 대상** | 공간 — 일의 분할과 조율 | 시간 — 반복과 종료 |
| **작동 방식** | 여러 에이전트 병렬/협업 | (보통) 단일 흐름을 순차 반복 |
| **품질의 원천** | 전문성·병렬성·상호검증 | 지속성·반복적 자기교정 |
| **완료 판단** | 오케스트레이터가 산출물 종합 | 완료 조건/프라미스·최대 반복·인터벌 |
| **산출물** | 에이전트·스킬·오케스트레이터 정의 | 루프 실행(훅/스케줄/셀프페이스) |
| **재사용 단위** | 파일로 남는 팀 아키텍처 | 반복 규칙(프롬프트+종료조건) |
| **주 리스크** | 조율 오버헤드·과설계 | 무한 루프·발산·비용 폭증 |
| **비유** | 조직도를 짠다 | 될 때까지 돌린다 |

## 자주 하는 오해

### "goal은 목표 달성하면 멈추고, 루프는 계속/무한히 돈다?" — 아니다

**"멈추냐 계속 도냐"는 둘을 가르는 축이 아니다.** 루프 엔지니어링도 대부분 **종료 조건이 있다** — 완료 조건/프라미스, 최대 반복 횟수, 인터벌이 그것이다. 자기참조 루프(Ralph Loop)는 "테스트 통과까지 고쳐라"처럼 **완료 조건에 도달하면 멈추는** 형태다. 즉 "목표 달성 시 종료"는 goal 스킬만의 특징이 아니라 루프에도 그대로 있다.

이 오해에 걸리는 이유는 루프의 두 형태 중 **인터벌/셀프페이스 루프**(배포 감시, 상태 폴링 등)가 실제로 주기적으로 계속 돌아가는 지속 작업용이기 때문이다. 하지만 이는 루프의 한 형태일 뿐, 루프 전체의 정의가 아니다.

**진짜 차이 축은 공간 vs 시간이다:**

- **goal(하네스)** = 공간 설계. "누가, 무엇을, 어떤 순서로 협업하나"를 정의하는 빌드타임 팀 아키텍처.
- **루프** = 시간 설계. "몇 번, 언제까지 반복할지"를 정의하는 런타임 제어 흐름.

둘 다 끝나는 시점이 있을 수 있고, 축이 다르기 때문에 결합도 가능하다(아래 참고).

## 언제 무엇을 쓰나

**하네스를 쓴다** — 작업이 뚜렷이 다른 전문 영역으로 나뉘고(예: 분석 → 생성 → QA), 병렬 처리나 상호 검증이 품질에 직결될 때. 2명 이상이 협업할 가치가 있을 때.

**루프를 쓴다** — 한 목표가 명확하고 **한 번에 안 되지만 반복하면 수렴**할 때(테스트 통과까지 고쳐라), 또는 **주기적으로 계속 돌아야** 할 때(배포/PR 감시).

**과설계 경고** — 단발 작업에 하네스를 세우는 건 조직도부터 그리는 낭비다. 반대로, 병렬 전문성이 필요한 일을 루프 하나로 밀어붙이면 반복만 늘고 품질은 안 오른다.

## 함께 쓰기

둘은 배타적이지 않다. **축이 다르므로 겹쳐 쓸 수 있다.**

- **루프 안에 하네스**: 매 반복마다 하네스가 팀을 소집해 한 사이클을 처리하고, 루프가 완료 조건까지 이 사이클을 반복한다. (반복적 팀 작업)
- **하네스 안에 루프**: 오케스트레이터가 특정 Phase를 "될 때까지" 반복하도록 루프를 끼운다. 하네스의 **생성-검증 패턴**이나 Phase 6의 "반복 개선 루프"가 사실상 이 결합이다.

즉 하네스가 **한 사이클의 구조**를 정의하고, 루프가 **그 사이클을 몇 번 돌릴지**를 정의한다.

## 결론

- 문제가 **여러 전문성으로 쪼개지면** → 하네스.
- 문제가 **반복하면 수렴하거나 주기적으로 계속돼야 하면** → 루프.
- 대부분의 진지한 자동화는 결국 **둘의 결합**이다. 구조를 세우고(하네스), 될 때까지 돌린다(루프).
