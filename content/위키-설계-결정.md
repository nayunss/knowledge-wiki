## 왜 노션이 아니라 마크다운인가
노션은 데이터가 독자 포맷에 갇혀 위키·RAG로 자유롭게 꺼내기 어렵다.
**md 평문을 단일 원본으로 두면** 같은 파일을 여러 렌즈(위키/RAG/편집)로 바꿔 낄 수 있다 — 이게 파편화를 막는 핵심.

## 세 렌즈, 하나의 원본
- **원본**: GitHub 레포 `content/*.md` + `[[위키링크]]`
- **위키 뷰**: Quartz → GitHub Pages 자동 배포
- **RAG**: 같은 레포를 claude.ai Project에 연결

## 로컬에 상시 사본을 안 두는 이유
원본은 클라우드(GitHub) 하나면 충분하고, 커밋 이력이 곧 스냅샷이다.
작성/편집이 필요한 순간에만 **임시 clone → 작업 → push → 삭제**한다(이 노트도 그렇게 올라왔다).
디스크 절약이 목적이 아니라 "원본은 클라우드 하나, 컴퓨터는 깨끗하게"라는 원칙.

## RAG는 기성품부터
claude.ai Project는 파일 30MB·개수 무제한이고 컨텍스트 초과 시 자동 RAG로 전환된다.
md 노트는 개당 수 KB라 수천 개까지 이걸로 충분. 검색 정밀도가 아쉬워지면 그때 같은 폴더에 임베딩 스크립트를 얹는다(원본은 불변).

## 공유는 템플릿으로
git/GitHub만 각자 개인이 설정하고, 나머지(Quartz 설정·배포 워크플로우·작성 스킬)는 GitHub Template로 한 번에 복제되게 했다.

여기서 "작성 스킬"(임시 clone→작성→push→삭제로 노트를 올리는 Claude Code 스킬)은 에이전트 자동화의 한 형태다 — 이걸 설계·확장하는 관점은 [[ai-엔지니어링-4계층]] 참고.
