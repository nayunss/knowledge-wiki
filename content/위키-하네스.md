이 위키의 기술 글은 사람이 직접 쓰지 않는다. **에이전트 하네스**가 쓰고, 검증하고, 발행한다. 이 노트는 그 파이프라인의 구조와 사용법을 기록한다. 하네스라는 개념 자체는 [[하네스-엔지니어링]], 이 위키의 전체 설계는 [[위키-설계-결정]] 참고.

## 구조 — 4개 에이전트, 4개 스킬

```
"~에 대해 글 써서 위키에 올려줘"
        │
[선행 리서치] (선택) 원문 정독 / deep-research → 00_*_facts.md
        │   ⚠️ 이 파일은 1차 소스가 아니다 — 파일 맨 위에 자기 경고를 박고,
        │      fact-checker가 원문으로 재검증한다. 리서치는 사실만 모으고
        │      엣지(인과·비교·일반화)는 만들지 않는다
[tech-writer] 초안 작성 ─ tech-writing 스킬(작성 가이드)
        │ 01_writer_draft.md + (검증 필요) 마커
[fact-checker] ∥ [copy-editor]   ← 병렬, wiki-verify 스킬(검증 프로토콜)
  최신성·팩트        오탈자·윤문
        │ FAIL이면 tech-writer 수정 루프 (최대 2회)
[병합 게이트] validate-note.py ─ frontmatter·마커 잔존·깨진 강조를 기계적으로 차단
        │ 통과 전 다음 단계 금지
[readability-reviewer] 최종 가독성 게이트 ─ readability-review 스킬
  잔존 AI 티·용어 접근성·자연스러운 가독성을 독자 눈으로 판정
        │ FAIL이면 용어→tech-writer 보강 / 문체→copy-editor 재윤문 (최대 2회)
[발행] wiki-note 방식 ─ temp clone → OKF md → push → 자동 배포
```

| 구성요소 | 역할 |
|---|---|
| **tech-writer** (에이전트) | 테크에반젤리스트 10년차 + CTO 페르소나. 독자 우선·두괄식·트레이드오프 필수. 확신 낮은 대목엔 `(검증 필요)` 마커를 남긴다 |
| **fact-checker** (에이전트) | 작성일 기준 최신성 + 반증 지향 팩트체크(웹 검색). 마커 최우선 검증, 출처 등급 A(공식)>B(블로그)>C(커뮤니티) |
| **copy-editor** (에이전트) | 오탈자 교정 + AI 티 윤문(humanize-korean 활용). 의미·수치·링크 불가침, 변경률 30% 가드 |
| **readability-reviewer** (에이전트) | 발행 직전 병합본을 독자 눈으로 통독해 잔존 AI 티·용어 접근성·자연스러운 가독성을 **판정**(고치지 않음). copy-editor가 초안을 *윤문*한다면 이쪽은 최종본을 *검수*하는 최후 게이트 |
| **tech-writing** (스킬) | 글의 뼈대(후킹 도입→핵심 주장→본문→트레이드오프→실무 적용), 문장 규칙, OKF 형식. **연결의 근거** 규칙(인과·비교·일반화는 별개 주장) — wiki-verify ⑧의 작성 쪽 대응물. **집계 방법 병기** 규칙(직접 센 수치엔 재현 명령을 남긴다) |
| **wiki-verify** (스킬) | 4종 검증(§최신성·§팩트·§오탈자·§윤문)의 기준과 PASS/FAIL 리포트 형식, 수정 루프 상한 |
| **readability-review** (스킬) | 최종 가독성 3축(§AI 티·§용어 접근성·§자연스러움) 판정 기준, 과검열 금지 원칙, 리포트 형식 |
| **wiki-post** (스킬) | 오케스트레이터 — 위 전부를 순서대로 엮고 실패를 처리한다 |
| **validate-note.py** (게이트) | 발행 전 결정적 검증 — OKF frontmatter, 마커 잔존, 메타주석, 코드펜스 짝, 깨진 마크다운 강조를 기계 차단 |

## 설계 결정과 이유

- **작성과 검증의 분리.** 쓴 사람이 검증하면 자기 확증에 빠진다. fact-checker는 주장을 "확인"하지 않고 "반박"하려 시도하며, 반박에 실패한 주장만 통과시킨다.
- **검증 2종은 병렬.** 팩트(내용)와 윤문(문체)은 서로 간섭하지 않으므로 동시에 돈다. 같은 문장을 둘 다 고치면 팩트 수정이 우선한다.
- **수정 루프에 상한(2회).** 무한 루프 방지. 2회 후에도 FAIL이면 발행을 중단하고 쟁점을 사람에게 넘긴다 — 강행은 사람의 승인이 필요하다.
- **발행은 재구현하지 않는다.** 기존 wiki-note 절차(temp clone→push→삭제, 로컬 무잔류)를 그대로 쓴다.
- **병합도 작업이므로 검증한다.** 에이전트 산출물을 사람이/오케스트레이터가 병합하는 단계가 유일한 무검증 구간이었다 — [[에이전트-평가-evals]]의 MAST '작업 검증 실패' 원칙을 적용해 결정적 게이트를 발행 앞에 세웠다.
- **읽기 경험은 기계 게이트가 못 본다.** validate-note.py는 마커·frontmatter 같은 *결함*을 막지만, 발행본을 처음부터 끝까지 독자 눈으로 읽는 사람은 파이프라인에 없었다. 병렬 윤문은 라인 단위라 통독 인상(문단 흐름, 잔존 AI 티, 대상 독자가 걸리는 용어)을 놓친다. 그래서 병합 뒤·발행 앞에 readability-reviewer를 최후 게이트로 세웠다 — 고치는 게 아니라 *판정*하고, FAIL이면 용어는 tech-writer·문체는 copy-editor로 되돌린다. 과검열(정착 용어를 취향으로 풀어쓰라는 지적)로 발행을 막는 것은 잔존 AI 티보다 나쁘므로, 애매한 지적은 참고로만 남긴다.
- **근거 파일도 검증 대상이다.** 오케스트레이터가 원문에서 뽑은 사실 파일(`00_*_facts.md`)이나 선행 리서치 산출물은 **1차 소스가 아니다.** 실제로 오케스트레이터의 추출 오류가 초안까지 전파된 적이 있고, 초안만 봐서는 잡히지 않았다 — 병합 단계가 그랬듯 이 구간도 무검증이었다. fact-checker가 원문으로 되짚는다.
- **경고를 단 근거 파일은 지켜지고, 안 단 파일은 전파된다.** ⑦은 *검증* 규칙이라 오류를 하류에서 잡을 뿐, 애초에 하류가 근거 파일을 어떻게 대하는지는 규정하지 않았다. AIDE² 편이 우연히 대조군이 됐다 — 오케스트레이터가 `⚠️ 1차 소스 아님, 전부 가설, 그대로 옮겨 쓰지 마라`를 맨 위에 박은 `00_critique_leads.md`는 작가가 그대로 지켰고, 같은 경고가 없던 `00_research.md`의 척도 비교는 작가가 **framing으로 물려받아** 초안까지 갔다(그게 그 편의 FAIL이다). 그래서 리서치 프롬프트에 둘을 넣는다: ⑴ **엣지를 만들지 마라** — 리서치는 사실을 모으고, 사실 사이의 인과·비교·일반화는 작가가 원문 근거로 만든다(특히 출처가 다른 수치끼리 비교 금지), ⑵ **산출 파일 맨 위에 자기 경고를 박아라.** 규칙을 사람의 기억에 맡기지 않고 파일 자신이 들고 다니게 하는 쪽이 싸다.
- **수치가 맞아도 해석은 틀릴 수 있다.** 검증에서 반증된 지적의 대부분은 틀린 수치가 아니라 **맞는 수치 위에 얹은 틀린 해석**(원문에 없는 인과·과잉 일반화)이었다. 그래서 §2 팩트 검증은 수치 전수 대조와 **별개로** 인과·비교·일반화를 반증한다.
- **노드는 검증되는데 엣지가 뚫린다 — 그래서 예방도 엣지에 건다.** 해석 FAIL이 세 편 연속 나고서야 공통 구조가 보였다. 사실 하나하나(노드)는 통과하는데(AIDE² 편은 수치 22건 전수 일치) **사실 둘을 잇는 순간**(엣지 — 인과 A→B, 일반화 A⊆B, 비교 A>B) 무너진다. tech-writing의 기존 규칙("모든 주장에 근거")이 안 문 이유가 여기 있다 — 양 끝 수치엔 근거가 있었고, 없던 건 *둘을 이어도 된다는 근거*였다. 규칙이 노드 수준에서 읽히고 실패는 엣지에서 났다. 검증 쪽엔 ⑧이 있는데 작성 쪽엔 대응물이 없어서, 같은 오류를 매번 새로 만들고 매번 잡았다(회당 약 126k, 사이클의 약 27%). 그래서 ⑧을 tech-writing에 미러링했다 — 탐지기를 더 붙이는 대신 이미 있는 규칙을 반대편에 복사하는 쪽. **근거는 실측이다**: AIDE² 편에서 오케스트레이터가 척도 비교를 콕 집어 경고한 곳은 초안이 지켰고(분리 각주까지 달았다), 경고하지 않은 곳에서 똑같이 넘어졌다. 능력이 아니라 커버리지 문제이니 일반 규칙이 통할 여지가 있다. **다만 과교정이 FAIL보다 나쁘다** — 해석은 이 위키의 값이고, 작가가 몸을 사려 전부 헤지하면 검증은 통과하고 글은 죽는다. 그래서 규칙은 "조심해라"가 아니라 "두 사실이 만나는 자리" 하나만 겨냥한다. 다음 2~3편에서 해석 FAIL이 줄지 않으면 규칙이 아니라 구조 문제다.
- **수치는 값이 아니라 방법과 함께 늙는다.** 발행 2주 뒤 전 위키 최신성 스윕을 돌리고 알았다. 논지는 거의 안 썩는데(사이트가 99 → 117개로 통째로 개편된 [[kinetics-뜯어보기]]조차 논지 8개가 전부 재현됐다) **직접 센 수치는 전부 썩는다.** 문제는 다시 세는 게 아니라 *같은 자로* 다시 세는 것이었다 — 집계 범위를 적어둔 수치는 재집계에 성공했고, 안 적은 수치는 세는 사람마다 값이 갈려(`ease` 82 vs 89 vs 101) 완화 표기로 물러나야 했다. 그래서 직접 센 수치엔 재현 명령·범위를 함께 남긴다. 1차 출처가 발표한 수치(논문 p값 등)는 해당 없다 — 거기선 출처 링크가 곧 방법이다.
- **하네스는 진화한다.** 실행에서 발견된 문제는 에이전트 정의에 반영하고 CLAUDE.md 변경 이력에 남긴다. 지금까지의 진화: ① copy-editor "대기 금지"(첫 실행에서 하위 작업 대기 중 정지), ② 병합 게이트·수치 전수 검사·UNVERIFIED 발행 규칙([[에이전트-평가-evals]] 원칙 자기적용), ③ 마크다운 렌더 검증·근사 표기 규칙(발행물에서 깨진 `**`·`~N` 발견), ④ 상류 근거 파일 검증·해석 검증(수치는 맞고 해석이 틀린 FAIL이 연속 발생), ⑤ 최종 가독성 게이트(readability-reviewer) 추가 — 윤문·병합을 통과하고도 발행본에 AI 티·난해 용어·비자연 문장이 남았고, 통독하는 검수자가 파이프라인에 없었다, ⑥ tech-writing에 '연결의 근거' 규칙 추가 — 해석 FAIL 3연속의 공통 구조가 '엣지'였고, 검증(⑧)에만 있던 대응물을 작성 쪽에 미러링했다, ⑦ 리서치 프롬프트에 엣지 금지·자기 경고 헤더 의무화 — 경고를 단 근거 파일은 지켜지고 안 단 파일은 framing째로 전파된다는 게 실측으로 드러남, ⑧ 직접 센 수치에 집계 방법 병기 — 전 위키 최신성 스윕에서 방법을 적어둔 수치만 재집계에 성공했다.

## 사용법

| 하고 싶은 것 | 말하는 법 | 무엇이 도나 |
|---|---|---|
| 글 작성부터 발행까지 | "~에 대해 글 써서 위키에 올려줘" | wiki-post 전체 파이프라인 |
| 완성된 노트 업로드만 | "이거 위키에 올려줘" | wiki-note (검증 없이 빠르게) |
| 부분 재실행 | "검증만 다시", "윤문만 다시", "가독성/최종 검수만 다시" | 해당 에이전트만 재호출 |

## 실전 기록 — 검증이 실제로 잡은 것

| 글 | 검증이 잡은 것 |
|---|---|
| [[에이전트-평가-evals]] | 리서치 단계의 수치 오류(MAST 표본 200+ → 150+) |
| [[skillopt-비판적-리뷰]] | 수치 20개 중 오류는 1건뿐이었으나, **원문에 없는 인과 3건**이 FAIL. 오케스트레이터의 사실 추출 오류가 초안까지 전파된 경로도 여기서 드러났다 |
| [[하네스의-미래]] | "벤더는 하네스라는 말을 쓰지 않는다"가 **반증**됨(Anthropic이 자사 SDK를 "general-purpose agent harness"로 부름). 좁은 사실을 넓게 일반화한 전형 |
| [[aide2-재귀적-자기개선]] | 수치 22건 전수 대조는 **불일치 0건**이었는데 FAIL은 또 해석에서 났다 — 척도가 다른 두 벤치마크(SWE-bench 해결률 %p vs MLE-Bench 페어드 델타)를 한 표에 놓고 대소를 비교. 초안이 같은 지면에서 "같은 자로 재면 안 된다"는 원칙을 스스로 세워놓고 어겼다 |

네 번 다 검증이 실질적 오류를 걸러냈다. 주목할 것은 **뒤 셋의 FAIL이 모두 수치가 아니라 해석에서 났다**는 점이다 — 그럴듯한 해석일수록 위험했다.

AIDE² 편은 규칙 ④(상류 근거 파일 검증)가 **설계대로** 작동한 첫 사례다. 그 전까지 상류 오류가 잡힌 건 오케스트레이터가 프롬프트에 우연히 적어준 덕이었는데, 이번엔 fact-checker가 규칙에 따라 리서치 산출물까지 되짚어 FAIL의 출처를 `00_research.md`로 특정했다. 규칙이 사람의 운을 대체한 지점이다.

## 종속 문서 동기화

이 노트와 레포 README의 하네스·비용 섹션은 **하네스 구현에 종속된 문서**다. 에이전트·스킬·게이트가 바뀌면 같은 커밋에서 이 문서들도 갱신한다 — 코드와 문서의 드리프트가 곧 신뢰 붕괴이기 때문. 변경 자체는 CLAUDE.md 변경 이력에 기록된다.
