# 프롬프트 엔지니어링 (Prompt Engineering)

모델 가중치를 건드리지 않고 **입력 텍스트(지시·예시·형식·역할)를 설계해** 생성형 AI의 출력 품질을 끌어올리는 실천. AI 엔지니어링 [[ai-엔지니어링-4계층]]의 가장 밑바닥 층으로, "한 번의 입력"을 다룬다.

## 기원과 왜 '엔지니어링'인가

- 학술적으로는 Reynolds & McDonell(2021)에서 용어가 등장했고, 2022년 GPT-3 위에서 앱을 만들던 개발자들이 "단어를 조금만 바꿔도 출력이 크게 달라진다"를 발견하며 일반 용어로 퍼졌다.
- 초기엔 "prompt whisperer" 같은 신비주의(art)에 가까웠다. 이후 문서화된 패턴·내장 평가(eval)를 갖춘 **측정 가능하고 반복적인 공학 규율**로 자리 잡으며 이름값을 했다. 즉 프롬프트를 코드처럼 다루고, 엣지 케이스로 테스트하고, 실패를 근거로 고친다.
- 프론티어 모델에서도 잘 설계된 프롬프트는 파라미터를 전혀 안 건드리고 태스크 성능을 크게 올린다(보고에 따라 40–70%). 모델이 아니라 입력 설계가 지렛대라는 뜻이다.

## 핵심 기법

| 기법 | 한 줄 설명 |
|---|---|
| Zero-shot | 예시 없이 지시만으로 수행 |
| Few-shot | 입력–출력 예시 몇 개로 패턴을 학습(in-context learning) |
| Chain-of-Thought (CoT) | "단계별로 생각하라"로 중간 추론을 유도, 복잡 문제를 분해 |
| Self-consistency | 여러 추론 경로를 뽑아 다수결로 최종 답 선택 |
| ReAct | 추론(reason)과 도구 호출(act)을 번갈아 — 에이전트의 원형 |
| Role / System 프롬프트 | 역할·페르소나·전역 규칙을 부여해 톤·제약 고정 |
| 출력 형식 강제 | JSON·스키마 등으로 파싱 가능한 구조 산출 |
| 디컴포지션 | 거대 태스크를 독립된 작은 하위 태스크로 분할 |
| Prompt chaining | 앞 프롬프트 출력을 다음 입력으로 잇는 파이프라인 |
| Tree of Thoughts | 여러 해법 가지를 탐색·평가하는 CoT 트리 확장 |
| Reflexion / self-refine | 자기 출력을 스스로 평가해 반복 개선 |

CoT는 어떤 추론 벤치에서 정확도를 17.7%에서 78.7%까지 끌어올린 사례가 보고됐다.

## 언제 강하고 어디서 부족한가

- **강함**: 단발·경계가 명확한 태스크(single-turn, bounded). 2022–2024년 LLM 품질 개선의 주 지렛대였다.
- **부족**: 여러 턴에 걸친 복잡·장기(long-horizon) 작업. 제어 단위가 프롬프트(한 번의 입력)를 넘어 세션(컨텍스트)·시스템(하네스)으로 올라간다.
- 방증: GPT-3.5를 단발 프롬프트로 쓰면 어느 코딩 벤치에서 48.1%인데, 같은 모델을 반복 에이전트 워크플로로 감싸면 95.1%까지 오른다. 성능은 "마법의 단어"보다 워크플로·컨텍스트 설계에서 나온다.
- 다 밀어넣기도 답이 아니다 — 긴 컨텍스트의 중간 정보가 무시되는 "lost in the middle", 창이 차기 전부터 품질이 떨어지는 "context rot" 문제가 있다(→ [[컨텍스트-엔지니어링]]).

## 흔한 안티패턴

- **취약성·과적합**: 특정 사례에 맞추면 다른 상황에서 무너진다.
- **예시 오염**: 현재 지시와 모순되는 낡은 예시가 남아 조용히 실패한다.
- **지시 쌓기**: 대략 8–10개를 넘는 개별 지시가 쌓이면 어느 하나에 대한 주의가 침식된다.
- **형식 취약성**: 예시가 JSON일 때만 JSON을 뱉으면 견고하지 않다 — 다른 입력으로도 테스트.
- **거대 단일 프롬프트**: 분류·추출·갱신·작성을 한 번에 시키면 추론이 저하 → 디컴포지션.
- 처방은 하나다. 프롬프트를 코드처럼 다뤄 "사용자가 아무 말 안 하면?", "데이터가 불완전하면?"으로 테스트하고 실패를 근거로 고친다.

## "프롬프트 엔지니어링은 죽었나?" (2025–2026)

- 2025년 6월 Andrej Karpathy와 Tobi Lütke(Shopify CEO)가 **컨텍스트 엔지니어링**을 후계 개념으로 대중화했고, Gartner도 "context in, prompt out" 식으로 전환을 선언했다. 초점이 "더 좋은 문장 쓰기"에서 "더 좋은 입력을 조립하기"로 이동했다.
- 그러나 죽은 게 아니라 **포섭됐다.** Anthropic은 컨텍스트 엔지니어링을 "프롬프트 엔지니어링의 자연스러운 진화"로 규정한다 — 프롬프트는 이제 더 큰 컨텍스트 조립 파이프라인의 부분집합이다.
- 에이전트 시대에도 프롬프트 설계는 살아 있다. 오히려 **도구 설명(tool description)을 프롬프트 엔지니어링하는 것**이 가장 효과적인 개선법 중 하나다 — Claude Sonnet 3.5는 도구 설명을 정밀화해 SWE-bench Verified에서 SOTA를 찍었다. 단, 자율 루프에서는 경직된 few-shot 예시가 역효과를 낼 수 있어, 예시보다 원칙·휴리스틱을 주는 "개념적" 프롬프팅이 권장된다.

## 다른 층과의 관계

- **프롬프트**(이 노트): 한 번의 입력을 설계. → 그 위에 [[컨텍스트-엔지니어링]](무엇을 보게 할지), [[하네스-엔지니어링]](모델을 둘러싼 환경), [[루프-엔지니어링]](반복 실행)이 쌓인다. 전체 지도는 [[ai-엔지니어링-4계층]].
