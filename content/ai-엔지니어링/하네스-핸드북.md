에이전트 하네스를 한 번이라도 고쳐본 사람은 안다. 정작 시간을 잡아먹는 건 코드를 바꾸는 순간이 아니다. "이 행위를 바꾸려면 대체 어느 파일, 어느 함수, 어느 실행 단계를 건드려야 하지?"를 찾는 그 앞 단계다. 프롬프트 조립, 상태 관리, 툴 호출, 실행 조율이 뒤엉킨 프로덕션 하네스에서 하나의 행위는 여러 곳에 흩어져 있다. Tencent HY LLM Frontier 팀 등이 2026년 7월 내놓은 논문 [Harness Handbook](https://arxiv.org/abs/2607.13285)은 바로 이 지점을 정면으로 겨냥한다.

## 이 논문의 진짜 기여는 문제를 다시 정의한 것

핵심부터 말하면, 이 논문의 가장 큰 기여는 새 도구가 아니라 **문제의 이름**이다. 논문은 하네스 수정의 병목을 **behavior localization(행위 지역화)** — 목표 행위를 구현한 모든 코드 위치를 찾는 일 — 로 정의하고, 편집을 "생성"하는 것보다 "어디를 편집할지 결정"하는 것이 그만큼 중요하다고 못 박는다(§5). 이 재정의가 왜 중요한지는 곧 설명한다. 방법론(코드와 동기화된 행위 중심 문서 트리)과 실험(두 오픈소스 하네스에서 계획 품질 상승 + 토큰 비용 하락)은 이 정의를 뒷받침하는 근거다.

에이전트 빌더에게 이 재정의는 실무의 무게중심을 옮긴다. 우리는 그동안 "모델이 좋은 편집을 만들어내는가"에 집중했다. 하지만 [[하네스-엔지니어링]]의 관점에서 보면, 편집 품질은 **정확한 위치를 먼저 짚었을 때만** 의미가 있다. 엉뚱한 파일을 고치는 완벽한 diff는 그냥 버그다.

## 왜 지역화가 어려운가 — 행위와 코드는 축이 다르다

논문이 짚는 어려움의 근원은 간단하다. **수정 요청은 "무엇을 해야 하는가"(행위)로 기술되는데, 레포는 파일·함수·모듈로 조직된다**(§2). 이 둘은 서로 다른 축이다. 그래서 누군가는 매번 "행위 → 코드" 매핑을 손으로 복원해야 한다.

여기서 논문은 기존 도구들의 한계를 정확히 짚는다. 레포 맵, 코드 검색, 코드 요약, 레포 메모리, 롱컨텍스트 편집 — 이들은 코드를 "보기 쉽게"는 만든다. 하지만 조각들이 어떻게 **하나의 행위**를 이루는지, 그리고 관련 위치를 **다 찾았는지**는 알려주지 않는다. 이 빠진 연결이 논문이 겨냥한 갭이다.

프로덕션 하네스가 크고 강결합이며 행위가 파일·함수·실행 단계·상태 전이에 분산돼 있을수록 이 문제는 심해진다. 키워드 검색으로는 안 걸리는 위치, 여러 파일에 걸친 end-to-end 기능이 특히 그렇다.

## Harness Handbook의 구조 — 코드에 링크된 3층 행위 트리

논문의 해법은 **behavior-centric representation**(행위 중심 표현)이다. 정적 프로그램 분석과 LLM 보조 구조화를 결합해 자동 합성하며, 각 행위를 대응 소스코드에 링크한다. 구조는 L1–L3 문서 트리 D와 이를 보완하는 상태 레지스터 뷰 Z로 이뤄진다(Figure 1).

| 층위 | 담는 것 |
|------|---------|
| **L1 System Overview** | 아키텍처, 실행 모델, 주요 단계, 시스템 목표, 전역 데이터 흐름 |
| **L2 Component Overview** | 단계별 책임·목표, 입출력, 의존성·상호작용, 로컬 상태 |
| **L3 Unit Deep Dive** | 소스에 grounded된 구현 항목 — 각 단계를 **실제 코드 위치에 연결**. 내부 로직·상태 전이·엣지 케이스 |
| **Z (상태 레지스터 뷰)** | 단계 경계를 넘는 상태 관계 기록 |

이 트리를 지탱하는 두 규칙이 이 논문에서 가장 실무적으로 배울 지점이다.

첫째, **progressive disclosure(점진적 공개)**. 작업이 더 상세를 요구할 때만 L1에서 L3로 내려간다. 처음부터 전부 펼치지 않는다 — [[컨텍스트-엔지니어링]]의 핵심 원리와 같은 발상이다.

둘째, **behavior–implementation alignment(행위-구현 정합)**. 활성 L3 로케이터(locator, 코드 위치 포인터)는 반드시 **현재 레포에 resolve돼야** 한다. 재검증이 안 되면 그 항목은 **frozen(동결)** 처리되어, 다시 갱신되기 전까지 지역화 후보에서 빠진다. 이 설계의 철학은 한 문장으로 요약된다: **레포가 구현의 authority로 남는다**. 문서가 코드보다 오래돼 거짓말을 하기 시작하면 차라리 입을 다물게 만드는 것이다.

### 두 가지 leaf 모드

L3 항목의 단위는 하네스 사정에 따라 둘 중 하나다.

- **function-as-leaf:** L3 항목이 함수(또는 연속 영역) 단위. 신뢰할 seed skeleton(실행 단계 뼈대의 초기 시드)이 있고 예산에 맞을 때. 논문 실험에서 **Terminus-2** 하네스가 이 모드를 쓴다.
- **file-as-leaf:** L3 항목이 파일 단위. seed가 없거나 함수 단위가 예산을 초과할 때, 레포 스케일에서 skeleton을 추론. 실험에서 **Codex** 하네스가 이 모드를 쓴다.

## 어떻게 만들고, 어떻게 다시 맞추는가

핸드북 구축은 3단계 파이프라인이다(Figure 2).

1. **Phase I — Static Fact Extraction.** 언어별 어댑터가 레포를 파싱해 함수·명명된 외부 경계·소스 위치·시그니처·call edge를 추출하고 프로그램 그래프 G를 만든다. 내부 함수나 명명 경계로 resolve되는 호출만 남기고, **미해결 호출은 추측하지 않고 로그로 남긴다.** 이 단계는 결정론적이며 LLM 호출이 없다.
2. **Phase II — Behavioral Organization.** 소스 유닛을 실행-단계 skeleton S로 매핑한다. LLM이 함수→단계 할당을 propose하고 반복 review로 정제하는 루프다.
3. **Phase III — Hierarchical Synthesis & Packaging.** 단계 skeleton과 소스 조직을 L1–L3 트리 + 상태 레지스터 뷰로 변환한다. 각 L3는 정적으로 식별된 소스 위치에 링크되고 현재 레포에 대해 검증된다.

수정이 실제로 일어나는 워크플로는 Algorithm 1이다. 여기서 지역화를 담당하는 것이 **BGPD**(Behavior-Guided Progressive Disclosure)다. coarse에서 fine으로 좁혀 간다 — 실행 단계 레벨에서 시작해 L1·L2로 요청에 직접 관련된 단계를 식별하고, Z를 따라 **공유 상태로 결합된 단계까지** 포함한다(구조적으로 멀어도). 그다음 그 단계 안에서 가장 관련된 L3 항목을 골라 소스 로케이터를 회수하고, call 관계로 후보를 확장한다. 마지막에 현재 레포를 열어 로케이터를 resolve하고, 여전히 관련된 site만 **검증된 evidence**로 남긴다. 여기서 외부 경계 노드는 문맥만 제공하고 편집 대상으로는 반환되지 않는다.

지역화가 끝나면 planner가 evidence로 편집 계획 P와 액션 선언 Γ(modify/add/remove)를 만들고, 별도 executor가 레포에 적용한다. **비어있지 않은 diff가 나오면 핸드북 resync가 트리거된다** — 가능하면 영향받은 부분만. 재동기화에서도 파싱·분류가 불가한 것은 동결하거나 커버리지 레코드로 **보수적으로 처리한다(추측 금지)**. 이 "모르면 추측하지 않고 동결한다"는 태도가 이 시스템 전체를 관통하는 원칙이다.

## 실험 — 약한 planner가 강한 모델의 위치 정확도에 근접한다

논문의 실험 설계는 깔끔하다. 평가 대상은 수정 워크플로의 **지역화 + 편집 계획**이다(§4.1). planner는 read-only로, NexAU로 구축하고 DeepSeek-V4-Pro로 구동한다. 두 arm을 비교하되 — Baseline(레포 직접 탐색)과 Handbook-Assisted(핸드북 안내를 받는 BGPD 내비게이션) — **핸드북 접근·내비게이션을 제외한 모든 조건(요청·모델·레포·툴 권한·디코딩)이 동일하다.** 이 통제가 결과 해석의 신뢰를 지탱한다. 판정은 GPT-5.5, Opus 4.8, DeepSeek-V4-Pro 세 judge가 독립으로 맡는다([[에이전트-평가-evals]]의 LLM-as-judge 방식).

벤치마크는 오픈소스 하네스 두 종 — Terminus-2(function-as-leaf)와 Codex(file-as-leaf) — 각각 30개의 행위 기반 수정 요청이다. 요청은 Query(위치를 안 알려주고 기존 행위 수정), Cross-file(파일·모듈을 넘는 기능 추가), Search-Hostile(키워드 검색으로 찾기 힘든 위치) 세 유형에 Easy/Medium/Hard 난이도가 붙는다.

> 참고: 아래 수치에서 Terminus-2와 Codex는 **서로 다른 하네스**다. 구조도 난이도도 다르므로 둘의 값을 서로 비교하지 않는다. 각 하네스 안에서 Baseline 대비 Handbook의 변화만 읽는 것이 옳다. 마찬가지로 Opus 4.8 reference와 GPT-5.5 reference는 **서로 다른 두 개의 독립 측정 기준**이라 우열을 매기지 않는다 — 논문도 순위 없이 병렬 제시한다.

### RQ1: 계획은 좋아지고 토큰은 줄었다 (Figure 3)

계획 품질은 judge 3종이 Localization·Scope Control·Reasoning 3차원으로 채점하고, 가중 점수 **S = 0.5·S_Loc + 0.25·S_Scope + 0.25·S_Reason**(0–100)로 합산한다. 두 arm의 점수차가 δ=3점 이상이면 승자를 가리고, win rate는 승 / 유효 비교 수다.

각 하네스 안에서, Handbook-Assisted가 Baseline보다 이겼다.

| 하네스 | Baseline win rate | Handbook win rate | 요청당 토큰 |
|--------|------------------|-------------------|-------------|
| Codex (file-as-leaf) | 28.3% | **38.3%** (+10.0pp) | 0.102M → **0.089M** (−12.7%) |
| Terminus-2 (function-as-leaf) | 26.7% | **45.6%** (+18.9pp) | 0.058M → **0.053M** (−8.6%) |

주목할 연결은 **win rate가 오르는 동시에 토큰이 줄었다**는 점이다. 논문은 이것으로 개선이 "토큰 예산을 더 써서 얻은 것이 아님"을 논증한다 — 계획이 좋아졌는데 비용까지 내려갔으니, 원인을 예산 증가로 돌릴 수 없다. 방향성도 특정 judge에 기대지 않는다: per-judge로 봐도 Codex는 세 judge 모두 10.0pp gap, Terminus-2는 13.3–26.7pp gap으로 전부 양수다(Figure 3b).

### RQ2: 위치 정확도가 강한 모델에 근접한다 (Table 1)

이 논문에서 가장 인상적인 결과다. 예측한 편집 계획을 Opus 4.8과 GPT-5.5가 각각 만든 독립 reference plan과 대조해 Recall·Precision·F1·Wrong(reference와 겹침이 0인 요청 비율, 낮을수록 좋음)을 잰다.

**양 하네스·양 reference·양 granularity에 걸친 24개 Recall/Precision/F1 비교가 전부** Handbook-Assisted에서 높았다(F1 gain 5.0–18.8pt). 중요한 건 **Recall과 Precision이 함께 올랐다**는 것 — 후보를 남발해 Recall만 끌어올린 게 아니라는 뜻이다. 완전 실패를 뜻하는 Wrong은 늘어난 적이 없고 최대 25.9pt 하락했다.

절대 일치도가 가장 높았던 건 Terminus-2다. 약한 planner(DeepSeek-V4-Pro)가 핸드북의 도움만으로:

- file-level F1이 **84.7%**(Opus 4.8 reference 기준), **89.3%**(GPT-5.5 reference 기준)
- symbol-level F1이 **77.1%**(Opus 4.8 reference 기준), **89.3%**(GPT-5.5 reference 기준)
- GPT-5.5 reference에 대해 Precision이 양 granularity 모두 **93.3%**

에반젤리스트의 시각에서 이 결과의 의미는 크다. **위치를 잘 구조화해 건네주면, 상대적으로 약한 planner도 강한 모델이 짚을 법한 곳을 짚는다.** 지역화 병목이 풀리면 모델 체급이 덜 중요해진다는 신호다. 이것이 앞서 재정의가 왜 중요한지에 대한 실증적 답이다.

### RQ3: 요청 유형·난이도 전반에서 일관 (Figure 5)

개선은 특정 조건에만 몰리지 않았다. 유형별로 6개 하네스×유형 비교가 전부 Handbook 우세(gain 16.3–33.3pp), 난이도별로도 6개 비교 전부 양수(3.7–33.3pp)다. 다만 gain이 난이도에 단조롭게 커지거나 작아지지는 않는다 — 난이도 라벨만으로 변동을 설명할 수 없다는 뜻이라, 논문도 여기서 과한 일반화를 하지 않는다.

## 트레이드오프 — 무엇을 얻고 무엇을 감수하는가

날 선 결과일수록 경계를 분명히 해야 한다. 이 논문이 **측정한 것과 측정하지 않은 것**을 구분하자.

**평가는 계획 품질까지다.** 실험이 채점한 것은 지역화와 편집 계획의 품질이지, 편집을 실제로 적용해 병합한 최종 결과가 아니다. planner도 read-only다. "좋은 위치와 좋은 계획"이 "실제로 동작하는 병합"으로 이어지는 마지막 구간은 이 실험의 채점 대상 밖이다.

**reference 자체가 모델이 만든 것이다.** 위치 정확도의 ground truth는 사람이 아니라 Opus 4.8·GPT-5.5가 생성한 reference plan이다. 판정 judge도 LLM이다. 따라서 "정답에 근접했다"는 곧 "강한 모델들의 판단에 근접했다"는 뜻으로 읽는 게 정확하다 — 사람 정답과의 일치는 이 설계에서 측정되지 않는다.

**동결은 안전장치이자 커버리지 구멍이다.** 로케이터가 현재 레포에 resolve되지 않으면 항목이 frozen되어 지역화에서 빠진다. 코드가 빠르게 바뀌는 구간에서는 재검증 전까지 그 행위가 핸드북의 사각지대가 된다. 거짓 안내를 막는 대가로 일시적 미커버를 감수하는 설계다 — 논문이 "추측 금지"를 택한 이상 이건 버그가 아니라 의도다.

**구축·재동기화에 LLM 비용이 든다.** Phase I은 결정론적이지만 Phase II·III과 resync는 propose-review 루프로 LLM을 부른다(resync 내 모델 호출은 4개 semantic step으로 제한). 즉 핸드북은 공짜로 유지되지 않는다. 하네스가 거의 안 바뀌거나 충분히 작아 사람이 머릿속에 다 담을 수 있다면, 이 표현을 세우고 동기화하는 비용이 이득을 넘어설 수 있다.

## 실무 관점 — 언제 이걸 떠올릴까

CTO 시각에서 이 논문을 도입 판단으로 번역하면 이렇다.

- **하네스가 크고 자주 바뀌며 여러 사람이 손대는가?** 그렇다면 "행위 → 코드" 매핑을 매번 사람 머리로 복원하는 비용이 실재한다. 이 논문이 파는 것은 편집 자동화가 아니라 **그 매핑을 코드와 동기화된 상태로 유지하는 아이디어**다.
- **당장 이 시스템을 붙일 게 아니어도 원리는 가져갈 수 있다.** progressive disclosure(필요할 때만 상세를 펼침), "레포가 authority"(문서가 코드보다 오래되면 동결), "미해결은 추측 말고 로그·동결" — 이 세 원칙은 어떤 코드 문서화·에이전트 컨텍스트 설계에도 적용된다.
- **확장 용도를 눈여겨보라.** 코드와 동기화된 행위 중심 표현은 behavior auditing이나 regression-impact 분석에도 쓸 수 있다고 논문은 말한다(§5). 지역화를 위해 만든 자산이 감사·영향 분석으로 재활용되는 그림이다.

논문이 예고한 다음 단계는 **harness self-evolving**이다. 핸드북을 공유 behavioral memory로 삼아 에이전트가 localization→planning→execution→resync 루프를 자율적으로 닫으며 하네스를 스스로 개선하는 방향 — [[하네스-자기개선]]과 [[하네스의-미래]]가 그리는 궤적과 같은 곳을 향한다. 다만 [[aide2-재귀적-자기개선]]류의 자기개선 논의가 늘 그렇듯, "루프를 자율로 닫는다"는 야심과 위 트레이드오프(모델 기반 평가, 동결 사각지대) 사이의 거리는 아직 실험으로 좁혀지지 않았다.

## 메타 훅 — 이 위키를 만든 하네스도 같은 문제를 겪는다

한 가지 유비로 마무리하자. 이 글을 낳은 파이프라인([[위키-하네스]])도 CLAUDE.md 변경 이력 표와 에이전트·스킬 정의로 이뤄진, **손으로 유지하는 일종의 behavior handbook**이다. "copy-editor의 대기 금지 지침을 어디에 적었더라", "해석 검증 규칙은 어느 스킬에 있지"를 찾는 순간이 바로 이 논문이 말하는 지역화다.

물론 층위는 다르다. 논문은 프롬프트·상태·툴이 얽힌 **코드 하네스**를 다루고, 이 위키의 하네스는 마크다운 스킬·에이전트 정의다. 둘을 같은 것으로 볼 수는 없다. 그래도 문제의 형태는 닮았다 — 행위는 "무엇을 해야 하는가"로 말하는데 시스템은 파일로 조직돼 있다는, 그 축의 어긋남. 그래서 [[ai-엔지니어링-4계층]]에서 하네스를 하나의 층으로 떼어 이름 붙인 일이 의미가 있다. 이름이 붙은 문제는 풀 수 있다. 이 논문이 한 일이 정확히 그것이다.
