"모델이 충분히 좋아지면 하네스는 사라진다." 에이전트를 만드는 사람이라면 한 번쯤 들어봤을 전망이다. 루프도, 컴팩션도, 툴 호출 재시도도 결국 모델이 알아서 하게 될 테니, 지금 우리가 짜는 배관 코드는 과도기의 유물이라는 이야기다.

절반은 맞다. 그리고 나머지 절반이 틀린 방식이 흥미롭다.

## 결론부터

하네스는 실제로 흡수되고 있다. 다만 **모델 가중치가 아니라 벤더의 API와 플랫폼으로** 흡수된다. 이 둘은 전혀 다른 명제다. 그리고 흡수되는 것에는 일관된 패턴이 있다 — 벤더가 가져가는 건 예외 없이 **메커니즘**(컨테이너, 요약 알고리즘, 루프)이고, 벤더가 명시적으로 애플리케이션에 남겨두는 건 **정책과 상태 소유권**(무엇을 언제 자를지, 무엇을 어디에 저장하고 누가 감사할지)이다.

용어부터 정리하자. 이 글에서 [[하네스-엔지니어링|하네스]](harness)란 모델을 감싸 실제로 일하게 만드는 모든 것 — 실행 루프, 툴 정의, 컨텍스트 관리, 샌드박스, 검증 게이트 — 을 가리킨다. `Agent = Model + Harness`의 그 하네스다.

---

## 1. 흡수는 실재한다

먼저 반대편 주장을 최대한 강하게 세워보자. 하네스가 벤더 쪽으로 빨려 들어가고 있다는 증거는 빈약하지 않다. 오히려 넘친다.

**OpenAI는 실행환경 자체를 API 안으로 넣었다.** Responses API에 shell 툴과 호스팅 격리 컨테이너(Debian 12, `/mnt/data` 작업공간, Python·Node·Java 사전설치)가 들어갔다. OpenAI의 표현이 노골적이다 — "개발자가 각자 실행환경을 만들게 하는 대신, 우리가 Responses API에 컴퓨터 환경을 갖추는 데 필요한 구성요소를 만들었다." 컴팩션(대화 이력을 압축해 컨텍스트를 비우는 작업)도 같은 API 안에 들어갔고, Agent Skills 포맷(`SKILL.md` + 리소스 폴더)도 API 프리미티브로 채택됐다.

**Anthropic은 컨텍스트 관리를 파라미터 하나로 만들었다.**

```python
context_management={
    "edits": [{
        "type": "compact_20260112",
        "trigger": {"type": "input_tokens", "value": 150_000}
    }]
}
```

서버가 요약을 생성해 이전 턴을 대체한다. 문서는 서버측 컴팩션을 "장기 실행 대화와 에이전틱 워크플로에서 컨텍스트를 관리하는 권장 전략"이라고 못 박으며, 그것이 "클라이언트측 요약 코드 없이" 처리된다고 덧붙인다. 어제까지 하네스 코드로 짜던 것이 오늘 딕셔너리 한 줄이다.

**Google은 하네스 레이어 자체를 제품으로 팔기 시작했다.** Antigravity는 2025년 11월 20일 퍼블릭 프리뷰로 나온 에이전틱 개발 플랫폼이다(당시 개인 무료, Mac/Win/Linux). 에디터·터미널·브라우저를 툴 표면으로 내장하고, Agent Manager라는 "여러 에이전트를 스폰·오케스트레이션·관찰하는 전용 인터페이스"를 제공한다. 2026년 7월 기준 Gemini 3.x, Claude Sonnet/Opus 4.6, 오픈웨이트 GPT-OSS 120B를 함께 물릴 수 있다.

그런데 가장 생생한 증거는 벤더 발표문이 아니다. **기존 하네스들이 자기가 짠 코드를 지우고 있다는 사실이다.** opencode(#2871)와 openclaw(#10213)는 클라이언트측 컴팩션 구현을 **삭제하고** 대신 API를 호출하자는 이슈를 열었다. 벤더가 "우리 게 더 낫다"고 말한 게 아니라, 하네스 개발자들이 스스로 자기 코드를 버리기로 한 것이다. 흡수의 증거로 이보다 강한 것은 찾기 어렵다.

## 2. 그런데 어디로 흡수되는가 — 이 글의 핵심 구분

여기서 대부분의 논의가 미끄러진다. "하네스가 흡수된다"와 "하네스가 **모델 안으로** 흡수된다"는 완전히 다른 명제인데, 둘을 같은 문장으로 쓴다.

위에 나열한 증거를 다시 분류해보자.

| 증거 | 어디로 흡수됐나 | 모델 가중치인가? |
|---|---|---|
| Antigravity (Google) | 벤더 IDE·제품층 | 아니오 |
| shell 툴·호스팅 컨테이너 (OpenAI) | API층 | 아니오 |
| `context_management` 파라미터 (Anthropic) | API층 | 아니오 |
| Agent Skills를 API 프리미티브로 (OpenAI) | API층 | 아니오 |
| compaction item 생성 훈련 (OpenAI) | **모델 가중치** | 예 |

가중치 내부화의 직접 증거는 마지막 한 줄뿐이다. OpenAI 문서의 한 문장 — "최신 모델은 이전 대화 상태를 분석해 암호화된 토큰 효율적 표현으로 핵심 상태를 보존하는 compaction item을 생성하도록 훈련됐다." 이게 전부다.

나머지는 전부 API층이거나 제품층이다. Antigravity가 오케스트레이션을 흡수했다는 건, Gemini의 가중치가 오케스트레이션을 배웠다는 뜻이 아니라 **구글의 IDE가 그 코드를 대신 갖고 있다는 뜻이다**. 서버측 컴팩션이 하네스 코드를 지웠다는 건, 모델이 압축을 내면화했다는 뜻이 아니라 **압축 코드가 Anthropic의 서버에서 돈다는 뜻이다**.

**"모델이 좋아지면 하네스가 모델에 탑재된다"는 전망은 절반만 맞다.** 실제로 벌어지는 일은 하네스가 모델사에게 **아웃소싱되는 것이지**, 모델에게 **학습되는 것이** 아니다. 두 시나리오의 실무적 함의는 정반대다. 전자에서 당신은 벤더 종속을 얻고, 후자에서는 벤더 중립을 얻는다.

한 가지 더. OpenAI의 호스팅 컨테이너는 여전히 **선택지다**. 로컬 shell 모드는 그대로 지원되고, 호스팅 컨테이너는 TTY도 sudo도 없으며 네트워크가 기본 차단된다. Anthropic의 컴팩션과 컨텍스트 편집은 아직 베타 헤더 뒤에 있다(GA 기본값이 아니다). 이건 강제된 흡수가 아니라 **선택지로 제공된 흡수다**.

## 3. 무엇이 밖에 남았나 — 벤더가 직접 말한다

가장 좋은 증거는 벤더 자신의 문서다. Anthropic 메모리 툴 문서를 그대로 옮긴다.

> Claude는 메모리 연산을 **요청만** 하고, 애플리케이션이 **당신이 통제하는 스토리지에** 대해 실행한다. 메모리는 **전적으로 당신의 애플리케이션 안에 산다**.

그리고 경로 순회(path traversal) 방어, 크기 제한, 만료 정책은 "**당신의 책임**"이라고 명시한다. 메모리 툴은 GA인데도 그렇다. 즉 벤더는 메모리의 **프로토콜은** 가져갔지만 **스토리지와 그 위의 보안·수명 정책은** 의도적으로 남겼다.

컴팩션도 같다. 요약 알고리즘은 Anthropic의 것이지만, `trigger` 임계값·`keep`·`clear_at_least`·`pause_after_compaction`은 전부 하네스가 정한다. 언제 자를지, 무엇을 반드시 남길지 — 그건 당신 도메인의 문제이지 모델의 문제가 아니다.

패턴이 보인다. **벤더가 흡수한 건 배관이고, 남긴 건 정책과 상태 소유권이다.**

### 정직하게 — 그래도 하네스의 "양"은 줄고 있다

이 지점에서 방어적으로 굴고 싶은 유혹이 생긴다. "봐라, 하네스는 안 죽는다." 하지만 그건 과장이다.

API가 메모리 프로토콜 시스템 프롬프트를 자동 주입하고, SDK 헬퍼(`BetaLocalFilesystemMemoryTool` 같은)가 보일러플레이트를 거의 0에 가깝게 줄인다. 예전엔 메모리 툴을 붙이려면 파일시스템 어댑터를 직접 짰다. 지금은 헬퍼 한 줄이다.

**남는 하네스는 부피가 아니라 책임이다.** 코드 라인 수로 하네스를 세면 하네스는 확실히 죽어간다. 사고의 대상으로 세면 죽지 않는다. 이 구분을 흐리는 글은 신뢰하지 마라(이 글도 포함해서).

## 4. "모델이 좋아지면 하네스가 얇아진다"에 대한 가장 강한 반증

낙관론의 핵심 가정은 이거다 — RL(강화학습) 사후학습이 충분히 돌면 모델이 하네스가 하던 일을 배우고, 하네스는 씻겨나간다.

2026년 6월 프리프린트(arXiv:2606.25447)는 정반대를 보고한다. **빈약한 하네스로 RL 학습시킨 에이전트는, 툴 스키마가 바뀌면 사후학습을 아예 안 한 베이스 모델보다도 못해진다.**

Qwen2.5-7B를 GRPO로, 정보량이 낮은 하네스(h-low)에서 학습시킨 뒤 새 툴 스키마(v2.0)에 던졌을 때 성공률은 **2.7%였다**. 손대지 않은 베이스 모델은 **13.5%**. **10.8점 아래다.**

이건 "RL이 하네스를 씻어낸다"의 반대다. **RL이 하네스에 과적합된다.** 학습 시점의 하네스가 모델 가중치 안에 화석처럼 박히고, 하네스가 바뀌는 순간 그 화석이 부채가 된다. 하네스 설계와 모델 훈련은 순차 문제가 아니라 **공동설계 문제다**.

단서는 반드시 붙여야 한다. 이 논문은 **단일 환경(ALFWorld), 오픈소스 모델 2개, RL 알고리즘 2개, 피어리뷰 없는 프리프린트다**. 그리고 논문 스스로 정보량 높은 하네스(h-high)의 적응조차 "**표면적**"이라고 부른다 — 유효한 도구 호출이 95.7%나 되지만 그중 34.9%는 현재 상태에서 실행 불가능한 호출이었다(별도 실험 조건인 GiGPO 계열 수치다. 위의 2.7/13.5와 섞어 읽으면 안 된다). 모델이 문법은 배웠지만 의미는 못 배웠다는 뜻이다.

그러니 이걸 "하네스 승리 선언"으로 읽으면 안 된다. 읽어야 할 것은 방향뿐이다 — **하네스는 훈련이 지워버릴 잡음이 아니라, 훈련이 흡수해버리는 가정이다.**

### 그리고 컨텍스트 로트

컨텍스트 로트(context rot)는 긴 컨텍스트에서 모델의 회상 정확도가 떨어지는 현상이다. Anthropic의 설명은 이렇다 — "컨텍스트 윈도의 토큰 수가 늘수록 모델이 그 컨텍스트에서 정보를 정확히 회상하는 능력은 떨어진다." 원인은 우연이 아니라 구조다. 트랜스포머의 n² 페어와이즈 관계, 그리고 짧은 시퀀스에 치우친 학습 분포. Chroma의 후속 연구는 **1M 토큰 윈도 모델을 포함한 18개 프런티어 모델 전부에서** 길이에 따른 열화를 재현했다.

주목할 점은 이해상충의 방향이다. 대형 컨텍스트 윈도를 파는 회사가 "윈도가 커지면 회상이 나빠진다"고 말하고 있다. 자사 마케팅에 역행하는 주장이라 신뢰도가 높다.

**하지만 여기서 혼동하면 안 된다.** 컨텍스트 로트는 "윈도가 커지면 하네스가 필요 없어진다"의 반증이지, "**벤더가 하네스를 흡수한다**"의 반증이 **아니다**. 오히려 아이러니는 반대편에 있다 — **Anthropic 자신이 바로 이 논거를 들어 memory tool과 context editing을 API 안에 집어넣고 있다.** "컨텍스트는 유한 자원이니 관리해야 한다"는 명제에서 "그러니 당신이 관리하라"가 아니라 "그러니 **우리가** 관리하겠다"가 나온 것이다.

컨텍스트 로트는 하네스가 필요하다는 증거다. 하네스가 **당신 것으로 남는다는** 증거는 아니다.

## 5. 다음 개념 후보 — 그리고 그 반전

"하네스 다음"으로 가장 구체화된 후보는 **스킬/메모리를 외부 상태로 다루는 프레임이다**. SkillOpt(arXiv:2605.23904)가 대표적이다.

이 프레임은 스킬 문서를 "동결된 에이전트의 **외부 상태로**, 가중치 공간 최적화를 재현 가능하게 만드는 것과 같은 규율로 훈련되어야 한다"고 본다. 별도 옵티마이저가 채점된 롤아웃을 제한된 편집으로 바꾸고, 홀드아웃 검증 점수를 "엄격히" 개선할 때만 그 편집을 수용한다. 요컨대 **모델 밖의 결정론적 검증 게이트다**. 실제로 강한 하네스 안에서도 헤드룸이 남는다 — GPT-5.5 기준 스킬 없음 대비 Codex +24.8점, Claude Code +19.1점이다(반드시 "스킬 없음 대비"임을 유의하라. 최선 베이스라인 대비로 깎으면 이득은 훨씬 작아진다. 상세는 [[skillopt-비판적-리뷰]]로 넘긴다).

### 반전: SkillOpt는 하네스를 대체하지 않는다. 전제한다.

"자기진화하는 스킬이 하네스를 삼킨다"는 서사를 이 논문으로 뒷받침할 수 없다. 이유는 단순하다.

**SkillOpt는 세 개의 실행 하네스 "안에서" 평가된다.** 그리고 스킬은 **가중치 업데이트의 대체재이지** **하네스의 대체재가** 아니다. 모델을 재학습시킬 수 없으니 텍스트 공간에서 대신 최적화하겠다는 것이고, 그 최적화된 텍스트는 여전히 하네스가 로드하고 하네스가 실행한다. 이 프레임은 하네스를 없애는 게 아니라 **하네스를 요구한다**. 검증 게이트라는 개념 자체가 하네스의 일부다. ([[하네스-자기개선]]의 논지와 정확히 같은 자리다 — 자기개선은 가중치가 아니라 하네스에서 일어난다.)

한편 [[에이전트-스킬은-워크플로다|스킬]] 로딩 계층 자체는 상품화되는 중이다. OpenAI 문서는 "스킬은 개방형 Agent Skills 표준(agentskills.io)과 호환된다"고 명시하고, Azure OpenAI가 같은 포맷을 미러링한다. **Anthropic이 연 포맷을 경쟁사가 API 프리미티브로 채택한 것이다**. 다만 수렴은 코어 포맷 수준이고 기능 수준에서는 부분적이다 — Codex의 `openai.yaml` 메타데이터를 Claude Code는 인식하지 못하고, Claude Code의 서브에이전트 컨텍스트 포킹은 Codex에 대응물이 없다. OpenAI에서 스킬은 독립 top-level API가 아니라 `tools[].environment.skills`로 붙는다.

### 반대 축: "하네스는 자산이다" — 한 입장 논문의 주장

균형을 위해 반대 전망도 놓는다. 다만 **이건 확인된 사실이 아니라 한 편의 입장 논문이 내놓은 주장이라는** 걸 분명히 해둔다.

2026년 4월 CAAF 논문(arXiv:2604.17025)은 도메인 불변식(domain invariant, 그 업계에서 반드시 참이어야 하는 규칙)이 실행 가능한 하네스로 형식화되면 "하네스 자체가 파운데이션 모델이 상품화될수록 가치가 복리로 증가하는 1급 기업 자산이 된다"고 주장한다. 프로덕션의 새 엣지케이스가 어서션을 하나씩 더하고, 그 어서션 더미가 모델에 종속되지 않는 독점 지식베이스가 된다는 그림이다.

솔깃하다. 그래서 더 조심해야 한다. **피어리뷰 없는 단일 저자(독립 연구자) 프리프린트이고, 인용도 비판도 독립 검증도 없다.** 이 글은 이 테제를 "하네스가 살아남는다는 증거"로 격상시키지 않는다. 그런 논문이 있고, 그런 주장이 있다는 것까지가 확인된 전부다.

## 6. 흡수는 언어에도 남는다 — 벤더가 "하네스"라는 이름표를 집어 들었다

이 글을 구상할 때 나는 반대 가설을 들고 있었다. **벤더는 "하네스"라는 단어를 쓰지 않는다, 팔 수 없는 것에는 이름을 붙이지 않으니까.** 검증에서 뒤집혔다. 그리고 뒤집힌 결과가 이 글의 논지를 오히려 강하게 만든다.

사실관계는 이렇다. Anthropic의 컨텍스트 엔지니어링 대표 글에는 `harness`도 `scaffolding`도 **한 번도 나오지 않는다**. 여기까지는 참이다. 그래서 "하네스는 실무자 용어"라고 결론짓기 쉽다. 그런데 두 달 뒤인 2025년 11월 26일, Anthropic은 제목에 그 단어를 박은 엔지니어링 글을 냈다 — "Effective harnesses for long-running agents". 본문의 한 문장이 결정적이다.

> The Claude Agent SDK is a powerful, general-purpose agent **harness**…
> (Claude Agent SDK는 강력한 범용 에이전트 **하네스다**…)

그리고 memory tool API 문서가 이 글을 직접 링크한다. 즉 벤더는 (a) 그 단어를 쓰고, (b) **자사 SDK에 그 이름표를 붙였으며**, (c) API 문서에서 개발자를 그 글로 유도한다.

**이건 "하네스가 살아남는다"의 증거가 아니라, 앞 장들에서 본 구조적 흡수의 언어적 방증이다.** 실무자 용어였던 것을 벤더가 자기 제품 이름으로 집어 들었다. 어떤 범주에 이름을 붙이고 자기 제품을 그 이름으로 부르는 것 — 그게 시장을 정의하는 방식이다. 컨테이너를 API에 넣고(OpenAI), 컴팩션을 파라미터로 만들고(Anthropic), IDE를 에이전틱 플랫폼으로 파는(Google) 흐름의 언어적 대응물이 여기 있다.

한때 "모델 바깥에 남은 것 전부"를 가리키던 잔여 범주가, 이제 벤더 제품 카탈로그의 한 줄이다.

다만 이름을 가져갔다고 해서 내용물까지 다 가져간 건 아니다. 3장에서 봤듯 같은 회사가 메모리 스토리지와 그 위의 보안 정책은 **명시적으로** 애플리케이션에 남겨뒀다. 이름은 넘어갔고, 정책은 남았다. 이 글의 구분은 그대로 유지된다.

Anthropic이 컨텍스트 엔지니어링 글의 결론에서 하는 말도 같은 자리를 가리킨다.

> 역량이 확장되더라도, 컨텍스트를 귀중하고 유한한 자원으로 다루는 것은 신뢰할 수 있고 효과적인 에이전트 구축의 중심으로 남을 것이다.

이 문장을 "하네스는 흡수되지 않는다"로 읽으면 오독이다. Anthropic은 **원칙이 중심으로 남는다고** 말했지, **메커니즘이 당신 코드에 남는다고** 말하지 않았다. 메커니즘은 지금 이 순간에도 흡수되고 있다. 같은 회사의 API 문서가, 그리고 자사 SDK를 하네스라 부르는 그 글이 증거다.

## 7. 3–5년 전망 — 사라질 것과 남을 것

**주의: 이 표는 단일 소스의 문장이 아니라 위 증거들을 묶은 종합 추론이다.** 근거 패턴은 세 갈래다 — (a) 벤더가 흡수한 것은 예외 없이 메커니즘이었고, (b) 벤더가 명시적으로 앱에 남긴 것은 정책과 상태였으며, (c) SkillOpt의 검증 게이트와 CAAF의 어서션 레지스트리는 서로 다른 진영인데도 똑같이 "모델 밖의 결정론적 게이트"를 핵심 자산으로 지목했다.

| 상품화되어 사라질 것 | 하네스에 남을 것 |
|---|---|
| 에이전트 루프 구현 | **컨텍스트 정책** — 무엇을 언제 자를지의 임계·keep 규칙 |
| 컴팩션·요약 로직 | **스토리지와 감사 가능한 상태 소유권** |
| 샌드박스 컨테이너 프로비저닝 | **결정론적 검증 게이트** — 도메인 불변식을 실행 가능한 어서션으로 |
| 툴 호출 파싱·재시도 | **툴 스키마 설계** — 하네스-훈련 공동설계 문제이므로 |
| 스킬 로딩·버저닝 | **도메인 특화 평가** ([[에이전트-평가-evals]]) |
| 기본 멀티에이전트 스폰·관찰 UI | |

왼쪽 열을 경쟁 우위라고 믿고 팀 리소스를 붓고 있다면, 3년 뒤 그 코드는 삭제될 코드다. 오른쪽 열은 벤더가 대신 정해줄 수 없다 — 당신 도메인을 모르기 때문이다.

실무적으로 이건 채용과 아키텍처 결정에 바로 꽂힌다. 하네스 배관을 잘 짜는 엔지니어보다, **무엇이 잘렸는지 증명할 수 있게 설계하는 엔지니어가** 3년 뒤에 비싸다.

## 8. 한계 — 이 글이 말하지 못하는 것

이 글의 논지가 "검증된 것만 말하라"이므로, 이 글 자신의 한계도 정직하게 적는다.

**핵심 논문 3편이 전부 피어리뷰 없는 프리프린트다.** 하네스-훈련 상호작용(2606.25447), SkillOpt(2605.23904), CAAF(2604.17025). 특히 CAAF는 저자가 독립 연구자 한 명이고, 인용은 0이다.

**RL 축의 증거 기반이 얇다.** 단일 환경(ALFWorld), Qwen2.5 계열 모델, 피어리뷰 없음. 4장의 반증은 방향을 시사하지만 정착된 결론이 아니다.

**오케스트레이션 계층의 상품화 정도는 미측정이다.** 이 축에 대해 검토한 주장들이 검증을 통과하지 못했다. 그래서 "멀티에이전트 오케스트레이션이 얼마나 벤더 쪽으로 넘어갔는가"에 대해 이 글은 **답하지 않는다.** 표의 "기본 멀티에이전트 스폰·관찰 UI"는 Antigravity의 Agent Manager라는 단일 사례에 기댄 추론이다.

**벤더 1차 소스는 "무엇이 출시됐는가"에는 권위가 있지만 "얼마나 잘 작동하는가"에는 없다.**

**유통기한은 길게 잡아 6개월이다.** Anthropic의 컴팩션·컨텍스트 편집은 아직 베타 헤더 뒤에 있고, Antigravity의 "개인 무료·프리뷰"는 2025년 11월 스냅샷이다(모델 라인업은 2026년 5월에 갱신됐다). 여기 적힌 날짜를 지우고 현재형으로 읽지 마라.

## 9. 열린 질문 — 하네스가 살아남는 가장 실질적인 이유

마지막으로, 이 글을 쓰면서 가장 답이 궁금해진 질문 하나를 남긴다.

OpenAI 문서는 서버측 compaction item을 "사람이 해석할 수 없는 불투명한" 표현이라고 설명한다. 암호화된, 토큰 효율적인, 모델만 읽는 상태 요약이다. 성능 관점에서는 훌륭한 설계다.

그런데 **감사(audit) 관점에서는 블랙박스다.**

에이전트가 200턴을 돌면서 여덟 번 컴팩션했다고 하자. 그 과정에서 무엇이 버려졌는지, 당신은 증명할 수 있는가? 규제 당국에 제출할 수 있는 형태로?

금융과 의료를 생각해보자. 대출 심사 에이전트가 신청인의 어떤 정보를 "잊었는지" 설명하지 못한다면, 그 심사는 감사를 통과하는가. 진단 보조 에이전트가 초기 문진 내용을 컴팩션 과정에서 버렸는데 그 사실을 로그로 재구성할 수 없다면, 그건 허용되는가.

**"모델이 무엇을 잊었는지 증명할 수 없는" 상태를 받아들일 수 없는 산업이 있다.** 그리고 그 산업들은 자기 컴팩션을 자기가 짤 것이다. 트리거를 자기가 정하고, 버려진 것을 자기가 로깅하고, 그 로그를 자기가 보관할 것이다.

하네스가 살아남는 이유는 모델이 부족해서가 아닐지도 모른다. **책임을 아웃소싱할 수 없어서일지도 모른다.**

## 이 글의 근거

이 글은 111개 에이전트가 28편의 1차 소스를 읽어 134개 주장을 추출하고, 그중 25개를 적대적 검증(3표 중 2표가 반증하면 기각)에 부친 결과를 재료로 썼다. 18개가 확정됐고 7개가 기각됐다. **기각된 7개는 이 글에 단 한 줄도 들어가지 않았다** — 그중 한 축은 검증 전까지 이 글의 주요 근거가 될 예정이었다. 그 축이 빠진 자리가 8장의 "오케스트레이션 계층 미측정"이다.

발행 직전 팩트체크에서 한 건이 더 뒤집혔다. "벤더는 하네스라는 단어를 쓰지 않는다"는 6장의 원래 전제가 반증됐고(Anthropic은 2025년 11월 자사 SDK를 "agent harness"라고 불렀다), 6장은 그 반증을 흡수해 결론을 뒤집은 상태로 다시 썼다.

각 주장의 확신도(high/medium/low)는 본문 서술의 강도에 반영했다. CAAF의 하네스-자산론(5장)은 확신도 LOW이므로 "한 입장 논문의 주장"으로만 서술했고, RL 반증(4장)은 medium이므로 방향만 취하고 결론으로 격상하지 않았다.
