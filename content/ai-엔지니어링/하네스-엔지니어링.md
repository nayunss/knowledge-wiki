# 하네스 엔지니어링 (Harness Engineering)

**Agent = Model + Harness.** 하네스는 모델 가중치를 뺀 나머지 전부 — 프롬프트·도구·컨텍스트 정책·훅·샌드박스·서브에이전트·관측을 감싼 실행 발판이다. 날것의 모델은 에이전트가 아니고, 하네스가 상태·도구 실행·피드백 루프·강제 가능한 제약을 줄 때 비로소 에이전트가 된다. AI 엔지니어링 [[ai-엔지니어링-4계층]]에서 "모델을 둘러싼 환경"을 담당한다.

> "당신이 모델이 아니라면, 당신은 하네스다(If you're not the model, you're the harness)." — LangChain

## 용어 주의 — 광의와 협의

정의가 갈린다. 노트는 **광의**를 기본으로 쓴다.
- **광의**(LangChain·Addy Osmani): 프롬프트·도구·샌드박스·오케스트레이션·관측을 모두 포함하는 바깥 껍데기 전체. "하네스 엔지니어링" 담론의 지배적 용법.
- **협의**(Hugging Face 글로서리, 2026): 스캐폴드(모델이 *무엇을* 보는가 — 시스템 프롬프트·도구 설명·파싱)와 하네스(모델을 *어떻게* 실행하는가 — 호출·툴콜 처리·중단 결정)를 나눈다. 이 관점에선 스캐폴드가 하네스 안에 있다.

## 왜 중요한가 — "모델 문제가 아니라 설정 문제다"

- 핵심 명제: 오늘날 모델이 *할 수 있는 것*과 실제로 *하는 것* 사이의 격차는 대부분 <strong>하네스 격차(harness gap)</strong>다. 에이전트의 실수는 능력 부족이 아니라 설정 결함(없는 컨벤션, 빠진 안전 훅, 잘못된 작업 분해)을 드러낸다.
- 실패의 성격: 코드 에이전트 실패의 상당수는 모델 생성이 아니라 레포 컨텍스트 누락·취약한 도구 인터페이스·약한 검증기·부실한 재시도·권한 경계 불일치에서 온다.

### 벤치마크 증거 (Terminal Bench 2.0)
LangChain은 모델을 `gpt-5.2-codex`로 **고정**한 채 하네스만 손봐 **52.8에서 66.5로(+13.7점)**, Top 30에서 Top 5로 올렸다. 바꾼 건 세 개의 노브뿐이다.
- **시스템 프롬프트**
- **도구(Tools)**
- **미들웨어(훅)**: 종료 직전 검증을 강제하는 `PreCompletionChecklistMiddleware`("모델은 첫 그럴듯한 답에 편향된다"), 시작 시 작업 디렉터리·도구·환경을 미리 주입하는 `LocalContextMiddleware`, 계획·검증엔 많이 중간엔 절제하는 추론 예산 배분(reasoning sandwich).

한 정리에 따르면 결과 품질에서 <strong>모델은 약 10%, 하네스가 약 90%</strong>를 좌우한다.

## 구성요소

- **지시·룰 파일**: 시스템 프롬프트 + `AGENTS.md`/`CLAUDE.md`. Osmani는 `AGENTS.md`를 "가장 레버리지가 큰 설정 표면"으로 꼽는다.
- **도구·MCP·스킬**: 이름·설명·스키마로 컨텍스트에 주입되는 "에이전트의 손". 하네스가 등록·검증·실행·결과 캡처를 맡는다.
- **번들 인프라**: 파일시스템·Git(지속 상태·버전 관리), Bash/코드 실행, 브라우저.
- **샌드박스**: 격리된 실행 환경.
- **오케스트레이션·모델 라우팅**: 서브에이전트 스폰, 작업별 모델 배분.
- **훅/가드레일**: 라이프사이클 지점(도구 호출 전·파일 수정 후·커밋 전)에 도는 결정론적 코드. 원칙은 "성공은 침묵하고, 실패는 시끄럽다."
- **메모리·컨텍스트 관리**: compaction, 툴콜 오프로딩, 점진적 스킬 공개(→ [[컨텍스트-엔지니어링]]).
- **관측**: 로그·트레이스·평가·비용/지연 계측. 트레이스로 대규모 실패 모드를 분석하는 게 개선의 출발점이다.

설계 원칙: 어떤 컴포넌트가 제공하려는 *행동*을 이름 붙일 수 없다면, 그건 아마 없어야 한다.

## 코딩 에이전트 = 사실상 하네스

Claude Code·Codex·Cursor는 "특정 모델 + 정교하게 튜닝된 실행 층"이다. Claude Code 문서는 스스로를 "Claude를 둘러싼 에이전틱 하네스"라 부른다. 같은 모델이라도 하네스가 다르면 — 중단 조건·에러 처리·의사결정 방식이 달라 — 경험이 완전히 달라진다.

## SDLC에서의 위치

구글 백서 *The New SDLC with Vibe Coding*(Addy Osmani 등)은 하네스를 SDLC 전 단계에 관통시킨다: 요구·아키텍처에서 하네스를 **구성**하고, 구현에서 **가동**하고, 테스트에서 피드백 루프로 **자기교정**하고, 배포·유지보수에서 훅·관측으로 **감시**한다. 자세한 매핑은 [[agents-cli-lifecycle-sdlc]].

## 다른 층과의 관계

- [[프롬프트-엔지니어링]] ⊂ [[컨텍스트-엔지니어링]] ⊂ 하네스 엔지니어링(포함 관계). 프롬프트는 지시를, 컨텍스트는 모델에 제시되는 정보를, 하네스는 **운영 환경 전체**(도구 스키마·메모리 정책·샌드박스·검증 센서·권한·라우팅)를 다룬다.
- [[루프-엔지니어링]]은 이 하네스를 시간축으로 구동하는 층 — "하네스가 환경, 루프가 행동"이다. 둘의 결합은 [[ai-엔지니어링-4계층]] 참고.
