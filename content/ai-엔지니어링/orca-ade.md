에이전트를 두 개 이상 동시에 굴려본 사람은 같은 벽을 만난다. 한 세션이 `src/auth.ts`를 고치는 동안 다른 세션이 같은 파일을 고친다. 되돌리면 둘 다 죽는다. 그래서 대부분은 병렬을 포기하고 하나씩 순서대로 돌린다.

[Orca](https://github.com/stablyai/orca)는 이 문제에 뻔한 답을 낸다. 작업마다 git worktree(같은 레포를 서로 다른 브랜치로 여러 디렉터리에 동시 체크아웃하는 git 표준 기능)를 파서 에이전트를 격리한다. 답이 뻔한 만큼 이 도구의 진짜 이야기는 격리가 아니다. **CLI를 직접 뜯어보면 나온다.**

## 핵심 주장: 이 CLI의 사용자는 사람이 아니다

Orca를 "AI가 붙은 IDE"로 소개하는 글이 많지만, 필자가 로컬에 설치된 Orca CLI를 직접 조사한 결과는 다른 그림을 가리킨다. 커맨드 표면 전체가 **에이전트가 읽고 호출하도록** 설계돼 있다. 사람이 쓰기 편한 도구가 아니라, 에이전트가 다른 에이전트를 부리도록 만든 관제 API에 GUI를 얹은 물건이다. 그래서 Orca의 자기 규정도 IDE가 아니라 ADE(Agent Development Environment)다.

근거는 세 가지다. 하나씩 보자.

### 근거 1 — `agent-context`: 사람이 읽으라고 만든 문서가 아니다

Orca에는 `agent-context`라는 커맨드가 있다. `--help`에 적힌 설명은 "Print the machine-readable command schema for agents"다. 실제로 실행하면 이렇게 나온다.

```
$ orca agent-context
200 commands (schema v1).
Run `orca agent-context --json` for the full machine-readable command schema.

$ orca agent-context --json | wc -c
  130098
```

커맨드 200개, 스키마 v1, JSON 130KB. (2026-07-14 필자 머신에서 관측)

130KB짜리 커맨드 스키마를 읽을 사람은 없다. 하지만 에이전트는 읽는다. 한 번 읽고 컨텍스트에 넣으면 그 뒤로는 `--help`를 뒤질 필요 없이 200개 커맨드를 다 쓴다. 이건 문서가 아니라 **툴 정의(tool definition) 배포 채널**이다. 자기 CLI 표면을 통째로 기계 판독 형식으로 뱉는 개발 도구는 흔하지 않다.

### 근거 2 — `--json`, 그리고 `tui-idle`

커맨드 대부분이 `--json` 플래그를 받는다. `worktree list`, `terminal wait`, `tab list`, `worktree create` … 파이프 뒤에 파서가 있다고 가정한 설계다.

솔직히 이것만으로는 약한 근거다. `gh`도, `kubectl`도, `aws`도 `--json`을 단다. 사람이 셸 스크립트와 `jq`로 후처리하라고 붙인 것이다. `--json`은 정황이지 증거가 아니다.

무게는 다른 플래그에 있다.

```
$ orca terminal wait --terminal term_123 --for exit --timeout-ms 60000 --json
```

`--for`가 받는 값이 `exit` 또는 `tui-idle`이다. 프로세스 종료뿐 아니라 **TUI가 놀고 있는 상태**까지 대기 조건으로 노출한다. 즉 "Claude Code가 답을 끝내고 입력을 기다리는 시점"을 프로그램이 잡아낸다. 사람이야 화면을 보면 안다. `jq`를 쓰는 사람도 이 플래그는 필요 없다. 이건 화면을 볼 눈이 없는 호출자를 위한 것이다.

### 근거 3 — 에이전트가 에이전트를 스폰한다

결정적인 건 이 한 줄이다. `--help`에 공식 예시로 실려 있다.

```
$ orca worktree create --name agent-task --agent codex --prompt "hi" --json
```

새 워크트리를 만들고, 그 안에 터미널을 띄우고, 지정한 TUI 에이전트를 실행하고, 초기 프롬프트까지 밀어 넣는다. 한 줄이다. 이 한 줄을 **Claude Code 세션 안의 에이전트가 호출할 수 있다.** GUI로 창 다섯 개를 여는 것과 결과는 같지만, 사람의 노동이 사라진다.

`worktree create --help`를 더 파보면 설계 의도가 더 선명해진다.

| 플래그 | 하는 일 | 왜 에이전트용인가 |
|---|---|---|
| `--agent <id> --prompt <text>` | 워크트리 생성 + 에이전트 실행 + 프롬프트 주입 | 스폰이 원자적 연산 한 개가 된다 |
| `--parent-worktree <selector>` / `--no-parent` | 워크트리 계보(lineage) 기록 | 누가 누구를 낳았는지 추적 — 에이전트 트리를 도구가 안다 |
| `--issue` / `--linear-issue` | GitHub·Linear 이슈 연결 | 작업 출처가 메타데이터로 붙는다 |
| `--agent`와 함께 쓴 `--json` | `result.agentTerminalHandle` 반환 | 스폰한 쪽이 자식 터미널 핸들을 즉시 받아 조종한다 |

특히 lineage가 재밌다. `--help`는 "By default, Orca records the new worktree as a child of the caller context when it can infer one from the Orca terminal or current directory"라고 못 박는다. 호출자가 에이전트일 것을 전제하고, 그 호출자를 부모로 기록한다는 뜻이다. 그리고 `worktree ps`("Show a compact orchestration summary across worktrees")로 함대 전체를 한 화면에 요약한다. 필자 머신에서 돌리면 워크트리별로 살아 있는 터미널 수(`live:1`), PTY 여부, 읽지 않은 출력 여부, 그리고 각 에이전트의 최근 출력 미리보기가 한꺼번에 나온다.

관제탑이라는 표현은 비유가 아니다. 그게 `ps`가 하는 일이다.

[[하네스의-미래]]는 "흡수되는 것은 메커니즘이고 남는 것은 정책과 상태 소유권"이라고 정리한다. 그 글이 말한 흡수 주체는 모델 벤더의 API·플랫폼이었다. Orca는 벤더가 아니라 서드파티 도구다. 그러니 사례라기보다 **같은 패턴이 도구 층에서도 반복되는 것**으로 읽는 게 정확하다. 워크트리 생성, 터미널 스폰, 상태 폴링 — 각자 셸 스크립트로 짜던 배관을 Orca가 통째로 가져갔다. 남는 건 여전히 정책이다. 무엇을 병렬로 돌릴지, 무엇을 병합할지는 도구가 정해주지 않는다.

## 핸드오프냐 조율이냐 — 경계를 틀리면 돈이 샌다

Orca 레포는 코드만 배포하지 않는다. `skills/` 디렉터리로 Claude Code 스킬 여덟 개를 함께 배포한다(2026-07-14 GitHub 조회 기준). 그중 오케스트레이션 성격의 세 개가 이 도구의 사용 경계를 그대로 드러낸다. 필자 머신에도 이 셋이 설치돼 있다.

| 스킬 | 언제 | 비용 성격 |
|---|---|---|
| `orca-cli` | 소유권을 넘기고 잊는다(fire-and-forget). 워크트리·터미널·레포·automations·Orca 내장 브라우저 조작 | 던지고 끝. 대기 토큰 없음 |
| `orchestration` | 결과를 지켜보고 조율한다. 스레드 메시지, 블로킹 ask/reply, task DAG, 결정 게이트, 코디네이터 루프 | 부모 세션이 살아서 대기 — 대기하는 동안 컨텍스트를 물고 있다 |
| `computer-use` | Orca 밖 데스크톱 앱·브라우저 창 조작 (접근성 트리 기반) | 별개 축 |

핵심은 이거다. **지켜보지 않을 일에 조율을 쓰면 대기 비용만 태운다.** "이 리팩터링 맡기고 넌 다른 거 해"는 `orca-cli` 핸드오프다. "세 워커 결과를 모아서 판단해"만 `orchestration`이다. 이 구분을 안 하면 부모 에이전트가 자식을 지켜보느라 컨텍스트를 붙들고 앉아 있고, 병렬로 얻은 시간을 대기로 반납한다.

같은 문제를 [[codex-교차모델-위임]]은 다른 층위에서 푼다. 거긴 **세션 안의 도구 호출**로 다른 모델을 부른다. 컨텍스트를 넘기기 쉽고 결과를 바로 받지만, 부모 세션이 끝까지 살아 있어야 하고 격리는 없다. Orca는 **프로세스와 파일시스템을 갈라놓는다.** 격리는 진짜지만 컨텍스트는 프롬프트로 다시 실어 보내야 한다. 어느 쪽이 낫냐가 아니라, 격리가 필요한 작업이냐 아니냐의 문제다. 같은 파일을 안 건드리는 작업이라면 굳이 워크트리를 팔 이유가 없다.

## 브라우저: 픽셀이 아니라 접근성 트리

Orca는 앱 안에 Chromium을 물고 있고, CLI로 조작한다. 흐름은 단순하다.

```
$ orca snapshot                        # 접근성 트리 + element ref (e1, e2, e3…)
$ orca click --element e2
$ orca fill --element e5 --value "hello"
$ orca eval --expression "document.title"
```

좌표가 아니라 ref다. 픽셀 클릭은 해상도·스크롤·애니메이션에 쉽게 깨지지만 ref는 그렇지 않다. 다만 `orca tab current --help`가 직접 경고한다. "Element refs change after navigation — always re-snapshot before interacting." 페이지가 넘어가면 ref는 무효다. 그리고 워크트리를 여러 개 병렬로 돌릴 때는 `orca tab list --json`에서 `tabs[].browserPageId`를 받아 `--page <id>`로 고정하라고 권한다. 안 하면 두 에이전트가 같은 탭을 밟는다. 격리 도구인데도 브라우저 층에서는 격리를 사용자가 챙겨야 한다는 뜻이다.

Design Mode는 여기서 한 걸음 더 간다. 워크트리마다 실제 Chromium 창이 뜨고, UI 요소를 클릭하면 그 HTML·CSS·크롭 스크린샷이 에이전트로 전송된다(공식 문서 기준). "헤더 오른쪽 위 파란 버튼 있잖아, 그거 말고 그 아래"라는 좌표 설명을 없애는 게 목적이다.

## 트레이드오프: 격리의 청구서

여기까지가 좋은 소식이다. 이제 계산서를 보자.

### 함정 1 — 워크트리마다 환경을 다시 짓는다

git worktree는 추적되는 파일만 체크아웃한다. 추적 안 되는 건 안 따라온다. 즉 워크트리마다 `node_modules`가 없고, `.env`가 없고, 빌드 캐시가 비어 있다. **에이전트 5개를 띄우면 `npm install`을 5번 한다.** 모노레포에서 이건 농담이 아니다.

Orca도 이걸 알고 있다. 장치는 두 축이다. 하나는 **레포가 정의하는 셋업 훅** — `worktree create --help`가 "Repo-defined setup hooks follow the repository setup policy"라고 밝히고, `--setup run|skip|inherit`으로 그 정책을 제어한다. 다른 하나는 `orca.yaml`의 **워크스페이스 환경 레시피**이고, `vm recipe doctor`가 이걸 검증한다. 훅을 정의해두면 워크트리를 만들 때 알아서 돈다.

주의할 게 하나 있다. `--setup inherit`은 **정책을 상속하는 것이지 이미 설치된 의존성을 상속하는 게 아니다.** 다섯 번 설치는 여전히 다섯 번이다.

그리고 진짜 문제는 그 훅을 **누가 짜느냐**다. 도구가 대신 짜주지 않는다. 레포마다 "이 워크트리를 일할 수 있는 상태로 만드는 절차"를 명시적으로 코드화해야 하고, 그게 이 도구의 진짜 진입 비용이다. 이걸 안 하면 격리로 번 시간을 환경 구축으로 그대로 반납한다. 셋업 자동화가 안 된 레포에서 Orca를 켜는 건, 창을 다섯 개 여는 노동을 `npm install`을 다섯 번 기다리는 노동으로 바꾼 것에 지나지 않는다.

### 함정 2 — "같은 작업을 N개 에이전트에 던져 최고를 고른다"

병렬 격리가 되면 반드시 이 유혹이 온다. 같은 티켓을 에이전트 5개에 던지고 제일 나은 구현을 고르자. 토너먼트다.

거의 항상 손해라고 본다. 이건 실측이 아니라 추론이니 근거를 그대로 밝힌다. 첫째, 토큰이 정직하게 N배 나간다. 둘째, 대부분의 실무 작업은 정답의 폭이 좁아서 비교할 게 없다. 다섯 개의 "버그 수정"을 놓고 고르는 심사 비용이 그냥 하나를 리뷰하는 비용보다 크다.

인접한 실측은 하나 있다. [[디자인-스킬-비교-실험]]은 같은 명세의 투두 앱을 여덟 에이전트에 던지되 조건을 달리했다(디자인 스킬 7종 + baseline 1종). 동일 조건의 N회 반복이 아니라 조건 비교였으니 토너먼트를 직접 측정한 건 아니다. 다만 두 가지가 남았다. **조건을 여덟 가지로 흩뿌렸는데도 기능 완성은 전원 통과해 상수였고**, 갈린 품질은 대체로 투입 비용에 비례했다(baseline 29k 토큰 → 최상위 167k, 약 5.7배). 다양성은 공짜로 오지 않았다는 뜻이다.

**병렬은 독립 작업에 쓸 때 이득이다.** 같은 작업의 N중 실행은 병렬이 아니라 중복이다.

### 함정 3 — 작업이 하나뿐이면 이득이 얇아진다

Orca의 이득은 "동시에 진행 가능한 독립 작업이 실제로 여러 개 있다"는 전제 위에서 커진다. 그 전제가 없다고 이득이 0이 되지는 않는다. 작업이 하나여도 워크트리는 **메인 체크아웃을 보호한다.** 에이전트가 자기 트리를 휘젓는 동안 사람은 자기 체크아웃에서 계속 빌드하고 테스트할 수 있다. 이것도 엄연한 병렬이다.

문제는 그 이득 하나가 값을 하느냐다. 앱 하나 더, 워크트리 관리 개념 하나 더, 셋업 훅 유지보수 하나 더. 메인 체크아웃 보호만 원한다면 `git worktree add` 한 줄로도 된다. Orca를 얹을 이유는 함대가 생길 때 비로소 생긴다.

## 그래서 도입할 것인가 — 판단 기준

| 조건 | 판정 |
|---|---|
| 동시에 굴릴 독립 작업이 상시 3개 이상 (버그·기능·리팩터링이 서로 다른 파일) | 도입 검토 |
| 레포 셋업이 이미 스크립트 한 방으로 끝난다 (`make setup`, devcontainer 등) | 진입 비용 낮음 — 유리 |
| 셋업이 구전 지식이고 `.env`를 슬랙에서 받는다 | 먼저 셋업부터 코드화하라. Orca는 그다음이다 |
| 에이전트가 다른 에이전트를 스폰하는 하네스를 짜고 있다 | CLI만으로도 값어치가 있다 (`agent-context` + 스폰 한 줄) |
| 작업이 대체로 하나씩 순차적이다 | 서두르지 마라. `git worktree` 한 줄로 얻을 이득밖에 없다 |

레포는 MIT 라이선스다. GitHub 스타 약 1만 8,700개, 2026-03-17 생성이고, 조회 당일에도 커밋이 들어오고 있었다(2026-07-14 GitHub API 조회 기준). 생긴 지 넉 달 된 도구라는 뜻이기도 하다. 커맨드 표면이 200개인데 스키마 버전은 아직 v1이고, `orca --version`은 버전 문자열을 출력하지 않는다(관측). 빠르게 움직이는 프로젝트를 프로덕션 워크플로의 중심에 놓을 때의 통상적인 위험은 그대로 있다. 참고로 레포 경로는 `stablyai/orca`인데 LICENSE의 저작권자는 Lovecast Inc.로 표기돼 있다. 이 글은 회사명을 단정하지 않고 레포 경로로 지칭한다.

## 결론

Orca의 가치는 워크트리 격리가 아니다. 그건 git이 원래 하던 일이다. 가치는 **격리·스폰·상태 조회를 에이전트가 직접 호출 가능한 200개짜리 기계 판독 API로 만들었다**는 데 있다. 사람이 GUI로 창을 여는 대신, 에이전트가 한 줄로 함대를 띄운다.

그 대가는 정직하다. 워크트리마다 환경을 다시 지어야 하고, 그 셋업은 당신이 짜야 한다. 그리고 이 투자가 값을 하려면 병렬로 돌릴 독립 작업이 실제로 있어야 한다. 없으면 관제탑을 지어놓고 비행기 한 대를 관제하는 셈이다. 틀린 일은 아니지만, 관제탑까지 지을 일도 아니다.

[[하네스-엔지니어링]] 관점에서 Orca는 하네스의 실행 발판이다. 모델도 아니고 정책도 아니다. 터미널과 격리와 상태를 소유하는 층. 그 층을 직접 짜고 있었다면, 이 도구는 당신이 짜던 셸 스크립트를 대체하러 왔다.
