# AI 에이전트의 경쟁력은 모델이 아니라 통제 시스템에서 나온다

더 강한 모델을 붙이면 더 믿을 만한 에이전트가 될까. 한 번의 답변은 좋아질 수 있다. 그러나 사람이 자리를 비운 뒤에도 비용을 제한하고, 실패를 드러내고, 증거가 없으면 완료를 거부하는 능력은 모델 크기만으로 생기지 않는다.

Avid가 2026년 7월 공개한 에이전트 스택 가이드의 핵심은 특정 모델 조합이 아니다. **자율 실행의 품질은 모델을 둘러싼 종료 조건·검증·기록·권한 구조에서 결정된다.** 이 글은 그 주장을 실무에서 재사용할 수 있는 원칙으로 다시 정리하고, 그대로 받아들이기 어려운 수치와 일반화는 걷어낸다.

## 챗봇에서 루프로

일반적인 AI 사용은 사람이 질문을 입력하고 답을 확인하는 한 번의 상호작용이다. 에이전트는 다르다. 목표를 받고 파일을 읽고, 코드를 수정하고, 테스트하며, 실패하면 다시 시도한다. 이 반복 제어가 [[루프-엔지니어링]]이다.

```text
목표
  ↓
작업 수행
  ↓
환경에서 검증
  ├─ 통과 → 종료
  ├─ 실패 → 제한 안에서 재시도
  └─ 불능 → 중단하고 인간에게 전달
```

이 구조에서 가장 위험한 부분은 실패 자체가 아니라 끝이 정의되지 않은 실패다. “성공할 때까지 계속하라”는 지시는 자율적으로 들리지만, 실제로는 반복 횟수와 비용의 상한이 없는 프로그램이다.

따라서 모든 루프에는 두 종류의 끝이 필요하다.

| 종료 종류 | 예시 |
|---|---|
| 정상 종료 | 테스트 통과, 요구사항 체크 완료, 배포 상태 정상 |
| 비정상 종료 | 반복 횟수 초과, 예산 소진, 같은 오류 반복, 시간 초과 |

중단 조건은 비용 절약 장치에 그치지 않는다. 실패의 최대 반경을 미리 정하는 안전장치다.

## 완료를 의견에서 사실로 바꾸기

“구현을 완료했습니다”는 증거가 아니다. “엄격하게 검토했고 문제가 없습니다”도 마찬가지다. 둘 다 모델이 만든 문장이다.

완료는 환경에서 관찰되는 사실이어야 한다.

- 테스트 프로세스가 종료 코드 0을 반환했다.
- 타입 검사와 정적 분석이 통과했다.
- 최신 변경분을 대상으로 한 리뷰 결과가 저장됐다.
- 요구사항별 확인 항목이 기록됐다.
- 배포 환경의 상태 확인 요청이 성공했다.

이 원칙은 [[에이전트-평가-evals]]를 실행 루프 안으로 가져온다. 모델의 자신감이 아니라 외부 판정기가 작업을 닫는다. Avid는 이를 “판정은 의견이고, 행은 증거다”라는 말로 요약한다. 리뷰 문장 두 개보다 테스트 결과 한 행이 더 강한 이유다.

다만 테스트 통과가 제품의 올바름 전체를 증명하지는 않는다. 잘못된 요구사항을 정확히 구현하거나 테스트에 없는 사용성 문제를 만들 수 있다. 자동 검증은 인간을 제거하는 기술이 아니라 인간이 다시 볼 범위를 줄이는 기술이다.

## 작성자와 검증자의 컨텍스트를 분리하라

같은 에이전트가 구현과 검증을 모두 맡으면 작성 과정에서 생긴 가정이 리뷰에도 이어진다. “방금 만든 코드를 엄격하게 검토하라”고 말해도 컨텍스트는 공유된다.

더 나은 구조는 책임과 컨텍스트를 분리한다.

| 역할 | 책임 |
|---|---|
| 계획자 | 작업을 작은 단위와 검증 조건으로 분해 |
| 실행자 | 계획에 따라 실제 변경 생성 |
| 검토자 | 작성 과정 없이 요구사항과 변경분만 비교 |
| 게이트 | 테스트·정적 검사·증거 기록을 기계적으로 확인 |
| 인간 | 방향·위험·최종 병합 판단 |

다른 회사의 모델을 검토자로 쓰는 교차 모델 리뷰는 선택지이지 필수 조건은 아니다. 더 중요한 것은 새로운 컨텍스트, 별도의 체크리스트, 읽기 전용 권한이다. 서로 다른 모델도 같은 잘못된 전제를 받으면 같은 방향으로 틀릴 수 있다.

## 가장 강한 모델을 모든 일에 쓰지 않는다

원문은 최상위 모델을 계획·복잡한 진단·평가처럼 판단의 가치가 큰 지점에 두고, 반복적인 실행은 더 저렴한 모델에 맡긴다. 이 방향은 합리적이지만 가격표만 보고 모델을 고르면 안 된다.

```text
실제 작업 비용
= 호출당 비용 × 성공까지 필요한 호출 횟수
+ 재작업 비용
+ 검증 비용
+ 에이전트 간 조정 비용
```

OpenAI는 GPT-5.6을 Sol·Terra·Luna 세 계층으로 제공한다. 공개 당시 API 가격은 입력·출력 100만 토큰당 Sol 5·30달러, Terra 2.5·15달러, Luna 1·6달러다. 캐시 읽기에는 90% 할인이 적용된다. 반복되는 시스템 프롬프트와 도구 정의를 안정적인 접두부로 유지하는 일이 곧 비용 설계가 된 셈이다.

그렇다고 라우터를 추가하면 자동으로 좋아지는 것은 아니다. LLMRouterBench는 40만 건이 넘는 사례와 33개 모델을 통합 평가했고, 일부 최신 방법과 상용 라우터가 단순 기준선을 안정적으로 넘지 못했다고 보고했다. 라우터도 제품이다. 자체 업무 데이터에서 가장 좋은 단일 모델보다 낫다는 사실을 먼저 증명해야 한다.

실무 순서는 다음과 같다.

1. 적절한 단일 모델로 품질·비용 기준선을 만든다.
2. 실제 작업과 기계적 채점 기준을 모은다.
3. 저렴한 모델이 충분히 처리하는 작업 유형을 찾는다.
4. 작업 시작이나 리뷰 인계처럼 명확한 경계에서만 모델을 바꾼다.
5. 총비용과 성공률이 실제로 개선됐는지 다시 측정한다.

라우터가 기준선을 이기지 못하면 삭제하는 편이 낫다.

## 멀티 에이전트는 조건부 병렬화다

에이전트 수를 늘린다고 시스템이 자동으로 더 똑똑해지지는 않는다. 각 에이전트가 같은 문서를 다시 읽고 결과를 요약해 전달하면 토큰 사용량과 정보 손실 지점만 늘어난다.

동일한 추론 토큰 예산에서 다단계 추론을 비교한 2026년 연구에서는 단일 에이전트가 여러 멀티 에이전트 구조와 같거나 더 좋은 결과를 보였다. 반대로 다른 연구에서는 토론과 혼합 에이전트 구조가 일부 단일 에이전트 전략보다 계산 효율이 높았다. 과제와 비교 대상이 다르므로 어느 한쪽을 보편 법칙으로 읽으면 안 된다.

멀티 에이전트가 강한 자리는 독립적인 하위 작업을 동시에 진행할 수 있을 때다. Anthropic은 16개 에이전트와 약 2,000회의 Claude Code 세션, 약 2만 달러의 API 비용으로 10만 줄 규모 C 컴파일러를 구축했다. 이것은 멀티 에이전트의 가능성을 보여주는 프로젝트 사례이지, 일반적인 생산성 배수를 증명하는 실험은 아니다.

멀티 에이전트는 다음 조건에서 도입할 만하다.

- 목표를 서로 독립적인 하위 작업으로 분리할 수 있다.
- 각 결과에 기계적으로 확인할 완료 조건이 있다.
- 에이전트별 작업 공간을 분리할 수 있다.
- 병렬화로 절약하는 시간이 조정 비용보다 크다.
- 최종 통합과 병합은 순차적으로 수행한다.

하나의 깊은 추론 사슬을 여러 에이전트에게 억지로 나누면 병렬 컴퓨팅이 아니라 손실이 있는 전달 게임이 되기 쉽다.

## 트레이드오프: 통제가 곧 복잡성이다

게이트·라우터·리뷰어·증거 저장소를 추가할수록 실패를 더 잘 드러낼 수 있지만, 유지할 시스템도 늘어난다. 작은 저장소에서 하루 한 번 정해진 작업을 실행하는 정도라면 단일 에이전트와 테스트 스크립트로 충분할 수 있다.

또한 기계적 검증은 측정 가능한 것만 강화한다. 테스트가 쉬운 항목만 최적화하고 제품 방향·사용자 경험·법적 위험을 놓칠 수 있다. 모델을 교차 검증자로 배치해도 공통 데이터나 공통 벤치마크에서 온 편향까지 사라지지는 않는다.

그래서 원문의 구체적 휴리스틱을 법칙처럼 가져오면 안 된다. “단계가 올라갈 때 비용이 10배 증가한다”, “입력과 출력의 비가 100대 1이다”, “20회 실행에서 95%를 통과하면 자동 병합한다” 같은 숫자는 조직의 업무와 위험 수준에서 다시 측정해야 한다.

## 현실적인 도입 순서

처음부터 거대한 에이전트 공장을 만들 필요는 없다. 네 단계로 시작할 수 있다.

### 1. 완료와 중단 조건을 함께 적는다

```yaml
task: 사용자 세션 저장소 마이그레이션
done_when:
  - npm test -- tests/session
  - npm run typecheck
  - npm run lint
abort_when:
  - 동일 오류 2회 반복
  - 5회 수정 후에도 테스트 실패
  - 예상 비용 10달러 초과
```

### 2. 단일 실행 루프를 만든다

한 번에 작업 하나만 수행한다. 상태는 긴 대화 기록이 아니라 작업 파일·커밋·테스트 결과에 남긴다. 이것이 [[하네스-엔지니어링]]의 최소 형태다.

### 3. 독립 리뷰를 추가한다

변경을 작성하지 않은 새 세션이 요구사항과 diff만 검토하게 한다. 최종 판정은 리뷰 문장이 아니라 테스트 스크립트가 내린다.

### 4. 측정한 뒤에 라우팅과 병렬화를 추가한다

단일 에이전트의 비용과 성공률이 쌓이기 전에는 복잡한 라우터나 스웜을 만들지 않는다. 반복적이고 저위험이며 검증 가능한 작업이 충분히 발견된 뒤 확장한다.

## 인간의 자리는 마지막까지 남는다

에이전트가 목표를 효율적으로 달성해도 그 목표가 여전히 가치 있는지는 별도 문제다.

- 이 기능을 지금 만드는 것이 맞는가?
- 유지보수 비용보다 실제 가치가 큰가?
- 테스트가 놓친 사용자 피해는 없는가?
- 법률·보안·브랜드 위험을 감수할 만한가?
- 기술적으로 성공했지만 제품 전략으로는 실패한 것은 아닌가?

자동화가 강해질수록 인간의 역할은 모든 작업을 직접 수행하는 데서 목표를 선택하고 위험을 승인하며 시스템 규칙을 바꾸는 쪽으로 이동한다.

## 결론

좋은 AI 에이전트 시스템은 가장 강한 모델을 보유한 시스템이 아니다. 아무도 지켜보지 않을 때도 비용을 제한하고, 오류를 드러내며, 증거가 없으면 완료를 거부하는 시스템이다.

모델은 계속 바뀐다. 종료 조건, 독립 검증, 증거 기록, 기준선을 이긴 뒤에만 추가하는 복잡성은 더 오래 남는다. [[에이전틱-엔지니어링]]의 경쟁력은 모델 선택에서 시작할 수 있지만, 운영 가능한 [[위키-하네스|하네스]]에서 완성된다.
