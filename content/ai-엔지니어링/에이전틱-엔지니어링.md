**에이전틱 엔지니어링(Agentic Engineering)** 은 AI 에이전트를 실제로 설계·구축·운영하는 공학 분야다. 프롬프트·컨텍스트·하네스·루프를 하나의 실천으로 묶은 [[ai-엔지니어링-4계층]]의 상위 우산 개념으로, "좋은 프롬프트를 쓰는 법"이 아니라 "확률적 모델을 신뢰할 수 있는 시스템으로 감싸는 법"을 다룬다.

## 정의 — 에이전트 = 모델 + 하네스

기초 관점은 **에이전트 = 모델 + [[하네스-엔지니어링|하네스(harness)]]**. 모델은 세션 간 기억이 없어서(each new session begins with no memory) 여러 컨텍스트 윈도우에 걸쳐 작동하려면 스캐폴딩 — 초기화/코딩 에이전트, 구조화 파일, git, 셋업 스크립트 — 이 필요하다. 하네스가 그 발판이다.

그 위에서 핵심 축은 [[컨텍스트-엔지니어링|컨텍스트 엔지니어링]]이다. [[프롬프트-엔지니어링]]의 자연스러운 진화로, 추론 시점마다 최적 토큰 집합을 반복 큐레이션한다. 프롬프트가 일회성 작성이라면, 컨텍스트 엔지니어링은 매 스텝 큐레이션이 발생한다.

## 컨텍스트는 유한 자원 — 컨텍스트 로트

컨텍스트 윈도가 크다고 좋은 게 아니다. **토큰 수가 늘수록 정확한 회상 능력이 떨어지는 "컨텍스트 로트(context rot)"** 가 나타난다(Chroma의 18개 프론티어 모델 실험, Stanford lost-in-the-middle로 독립 확인). 그래서 핵심 컨텍스트 관리 기법:

- **컴팩션(compaction)**: 대화를 요약하고 새 컨텍스트로 재시작
- **구조화된 노트**: 외부 메모리로 상태 지속
- **저스트인타임 검색**: 경량 식별자로 런타임에 동적 로드
- **서브에이전트**: 하위 작업을 격리해 1,000–2,000 토큰 압축 요약만 반환

장기 실행 에이전트에선 컴팩션만으론 부족하고, 지속 상태를 **외부 산출물(progress 파일·git 이력·feature list)** 로 저장해 새 에이전트가 읽게 해야 한다. (이 위키의 "임시 clone → 작업 → push → 삭제" + git 이력 흐름이 사실상 이 패턴의 축소판이다.)

## 핵심 설계 패턴

학계 서베이는 에이전트 역량을 **추론(reasoning)·행동(acting)·상호작용(interacting)** 세 계열로 정리한다.

- **추론**: CoT, 자기반성(reflection), 탐색 트리, RAG
- **행동**: 도구 사용(tool use), API 호출, VLA
- **상호작용**: 멀티에이전트 롤플레잉, 사회 시뮬레이션

기반 루프는 **ReAct** — 추론 추적과 도구 행동을 교차 배치한다(추론이 계획·예외처리, 행동이 외부 인터페이스). 다만 ReAct식 반응형 루프는 점진적 단계 결정 탓에 복잡한 다중 도구 작업에서 **국소 최적 함정(local optimization trap)** 에 빠지기 쉬워, 복잡한 질의를 **DAG 실행 계획으로 바꾸는 플래너 중심(plan-execute)** 패러다임으로 보완된다. 실무는 대개 둘의 하이브리드(계획 + 국소 ReAct)다. 반복·자기교정 축은 [[루프-엔지니어링]] 참고.

## 평가·신뢰성 — 왜 어려운가

에이전트 평가는 LLM 단독 평가보다 **근본적으로 더 어렵다**. 동적·상호작용 환경에서 확률적으로 동작하므로 결정론적 소프트웨어 테스트로는 부족하다.

- **단일 실행(pass@1) 벤치마크는 프로덕션 신뢰성을 체계적으로 과대평가**한다. 작업 설명을 살짝 섭동(perturbation)만 해도 성능이 유의하게 떨어진다(한 실험: pass@1 96.9% → 88.1%).
- 새로 등장한 프레임워크:
  - **CLASSic** — cost·latency·accuracy·security·stability 5차원으로 아키텍처 선택과 실패 모드(환각·무한루프·프롬프트 인젝션)를 연결
  - **Reliability Surface** — 일관성(pass^k)·강건성·결함내성 3차원
  - **Agent-as-a-Judge** — 최종 출력만이 아니라 다른 에이전트의 전체 행동 체인을 평가. 단 원 논문의 결론은 인간 감독을 **대체가 아니라 보완**한다는 것
  - **MAST** — 멀티에이전트 실패를 14개 모드 → (1)시스템 설계결함 (2)에이전트 간 정렬 실패 (3)작업 검증 실패 3범주로 분류

> ⚠️ 위 신뢰성 수치(섭동 하락, 오류 전파확률 약 0.62, 런타임 인터셉터 −23%p 등)는 대부분 **2025–2026 단일 저자·비심사 소규모 프리프린트** 기반이다. 방향성은 신뢰하되 정확한 수치를 일반화하면 안 된다.

## 빌드 라이프사이클·프로덕션 (개관)

에이전틱 엔지니어링은 전통 SDLC를 **생성-검증(generate-verify) 루프 + eval 주도 개발**로 바꾼다. 구체 사례는 [[agents-cli-lifecycle-sdlc]], 이를 구현한 도구 사례는 [[google-agents-cli]]. 프로덕션에서 프로토타입과 갈리는 축은 **신뢰성·관측성(observability)·안전성**이다 — 구조화 텔레메트리(trace/span, 토큰·비용 분해, 실패 분류), 실시간 가드레일, 권한 최소화와 프롬프트 인젝션 방어.

## 2025–2026 현황

대표 프레임워크로 LangGraph, Claude Agent SDK, OpenAI Agents SDK, Google ADK/[[google-agents-cli|agents-cli]] 등이 경쟁하며 오케스트레이션·상태관리·멀티에이전트 모델에서 갈린다. (프레임워크별 정밀 비교는 대부분 블로그 출처라 이 노트에선 개관만 담는다.)

## 남은 질문

- 프레임워크별(LangGraph/Claude/OpenAI/Google) 아키텍처·상태관리 차이와 적합 작업
- 관측성·토큰비용·보안(권한 최소화·프롬프트 인젝션 방어)의 구체적 엔지니어링 실무
- 최신 프리프린트 신뢰성 수치가 대규모·독립 재현에서도 유지되는가
