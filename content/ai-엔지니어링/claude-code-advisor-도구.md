당신은 큰 리팩터링을 Sonnet에게 맡긴다. 400턴 중 380턴은 파일을 열고, 함수를 옮기고, 테스트를 돌리는 기계적인 작업이다. 그런데 딱 3–4개의 순간 — 어떤 설계로 갈지 정하는 순간, 같은 에러가 다섯 번째 반복되는 순간, "다 됐다"고 선언하기 직전 — 이 결과 전체를 좌우한다. 이 순간들 때문에 400턴 전부를 Opus로 돌려야 할까?

Claude Code의 **advisor(어드바이저) 도구**는 이 질문에 "아니오"라고 답한다. 메인 모델은 평범한 턴을 값싸게 처리하고, 판단이 필요한 핵심 순간에만 더 강한 두 번째 모델에게 전체 대화를 넘겨 조언을 받는다.

## 핵심 주장: 상시 구동과 버티기 사이의 세 번째 선택지

지금까지 모델 선택은 이분법이었다. 강한 모델을 처음부터 끝까지 돌리거나(비싸다), 약한 모델로 버티거나(품질이 흔들린다). advisor는 그 사이를 가른다. **프런티어급 추론을 필요한 순간에만 적용하고 나머지 실행은 실행자 모델의 비용으로 유지한다.**

Anthropic이 공개한 벤치마크가 이 구조의 효과를 뒷받침한다. Sonnet에 Opus 어드바이저를 붙였더니 SWE-bench Multilingual 점수가 2.7%포인트 오르면서 태스크당 비용은 오히려 11.9% 줄었다. Haiku에 Opus 어드바이저를 붙인 경우 BrowseComp 점수가 19.7%에서 41.2%로 두 배 넘게 뛰었다. Haiku+Opus 어드바이저 조합은 Sonnet 단독보다 점수는 29% 낮지만 태스크당 비용은 85% 저렴하다.

한 문장으로: advisor는 near-Opus 지능을 near-Sonnet 가격에 근접시키는 장치다. 단, **실험적 기능이다.** Claude Code v2.1.98 이상 + Anthropic API 전용이며, 동작·과금·가용성이 바뀔 수 있다.

advisor는 [[하네스-엔지니어링]] 관점에서 봐야 정확히 이해된다. 모델을 감싸는 실행 발판(하네스)이 모델에게 쥐여주는 또 하나의 도구, 곧 "막히면 더 똑똑한 동료에게 물어보라"는 능력이 advisor다. 모델 자체를 바꾸는 게 아니라, 모델이 쓸 수 있는 도구를 하나 더 준다.

## 어떻게 동작하나

advisor는 Anthropic 인프라에서 실행되는 **서버 사이드 도구**(server tool)다. 클라이언트가 아니라 Anthropic 서버가 조언 모델을 실행하고 결과를 되돌려준다.

호출 흐름은 이렇다.

1. 메인 모델(실행자)이 조언이 필요하다고 판단하면 `advisor`라는 이름의 도구 호출을 낸다. 흥미롭게도 이 호출의 입력은 **항상 비어 있다.** 실행자는 "지금 물어볼 타이밍"이라는 신호만 보내고, 무엇을 보낼지는 서버가 정한다.
2. 서버가 실행자의 **전체 대화 기록**(transcript)을 조언 모델에게 인용 컨텍스트로 넘긴다. 시스템 프롬프트, 도구 정의, 이전 턴과 도구 결과, 이번 턴에서 실행자가 지금까지 생성한 텍스트까지 전부 포함된다.
3. 조언 모델은 Anthropic이 제공하는 자체 시스템 프롬프트 아래에서 별도 추론을 돌려 계획이나 방향 수정을 반환한다.
4. 메인 모델이 그 조언을 반영해 작업을 이어간다.

여기서 두 가지가 핵심이다. 첫째, **호출 타이밍은 전적으로 메인 모델이 정한다.** 규칙 기반이 아니라 모델 주도다. 둘째, 조언 모델은 요약본이 아니라 **전체 대화**를 본다 — [[컨텍스트-엔지니어링]] 관점에서 조언 품질을 위해 컨텍스트를 아끼지 않는 설계다. 대신 그만큼 매 호출이 무겁다(뒤의 비용·캐시 절 참조).

세션 중에는 트랜스크립트에 조언 모델 이름과 함께 `Advising` 줄이 뜨고, 반환되면 검토 완료가 표시된다. `Ctrl+O`로 조언 전문을 펼칠 수 있다. 메인 모델은 대체로 조언을 따르되, 자기 증거가 조언과 충돌하면(권장한 단계가 실패했거나 파일 내용이 조언과 어긋나면) 무조건 따르지 않고 충돌을 드러낸다.

## 페어링 규칙: 어드바이저는 메인보다 같거나 강해야 한다

advisor 설계에서 가장 논리적인 부분은 모델 페어링 제약이다. **조언 모델은 메인 모델보다 최소한 같거나 더 유능해야 한다.** 자기보다 약한 모델에게 조언을 구하는 건 의미가 없으므로, 시스템이 이를 강제한다.

| 메인 모델 | 허용되는 어드바이저 | 논리 |
|---|---|---|
| Haiku 4.5 | Fable, Opus, Sonnet | Haiku는 조언을 부를 수는 있지만 조언자는 될 수 없다 |
| Sonnet 4.6 | Fable, Opus, Sonnet | |
| Sonnet 5 | Fable, Opus, Sonnet 5 | Sonnet 4.6 어드바이저는 거부 — 구세대라 더 약하다 |
| Opus 4.6 | Fable, Opus, Sonnet 5 | Sonnet 5와 Opus 4.6이 동급으로 평가돼 Sonnet 5 어드바이저 허용 |
| Opus 4.7 이상 | Fable, Opus 4.7, Opus 4.8 | Opus 4.7/4.8이 동급 — 서로를 어드바이저로 허용. Opus 4.6·Sonnet 5는 거부 |
| Fable 5 (v2.1.170+) | Fable | Fable이 Opus/Sonnet보다 상위 티어라 Fable만 허용 — 다만 Fable은 현재 어드바이저로 제공되지 않아 **Fable 5 메인은 어드바이저 없이 돈다** |

> **2026-07-14 갱신 (v2.1.210).** Claude Code는 현재 **Fable 5를 어드바이저로 제공하지 않는다.** Fable 5 접근 권한이 있는 조직에서도 `/advisor` 피커엔 `Fable 5 (temporarily unavailable)`로 흐리게 뜨는 선택 불가 행으로 표시되고, `/advisor fable`과 `--advisor fable`은 거부된다. Fable은 능력 검사 자체는 통과하므로 위 표의 Fable 항목은 **롤아웃이 옵션을 되돌린 뒤에 적용된다.** 복귀 시점은 원격 롤아웃이 통제한다. Fable 5를 **메인 모델**로 쓰는 것은 영향받지 않는다. (이 노트는 2026-07-09 발행 — 발행 시점엔 Fable 어드바이저가 제공됐다.)

이 표에서 CTO가 읽어야 할 것은 "동급 허용" 규칙이다. Opus 4.7 메인이 Opus 4.8을 어드바이저로 받는 이유는 둘이 같은 급으로 평가되기 때문이다. 즉 페어링 판정은 단순 세대 비교가 아니라 Anthropic이 매긴 **능력 서열**을 따른다. 새 모델이 나오면 이 서열표가 갱신되고, 어제 유효하던 조합이 오늘 거부될 수 있다. 실제로 발행 닷새 뒤 Fable 어드바이저가 목록에서 내려갔다 — 서열이 바뀌어서가 아니라 서버 사이드 실패 때문이었지만, 표가 고정된 사실이 아니라는 점은 같다.

주의할 실무 포인트:

- 어드바이저는 `opus`·`sonnet` 별칭(최신 버전으로 해석) 또는 `claude-opus-4-8` 같은 전체 모델 ID로 지정한다. `fable` 별칭은 롤아웃이 Fable을 어드바이저 옵션으로 되돌린 뒤에 쓸 수 있다.
- **서브에이전트는 설정된 어드바이저를 상속**하되, 자기 자신의 모델 기준으로 페어링을 다시 검사한다. 메인에선 어드바이저가 안 붙어도 조건을 만족하는 서브에이전트에선 붙을 수 있다.
- 전송 전 검증에서 어드바이저가 메인보다 약하면 아예 부착되지 않고 `/advisor` 출력과 알림에 표시된다.
- 조직의 `availableModels` 허용 목록이 저장된 어드바이저를 제외하면, 허용된 모델을 고를 때까지 어드바이저가 호출되지 않는다.
- 현재 메인 모델이 advisor를 지원하지 않아도 선택은 저장되며, `/model`로 호환 메인 모델로 바꾸는 순간 활성화된다.

## 활성화하는 세 가지 방법

| 방법 | 범위 | 쓰임 |
|---|---|---|
| `/advisor` 명령 | 세션 중 설정, 기본값으로 저장(`advisorModel`, 세션 간 지속) | 인자 없이 치면 선택기, `/advisor opus`로 바로 지정 |
| `advisorModel` 설정값 | 설정 파일에 `{ "advisorModel": "opus" }` | 프로젝트/사용자 기본 어드바이저 고정 |
| `--advisor` 실행 플래그 | 단일 세션, 설정값보다 우선 | `claude --advisor opus`. 메인 미지원이거나 허용 목록에서 제외되면 에러로 종료 |

끄는 것도 세 가지다. `/advisor off`(또는 선택기의 "No advisor")로 저장된 값을 지우고, 환경변수 `CLAUDE_CODE_DISABLE_ADVISOR_TOOL=1`로 완전히 비활성화한다(이 경우 `--advisor`는 받아들여지되 효과가 없어 스크립트가 깨지지 않는다).

## 비용 구조: 결정 지점에만 부과된다

advisor의 비용 논리는 한 문장으로 요약된다. **조언 모델의 토큰은 매 턴이 아니라 결정 지점에서만, 조언 모델의 요율로 별도 과금된다.**

호출 한 번마다 전체 대화가 조언 모델에게 전달되므로, 그 호출은 조언 모델의 입력·출력 요율로 계산돼 메인 모델 사용량에 **더해진다.** API 과금이면 조언 토큰이 조언 모델 요율로, 구독이면 플랜 사용 한도에 반영된다. `/usage` 세션 합계에도 잡힌다.

핵심은 호출 빈도다. 400턴짜리 작업에서 조언이 3–4번만 불린다면, 그 3–4번의 프리미엄 비용은 전체를 강한 모델로 돌리는 비용에 비하면 작다. 게다가 조언 모델의 출력은 대체로 짧다 — 블로그와 API 문서 모두 텍스트 기준 400–700 토큰을 전형값으로 제시한다. 여기에 thinking까지 포함하면 총 1,400–1,800 토큰이고, 난도 높은 추론에선 더 길어진다. 워크로드에 따라 편차가 크다는 뜻이다.

그래서 "빠른 메인 + 강한 어드바이저"가 강한 모델을 처음부터 끝까지 돌리는 것보다 대체로 싸다. 앞서 인용한 SWE-bench의 비용 11.9% 감소가 이 구조의 산술적 결과다 — 품질은 올라가고 총비용은 내려간 것.

비용을 통제하려면 호출 횟수를 관리해야 한다. Claude Code에는 호출을 강제하거나 상한을 거는 설정이 없고, 프롬프트로만 조절한다("계속하기 전에 어드바이저와 상의하라"로 늘리거나, 반대로 줄이거나). 참고로 저수준 API에는 요청당 상한 `max_uses`와 조언 출력 상한 `max_tokens`(최소 1024) 파라미터가 있지만, 이들은 Claude Code CLI에는 노출되지 않는다.

## 캐시 영향: 메인 캐시는 지키고, 어드바이저 읽기는 매번 새로 읽는다

비용을 논할 때 [[프롬프트-엔지니어링|프롬프트 캐시]]를 빼놓을 수 없다. advisor는 여기서 비대칭적으로 동작한다.

**메인 모델 쪽은 안전하다.** 세션 중 `/advisor`를 켜거나 끄는 것은 메인 모델의 프롬프트 캐시를 무효화하지 않는다. 모델이나 사고 강도(effort)를 바꾸면 캐시가 날아가지만, advisor 토글은 캐시된 프리픽스를 그대로 둔다. 조언이 반환되면 그 지침은 트랜스크립트의 일부로 이후 턴에서 캐시된다. 즉 켜고 끄는 데 캐시 페널티가 없다.

**어드바이저 모델 쪽은 다르다.** Claude Code 문서 기준, 조언 모델 자신의 읽기는 캐시되지 않는다 — 매 호출이 전체 트랜스크립트를 처음부터 다시 처리하고, 호출 간 재사용이 없다. 조언이 열 번 불리면 매번 그 시점까지의 전체 대화를 새로 읽는다는 뜻이다. 대화가 길수록 조언 한 번의 입력 비용이 커진다.

여기 실무자가 알아야 할 미묘한 층이 하나 더 있다. **저수준 Anthropic API에는 어드바이저 쪽 캐싱을 켜는 `caching` 옵션이 있다.** API 문서에 따르면 조언 프롬프트는 N번째 호출이 (N-1)번째 프롬프트에 한 세그먼트만 덧붙인 형태라 프리픽스가 안정적이고, 캐싱을 켜면 두 번째 호출부터 델타만 지불한다. 다만 캐시 쓰기 비용 때문에 조언이 2회 이하면 손해, 약 3회에서 손익분기, 그 이상이면 이득이다. 긴 에이전트 루프엔 켜고 짧은 작업엔 끄라는 게 공식 권고다. 단, 이 옵트인은 API를 직접 쓸 때 이야기다 — Claude Code CLI는 어드바이저 캐싱을 기본으로 끄며(그래서 위의 "캐시 안 됨"), 켜는 것은 저수준 API 사용자의 몫이다.

정리하면, CTO 관점의 캐시 계산은 두 축이다. 메인 캐시는 advisor를 켜도 손해가 없다. 어드바이저 호출은 긴 대화일수록 비싸지므로, 호출 빈도와 대화 길이가 그대로 조언 비용을 결정한다.

## 유사 기능과 비교: 강한 모델을 언제 돌리나

advisor를 제대로 쓰려면 "강한 모델을 언제 투입하는가"라는 같은 문제를 푸는 다른 도구들과 나란히 놓아야 한다.

| 방식 | 강한 모델이 도는 시점 | 시작 주체 |
|---|---|---|
| advisor 도구 | 작업 중 결정 지점에서만 | 메인 모델이 필요할 때 스스로 호출 |
| opusplan | 플랜 모드 동안 Opus로 계획, 실행은 Sonnet으로 전환 | 사용자가 플랜 모드 진입 |
| 서브에이전트(모델 지정) | 위임된 하위 작업 전체 동안 | 메인이 위임하거나 사용자가 호출 |
| `/model` | 이후 모든 턴 | 사용자가 전환 |

차이의 핵심은 **입도(granularity)와 주도권**이다.

- `/model`은 가장 거칠다. 전환 이후 모든 턴이 그 모델이다.
- **opusplan**은 "계획은 강하게, 실행은 싸게"를 시점으로 가른다. 단 계획 단계 전체가 Opus다. advisor는 계획 단계 안에서도 필요한 순간에만 강한 모델을 스팟으로 부른다. 결이 다르다.
- **서브에이전트**는 하위 작업을 통째로 다른 모델에 위임한다 — 컨텍스트가 분리되는 대신 그 작업 내내 강한 모델이 돈다. advisor는 컨텍스트를 분리하지 않고(전체 대화를 넘긴다) 메인의 흐름 안에서 조언만 주입한다.
- **advisor**만이 유일하게 "모델이 스스로, 순간 단위로" 강한 지능을 당겨온다.

이 비교는 [[에이전틱-엔지니어링]]의 오케스트레이션 관점을 뒤집는다. 전통적 계층 구조는 큰 오케스트레이터가 작은 워커들에게 일을 쪼개 내린다. advisor는 반대다 — **작은 실행자가 운전대를 잡고, 막히는 순간에만 위로 에스컬레이션한다.** Anthropic 블로그가 "advisor 전략"이라 부르는 이 역전이 비용 곡선을 실행자 수준으로 눌러 앉히는 이유다.

## 트레이드오프: 언제 쓰지 말아야 하나

advisor는 만능이 아니다. 안 맞는 경우를 분명히 하는 게 도입 판단의 절반이다.

- **짧은 작업엔 오히려 손해다.** 계획할 것이 거의 없는 단발성 Q&A, 몇 턴짜리 작업에선 조언 호출의 오버헤드가 이득을 넘는다. 어드바이저 캐싱 손익분기(약 3회 호출)를 넘지 못하는 작업이 여기 해당한다.
- **매 턴이 최고 모델을 요구하는 작업이면 그냥 메인을 바꿔라.** 결정 지점만이 아니라 모든 턴이 프런티어급 추론을 필요로 한다면, advisor로 매 턴 조언을 부르느니 `/model`로 메인을 강한 모델로 올리는 게 맞다. advisor는 "대부분의 턴은 평범하고 계획 품질이 결과를 가르는" 분포에서만 이득이다.
- **비결정성을 받아들여야 한다.** 호출 시점은 모델이 정하고, Claude Code에는 강제·상한 설정이 없다. 비용 예측 가능성이 중요한 팀에는 이 점이 불편할 수 있다. 프롬프트로 유도할 뿐 보장은 없다.
- **플랫폼이 제한적이다.** Anthropic API 전용이다. Amazon Bedrock, Google Cloud Agent Platform, Microsoft Foundry에서는 쓸 수 없다. `ANTHROPIC_BASE_URL`로 LLM 게이트웨이를 경유하는 경우, 게이트웨이가 서버 도구 호출을 그대로 전달하는지에 따라 갈린다.
- **실험적 기능이다.** 동작·과금·가용성이 바뀔 수 있다. API 쪽은 베타 헤더 `advisor-tool-2026-03-01`을 요구하는 베타 단계다. 프로덕션 파이프라인에 깊이 엮기 전에 이 점을 감안하라.

advisor의 "완료 전 독립 검토"는 [[에이전트-평가-evals]]와 통하는 대목이 있다. 자기 작업을 자기가 채점하면 보상 해킹에 빠지기 쉽다는 문제를, advisor는 더 강한 제3의 모델에게 완료 직전 대화를 통째로 검토시키는 방식으로 완화한다. 다만 이것은 정식 eval 파이프라인의 대체가 아니라 세션 내 즉석 검토라는 점을 혼동하지 말아야 한다.

## 실무 적용: 내일 무엇을 할까

1. **후보 워크로드를 골라라.** 길고 다단계이며, 대부분의 턴은 기계적이지만 계획·디버깅·완료 판정이 결과를 가르는 작업. 큰 리팩터링, 반복 에러가 있는 디버깅 세션, 완료 전 독립 검증이 필요한 태스크가 1순위다.
2. **메인을 한 단계 낮추고 어드바이저를 얹어라.** 지금 Sonnet으로 복잡한 작업을 한다면 Opus 어드바이저를 붙여 품질을 올리면서 총비용을 비슷하거나 낮게 유지한다. Haiku를 쓴다면 Opus 어드바이저로 지능을 끌어올리되, Haiku 단독보다는 비싸고 메인을 Sonnet/Opus로 올리는 것보다는 싸다는 점을 계산에 넣어라.
3. **첫 세션은 관찰 모드로.** `Ctrl+O`로 조언 전문을 열어 어떤 순간에 얼마나 자주 호출되는지, 조언이 실제로 방향을 바꾸는지 확인하라. `/usage`로 어드바이저 토큰이 세션 합계에서 차지하는 비중을 본다.
4. **호출 빈도를 프롬프트로 조율하라.** 조언이 과하면 줄이고, 핵심 결정에서 놓치면 "접근을 확정하기 전에 어드바이저와 상의하라"로 명시적으로 유도한다.
5. **결과는 워크로드마다 다르다.** 공식 문서와 블로그가 반복해 강조하듯, 벤치마크 수치는 참고치다. 당신의 실제 작업에서 A/B로 검증하라.

advisor는 [[하네스-자기개선]]과도 이어진다. 하네스가 모델에게 "막히면 더 강한 동료를 부를 권한"을 쥐여주는 것 자체가, 고정된 모델 성능을 실행 발판으로 끌어올리는 한 방법이기 때문이다. 모델을 바꾸지 않고도 결과 품질을 올리는 지렛대 — 그게 advisor의 정체다.
