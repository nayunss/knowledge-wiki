# 루프 엔지니어링 (Loop Engineering)

**같은 목표/프롬프트를 완료 조건에 닿을 때까지 반복 실행시키는 제어 흐름 설계.** 매 프롬프트를 손으로 쓰는 대신, 도구 호출 사이에 무엇을 할지·언제 자기 작업을 점검할지·언제 끝났다고 볼지를 설계한다. AI 엔지니어링 [[ai-엔지니어링-4계층]]에서 "시간" 축을 담당하며, 2026년 급부상한 신조어다.

> "에이전트에게 프롬프트하기를 멈춰라. 대신 당신을 대신해 프롬프트해줄 루프를 설계하라."

용어는 2026년 6월 한 주 사이에 세 사람이 거의 동시에 같은 것을 가리키며 굳었다. Peter Steinberger가 "이제 코딩 에이전트에 프롬프트할 게 아니라 그것을 프롬프트하는 루프를 설계하라"는 글을 올려 크게 퍼졌고, 비슷한 시점에 Boris Cherny(Anthropic·Claude Code 리드)가 "나는 더는 Claude에 프롬프트하지 않는다. Claude에 프롬프트하고 다음 할 일을 정하는 루프를 돌린다. 내 일은 루프를 쓰는 것"이라 말했으며, Addy Osmani(Google)가 이를 "Loop Engineering"이라는 글로 정리하며 이름 붙였다. 요지는 하나 — **자기 자신을 "프롬프트하는 사람" 자리에서 빼내고, 그 일을 대신할 시스템을 설계하는 것.** 위치가 엔진에서 엔진을 설계하는 사람으로 바뀐다.

## 에이전트의 기본 루프

에이전트를 단발 챗봇과 가르는 단 하나의 메커니즘은 **Reason → Act → Observe → Repeat** 순환이다. 목표를 추론하고, 도구를 호출·실행하고, 결과를 관측하고, 목표와 대조해 자기수정하고, 완료까지 반복한다. 전통 파이프라인이 사람이 단계를 미리 고정하는 정적 흐름이라면, 에이전틱 루프는 모델이 **런타임에 다음 단계를 정하므로** 적응·자기수정이 가능하다.

## 한 턴 = 5개 동작, 6개 부품 (Osmani 프레임)

"루프"는 헛도는 반복이 아니다. 한 턴은 구체적으로 **할 일을 찾고 → 에이전트에 넘기고 → 결과가 맞는지 검증하고 → 상태를 저장하고 → 다음을 예약한다.** 다섯 중 하나만 빠져도 루프는 돌지 않거나 제자리를 돈다.

| 동작(move) | 하는 일 | 실현하는 부품(part) |
|---|---|---|
| **Discovery(발견)** | 이번 턴에 무엇을 할지 스스로 찾는다(목록을 받는 게 아니라) | **스킬(Skill)** — 프로젝트 지식을 `SKILL.md`에 영구화 |
| **Handoff(넘김)** | 작업을 격리해 에이전트에 넘긴다 | **워크트리(worktree)** — 병렬 에이전트마다 독립 디렉터리 |
| **Verification(검증)** | 결과가 맞는지 **"아니오"라고 말한다** | **서브에이전트** — 작성자와 심판을 분리 |
| **Persistence(저장)** | 대화를 넘어 남는 곳에 상태를 쓴다 | **메모리** — 디스크의 markdown·보드 (+ 외부 연결은 **커넥터/MCP**) |
| **Scheduling(예약)** | 한 턴을 루프로 만든다(타이머·트리거) | **오토메이션** — 스케줄·이벤트로 스스로 시동 |

핵심 직관: **실수의 비용은 그것이 발각되기 전 살아남은 턴 수에 비례한다.** 같은 버그라도 프롬프트 층에선 즉시 보이고, 하네스 층에선 diff로 리뷰되지만, 루프 층에선 상태 파일에 사실처럼 기록돼 며칠간 그 위에 쌓인다. 루프는 구조상 "턴 수를 최대화하는 기계"라서, 검증자·휴먼 체크포인트·예산 상한이 전부 **실수와 발각 사이 거리를 줄이려고** 존재한다.

메모리 ≠ 컨텍스트: 컨텍스트는 이번 라운드에 보이고 리프레시로 비워지지만, 메모리는 라운드·날짜를 넘어 디스크에 남는다. "에이전트는 잊어도 레포는 잊지 않는다."

## Generator ↔ Evaluator — "아니오"라고 말할 검증자

루프에서 가장 어려운 건 돌리는 게 아니라 **안에 "아니오"라고 말할 무언가를 넣는 것**이고, 코드를 쓴 에이전트가 그 말을 제일 못 한다.

- **자기 채점은 자기 칭찬이 된다.** 방금 만든 걸 스스로 평가하라 하면 품질이 뻔히 별로여도 자신 있게 칭찬한다(Anthropic이 장기 실행 앱을 만들며 관찰). 똑똑함의 문제가 아니라, 코드를 쓴 컨텍스트가 이미 "왜 그렇게 썼는지"의 자기설득으로 가득 차 결과가 아니라 그 설득의 사슬을 보기 때문이다. 루프 안에선 매 라운드 스스로에게 고개를 끄덕여 품질에서 멀어진다.
- **작성자를 겸손하게 만들기보다 독립 심판을 까다롭게 튜닝하는 게 훨씬 쉽다.** 저자에게 자기 관점 밖으로 나가라 할 수는 없지만, **완전히 다른 지시(때로 다른 모델)를 받은 별도 에이전트**를 끼워 백지에서 보게 할 수는 있다. GAN(생성자↔판별자)에서 빌려온 구도다.
- **검증자는 읽지 말고 행동하게 하라.** 코드만 읽으면 "맞아 보이나"를 판단하지 "맞게 도나"를 못 본다. 프런트엔드라면 Playwright MCP를 물려 페이지를 열고·클릭하고·스크린샷을 찍어 DOM을 QA처럼 검사하게 한다. 기본 태도는 **신뢰가 아니라 의심** — "증명 전까지 이 코드는 고장 났다"고 가정시킨다.
- **최종 판정은 새 모델에게.** Claude Code `/goal`은 완료 조건을 주면 충족될 때까지 돌리되, **매 턴 뒤 작고 빠른 별도 모델**이 조건 충족을 확인한다. 큰 이체를 입력하는 사람과 승인하는 사람을 나누는 은행의 **maker–checker** 원칙을 정지 조건에 적용한 셈. ("배포를 재실행만 하는 `/loop`와 혼동하지 말 것.")

루프의 바닥은 검증자다. 생성자 수준이 "무엇을 만들 수 있나"를 정하면, 검증자 수준이 "무엇을 내보내지 않을 것인가"를 정한다.

## 5가지 실패 유형 (동작 하나씩 빠뜨리기)

다섯 실패는 다섯 동작이 하나씩 빠진 것과 정확히 대응한다.

- **끄덕이는 루프(Nodding)** — 검증 생략. 같은 에이전트가 쓰고 스스로 합격시켜, 그럴듯한 오류를 기계 속도로 쌓는다. 수백 턴 동안 한 번도 "아니오"가 없었다면 검증이 없다는 증거다. → generator/evaluator 분리.
- **건망증 루프(Amnesiac)** — 저장 생략. 결과가 컨텍스트 창에만 남아 다음 턴이 같은 일을 재발견·재작업·충돌한다. → 디스크 상태 파일.
- **수동 루프(Manual)** — 예약 생략. 네 동작이 좋아도 자동 시동이 없으면 그건 사람이 손으로 돌리다 잊는 스크립트다. 마지막 실행이 시연한 날이면 그 증상. → 진짜 트리거(타이머·이벤트).
- **눈먼 루프(Blind)** — 발견 생략. 매일 사람이 "이 버그 세 개 고쳐"라고 일을 떠먹인다. 무엇을 할지 고르는 게 대개 비싼 부분인데 그걸 자동화 못 한 것. → 발견 로직을 스킬로.
- **엉킨 루프(Tangled)** — 넘김 생략. 병렬 에이전트가 같은 디렉터리를 고쳐 편집이 충돌한다. 단일 에이전트일 땐 멀쩡하다 다섯이 동시에 도는 첫 아침에 터진다. → 작업당 워크트리 하나.

## 대표 형태 세 가지

### (a) 인터벌/스케줄 루프
일정에 따라 작업을 찾아 처리하는 주기 작업. 폴링·감시·정기 태스크에 쓴다. 예: Claude Code `/loop`(지정 간격 반복, 간격 생략 시 모델이 스스로 페이스 조절), `/schedule`(cron 기반 클라우드 루틴). "배포를 5분마다 확인" 같은 유형.

### (b) 자기참조 루프 = Ralph 기법
가장 단순한 형태는 `while :; do cat PROMPT.md | claude-code ; done` — 무한 bash 루프가 고정된 프롬프트 파일을 매번 다시 읽어 넣고, 디스크의 코드를 고치고, 다시 돈다. Geoffrey Huntley가 2025년 창안·대중화했고, 이름은 심슨의 Ralph Wiggum에서 땄다. 정밀함보다 끈기, 정제보다 반복이다.
- 원리는 **반복을 통한 결과적 일관성(eventual consistency)** — 한 번에 완벽할 필요 없이 완료 조건까지 반복을 강제한다.
- **파일시스템이 메모리**다. 대화 이력 대신 코드베이스·TODO·git 히스토리에 상태를 남기고, 매 반복은 **새 컨텍스트**로 시작해 이전 반복의 아티팩트를 마주친다. "한 반복에 한 태스크만"이 유효 컨텍스트 상한을 지키는 결정적 제약이다.
- Anthropic이 `claude-code`에 `ralph-wiggum` 플러그인으로 공식화했다. bash 무한 루프 대신 **Stop hook**으로 종료를 가로채 같은 프롬프트를 세션 안에 되먹인다(`/ralph-loop "<prompt>" --max-iterations <n> --completion-promise "<text>"`).
- Huntley는 엔지니어 무용론을 명확히 부정한다 — "Engineers are still needed."

### (c) 평가-수정 루프
테스트/eval이 통과할 때까지 반복. 미완료 태스크 선택 → 구현 → 테스트·타입체크·린트·CI·(UI면) 브라우저 검증 → 통과 시 커밋 → 상태·학습 기록 → 컨텍스트 리셋 후 반복. 검증이 실패하면 넘어가지 않고 그 에러 출력을 다시 먹여 자기수정을 유발한다. "루프는 빨간불에서 멈춰야 한다."

## LangChain의 4중 스택 루프 (loopcraft)

LangChain은 루프 엔지니어링을 "에이전트를 감싸 겹겹이 쌓는 중첩 루프"로 본다(Swyx의 "loopcraft — 루프를 쌓는 기예"). 핵심 원리는 <strong>바깥 루프가 안쪽 루프를 강화한다(outer loops enhance inner loops)</strong>는 것. "에이전트의 잠재력은 그 주위에 무엇을 쌓느냐에 있다."

| 루프 | 하는 일 | LangChain 프리미티브 |
|---|---|---|
| **1. 에이전트 루프** | 모델이 도구를 반복 호출해 과업 완료 — 작업을 자동화 | `create_agent` |
| **2. 검증 루프** | 출력을 루브릭에 대조(결정론적·LLM 그레이더), 피드백으로 재시도 — 품질 확보(지연·비용↑) | `RubricMiddleware`, `after_agent` 훅 |
| **3. 이벤트 구동 루프** | 웹훅·cron·메시지 트리거로 백그라운드 상시 실행 — 능동형 "심박(heartbeat)" 어시스턴트 | LangSmith Deployment, Fleet 채널 |
| **4. 힐 클라이밍 루프** | 프로덕션 트레이스를 분석해 **하네스 설정 자체를 개선**(프롬프트·도구·컨텍스트, 오픈웨이트면 파인튜닝 신호까지) | LangSmith Engine |

- 앞의 세 형태와 대응: 인터벌/스케줄 = 루프 3, Ralph·평가-수정 = 루프 1·2에 해당한다. **루프 4(힐 클라이밍)가 새 관점**이다 — 매 실행이 남기는 트레이스(모델 결정·툴콜·그레이더 피드백)를 고신호 신호로 삼아, 리턴 화살표가 맨 위로 되돌아가는 데 그치지 않고 **안쪽 에이전트 루프를 직접 고쳐 쓴다.** 즉 루프가 [[하네스-엔지니어링]]을 자동 개선하는 지점이다.
- 가치가 복리로 붙는 곳은 루프 3·4다 — 정적 설정을 넘어 적응하는 시스템으로 넘어간다.
- "자동화가 사람을 루프에서 빼는 건 아니다." 민감한 행동·그레이딩·승인·배포 결정마다 휴먼인더루프 체크포인트를 둔다.

## 실사례 — 개인부터 엔터프라이즈까지

규모는 천차만별이지만 뼈대는 같다: 트리거가 시동하고, 제약이 레일을 지키고, 끝에 사람 체크포인트가 있다. "자면서 돌린다"는 모델 세기가 아니라 이 뼈대의 견고함에 달렸다.

- **한 사람의 아침(Osmani의 triage 루프)**: 아침에 자동 시동 → triage 스킬이 어제 실패한 CI·열린 이슈·최근 커밋을 읽어 markdown/보드에 정리 → 처리할 항목마다 격리된 워크트리를 열어 한 서브에이전트가 수정 초안을, 다른 서브에이전트가 리뷰 → 커넥터가 PR을 열고 티켓을 갱신 → 못 다루는 건 인박스로, 상태 파일로 다음 날이 이어받는다. 자동화가 **거대한 지시 뭉치가 아니라 스킬을 호출**한다는 게 핵심.
- **Stripe의 Minions — 주 1,300 PR**(Steve Kaliski, How I AI): 손으로 쓴 줄은 없다. 트리거는 가볍다(Slack에서 봇 멘션 또는 이모지 반응). 신뢰성의 비결은 **모델이 깨어나기 전 구간** — 결정론적 오케스트레이터가 링크를 훑고 Jira를 당기고 Sourcegraph+MCP로 관련 코드를 찾아 컨텍스트를 조립한다. 규칙으로 풀 수 있는 일은 확률적 모델에 절대 넘기지 않는다. 놀랍게도 더 센 모델이 아니라 오픈소스 **Goose를 포크**한 것이고, 핵심 주장은 "**신뢰성은 모델 크기가 아니라 제약의 품질에서 온다**"는 것. 결정론적 게이트(린터 강제·커밋)와 창의적 LLM 단계가 맞물리고, 샌드박스는 "cattle not pets"로 수천 에이전트가 서로 안 밟는다. 그 1,300 PR도 **여전히 사람이 리뷰**한다 — 사람은 떠난 게 아니라 작성에서 리뷰로 자리를 옮겼다.

## 종료 조건과 리스크

**종료 조건**은 겹쳐 쓴다: 완료 프라미스(정해진 문자열 출력) · 최대 반복(`--max-iterations`, 무한 루프 1차 안전장치) · 결정적 종료(컴파일러·린터·테스트 통과) · 정체 서킷브레이커(N 반복 새 커밋 없으면 중단, 시간 상한) · 예산 상한(Claude Agent SDK `maxTurns`·`maxBudgetUsd`) · 반복 호출 감지(같은 행동 2회 초과 시 중단).

**루프맥싱(loopmaxxing)** — "무한 반복이 뭐든 풀어줄 것"이라는 함정. "UX를 개선하라" 같은 주관적 목표는 정지점이 없어 API 비용만 부푼다. 목표를 이진(pass/fail)으로 검증 가능한 한 문장으로("모든 테스트 통과", "린터 0 에러"). 그 밖에 **컴팩션 드리프트**(긴 실행 중 초기 지시가 요약되며 목표 이탈 — 지속 규칙은 `CLAUDE.md`/`AGENTS.md`에), **격리**(피처 브랜치·샌드박스·최소 권한·승인 게이트)도 기본 가드다.

## 4대 비용 — 조용히 쌓여 한꺼번에 청구되는 것

스스로 도는 루프는 스스로 실수하는 루프이기도 하다. 넷 다 도는 동안엔 경보가 없다가 함께 터진다. 서로를 강화하는 하나의 실패가 네 얼굴을 한 것이다.

- **검증 부채(verification debt)**: 열고 머지한 PR마다 아낀 시간이 "검증 안 된 출력"으로 쌓인다. "돈다(runs)"와 "맞다(right)" 사이, 테스트가 안 덮는 틈에 숨어 있다가 어느 배포 아침에 터진다. → 가드는 **독립 검증자**.
- **이해 부패(comprehension rot)**: 안 쓴 코드가 빨리 실릴수록, 존재하는 것과 내가 이해하는 것의 격차가 벌어진다. 읽기는 쓰기보다 지루한데 쓰기를 루프가 가져갔다. → 가드는 **매일 표본을 읽고 몇 개 변경을 스스로 설명해 보기**. 설명 못 하면 머릿속 지도가 뒤처진 신호.
- **인지적 항복(cognitive surrender)**: 루프가 잘 돌수록 의견 갖기를 그만두고 주는 대로 받게 된다. "시간이 없어서"가 아니라 "더는 신경 쓰기 싫어서". → 가드는 한 줄 — **루프는 실행하되 결정하지 못한다**. 최소한 "이건 틀렸다"고 말할 능력은 유지.
- **토큰 폭증(token blowout)**: 청구서에 직접 꽂히는 유일한 비용. 헬퍼를 까고 재시도하고 라운드를 거듭해, 버그 하나가 밤새 헛돌며 고친 코드 대신 낯선 청구서를 낳는다. → 가드는 **출시 전 하드 캡**(런당·일일 예산·최대 재시도).

넷의 공통점은 도는 동안의 침묵이다. 루프의 매력은 한 사람이 팀의 일을 하게 해주는 것이고, 위험은 같은 자리 — 팀은 서로 반박하지만 한 사람 + 루프 더미는 아무도 반박 않는 에코 챔버가 되기 쉽다.

## 언제 맞고 안 맞나

- **맞음**: 반복하면 수렴하고 **기계적으로 검증 가능한** 작업 — 마이그레이션·리팩터·의존성 업데이트·테스트 정리. "done"을 사전에 못박을 수 있고 그린필드(신규) 프로젝트에 강하다. 테스트 커버리지·문서화된 컨벤션·사람 감독이 있을 때.
- **안 맞음**: 레거시 코드베이스, 모호한 요구, 약한 테스트로 논리 오류를 못 잡는 경우, 창을 넘는 대형 스코프, 비즈니스 로직·UX 같은 주관적 목표.

## 컨텍스트·하네스와의 결합

- **컨텍스트 전략 두 갈래**: 누적형(한 세션에 쌓다가 한계 시 자동 compaction)과 리셋형(매 반복 메모리 리셋 — Ralph/자기개선 루프). 리셋형은 외부 메모리 4종(`AGENTS.md`·git 히스토리·`progress.txt`·`tasks.json`)으로 반복 간 학습을 복리로 쌓는다. → [[컨텍스트-엔지니어링]]
- **하네스 안의 루프, 루프 안의 팀**: [[하네스-엔지니어링]]이 도구·스킬·서브에이전트·훅의 구조를 정의하면, 루프가 그 구조를 시간축으로 구동하고, 한 반복 안에서 하네스가 서브에이전트 "팀"을 소집한다. 훅(`Stop`·`PreToolUse`)이 루프의 접합부다.

## 판단의 경제학과 규율

루프는 생성을 거의 공짜로 만든다 — 코드·계획·PR·수정이 흔해진다. 남는 희소 자원은 **판단**이다. 어느 계획이 맞는지, 어느 줄을 멈춰야 하는지, 어느 출력이 잘 돌지만 근본에서 틀렸는지. 루프는 백 개 후보를 뽑아도 "그럴듯함"으로 고르지 "진짜 맞음"으로 고르지 못하고, 그 둘의 간극이 엔지니어가 존재하는 이유다. 그래서 루프 엔지니어링은 판단을 평가절하하지 않는다 — 판단이 필요 없는 걸 다 걷어내고 판단만 남긴다.

- **루프는 충직한 곱셈기다.** 같은 루프를 두 사람이 지어도 반대 결과가 나온다 — 하나는 이미 통달한 일을 더 빨리 하려 루프를 쓰고(6개월 뒤 더 강해진다), 하나는 다시는 이해하지 않으려 쓴다(못 읽는 기계의 문지기가 된다). 코드는 90% 같아도 체크포인트 한둘이 6개월 뒤 누가 통제하는지를 가른다.
- 운영 규율 셋: **① 표본을 늘 읽어라**(이해 부패 방어 — 전부 말고 대표 표본을 매일, 설명해 보며). **② 출시 전에 상한을 걸어라**(토큰 폭증 방어 — 놀란 청구서 뒤가 아니라 첫 무인 실행 전에). **③ 문 하나는 열어 둬라**(인지적 항복 방어 — 사람이 멈출 수 있는 체크포인트를 최소 하나. 모든 문을 용접하면 들어가야 하는 날 열쇠가 없다).
- Osmani의 성숙 경로: ① 휴먼인더루프(모든 행동 승인) → ② 결정적 종료(컴파일러·린터·테스트) → ③ 정체 서킷브레이커(자동 종료) → ④ 반복 LLM 작업을 컴파일된 코드로 전환. **병렬화는 맨 마지막에** — 검증자가 진짜 실수를 잡는 걸 증명한 뒤에 에이전트 수를 늘린다.

한 줄로: 루프를 짓되, **그냥 시작 버튼 누르는 사람이 아니라 계속 엔지니어로 남을 사람처럼** 지어라.

## 다른 층과의 관계

프롬프트(한 번의 입력) → 컨텍스트(한 반복의 정보) → 하네스(환경) 위에서, 루프는 그 전부를 **반복으로 구동**한다. Boris Cherny(Anthropic)의 표현으로 "루프는 소스코드에서 에이전트로의 도약만큼 크다." 전체 지도는 [[ai-엔지니어링-4계층]].

## 실무 적용

루프 엔지니어링에서 가장 위험한 건 시작하기 쉽다는 점이다. `while` 한 줄이면 무한 루프가 돈다. 그래서 실무 규율은 "어떻게 돌리나"가 아니라 **"어떻게 멈추고 어떻게 검증하나"** 에 있다. 검증자 없는 루프는 그럴듯한 오류를 기계 속도로 쌓는 기계다.

- **언제 쓰나**: 반복하면 수렴하고 **이진(pass/fail)으로 검증 가능한** 작업 — 마이그레이션·리팩터·의존성 업데이트·테스트 정리. "done"을 사전에 못박을 수 있을 때. 반대로 레거시·모호한 요구·약한 테스트·UX 같은 주관적 목표엔 루프를 걸지 마라. "UX를 개선하라"는 정지점이 없어 API 비용만 부푸는 루프맥싱(loopmaxxing)의 전형이다.
- **도구에 붙이기**: 가장 단순한 건 Claude Code의 `ralph-wiggum` 플러그인(`/ralph-loop "<prompt>" --max-iterations <n> --completion-promise "<text>"`) — Stop 훅으로 종료를 가로채 같은 프롬프트를 되먹인다. 평가-수정 루프는 CI(린트·테스트·타입체크)를 정지 조건으로 물린다. 검증은 코드를 읽지 말고 **행동하게** — 프런트엔드면 Playwright MCP로 페이지를 열고 클릭하게 한다.
- **실패 패턴**: (1) 끄덕이는 루프 — 쓴 에이전트가 스스로 합격시킨다. 생성자와 검증자를 분리하고, 최종 판정은 별도 모델(maker–checker)에게 맡겨라. (2) 토큰 폭증 — 출시 전에 하드 캡(런당·일일 예산·최대 재시도)을 걸어라. 놀란 청구서 뒤가 아니라 첫 무인 실행 전에.

### 액션 아이템

- [ ] **오늘 30분**: 반복 작업 하나를 이진 완료 조건 한 문장("모든 테스트 통과", "린터 0 에러")으로 다시 써서 루프에 걸 수 있는 형태인지 판별한다.
- [ ] **이번 주**: `/ralph-loop`나 `/loop`로 검증 가능한 작업 하나를 자동화하되, 반드시 `--max-iterations`와 예산 상한을 먼저 건다.
- [ ] **이번 달**: 생성자와 독립 검증자를 분리한 평가-수정 루프를 구축하고, 병렬화는 검증자가 진짜 실수를 잡는 걸 증명한 뒤에만 켠다.
