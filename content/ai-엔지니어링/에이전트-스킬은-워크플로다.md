코딩 에이전트에게 팀의 개발 원칙을 길게 읽혀도, 막상 구현에 들어가면 명세와 테스트를 건너뛰기 쉽다. 문제는 모델이 원칙을 몰라서가 아니다. 원칙이 행동 순서와 완료 조건으로 바뀌지 않았기 때문이다.

## 핵심 주장

**Addy Osmani의 글에서 가장 가져갈 만한 생각은 “스킬은 참고 문서가 아니라, 증거를 남기며 끝나는 워크플로”라는 정의다.** 단계, 체크포인트, 종료 조건이 있는 스킬은 에이전트가 시니어 엔지니어의 작업 습관을 재현할 가능성을 높인다. 다만 스킬은 대개 모델이 읽는 지시이므로 그 자체가 강제 장치는 아니다. 실패 비용이 큰 규칙은 테스트, 훅, CI, 권한 경계 같은 결정론적 장치로 한 번 더 고정해야 한다.

## 글이 제안하는 구조

Osmani는 코딩 에이전트가 코드 diff에 드러나지 않는 일을 기본적으로 생략한다고 본다. 가정 드러내기, 명세 작성, 작은 변경 단위, 테스트, 리뷰, 출시 점검이 그 예다. `agent-skills` 프로젝트는 이를 개발 생명주기(SDLC)의 Define, Plan, Build, Verify, Review, Ship 단계에 대응하는 스킬로 만든다.

프로젝트가 내세우는 핵심 설계는 다섯 가지다.

| 원칙 | 의미 | 실무에서 확인할 증거 |
|---|---|---|
| 프로세스 우선 | 설명문 대신 실행 순서로 쓴다 | 단계별 산출물과 종료 조건 |
| 합리화 방지 | 생략할 때 나올 법한 핑계와 반박을 미리 적는다 | 예외를 택한 이유가 기록됨 |
| 검증 필수 | “그럴듯함”을 완료로 인정하지 않는다 | 테스트, 빌드, 런타임 로그, 리뷰 |
| 점진적 공개 | 필요한 스킬과 자료만 그때 읽는다 | 시작 컨텍스트는 메타데이터만 포함 |
| 범위 규율 | 요청받지 않은 인접 리팩터링을 막는다 | 작고 한 가지 목적을 가진 변경 |

Agent Skills 공개 규격도 이 방향과 맞는다. 최소 단위는 YAML frontmatter와 Markdown 본문을 가진 `SKILL.md`이며, `scripts/`, `references/`, `assets/`를 필요할 때만 읽도록 나눌 수 있다. 규격은 모든 스킬의 이름과 설명을 먼저 싣고, 선택된 스킬의 본문과 필요한 리소스를 차례로 불러오는 점진적 공개를 권한다.

## 무엇이 특히 설득력 있는가

### 1. 지식을 행동으로 컴파일한다

“테스트를 잘 작성하라”는 문장은 평가 기준이 없다. 반면 “실패하는 테스트 작성 → 실패 확인 → 최소 구현 → 통과 확인 → 리팩터링”은 실행할 수 있고 중간 결과를 검사할 수 있다. 스킬의 가치는 새로운 지식보다 **이미 아는 규율을 실행 가능한 순서로 바꾸는 데** 있다.

### 2. 종료 조건이 에이전트의 낙관적 완료 선언을 견제한다

에이전트는 결과를 생성하는 주체와 결과를 판정하는 주체가 같을 때 쉽게 “완료”를 선언한다. 스킬 끝에 구체적 증거를 요구하면 최소한 무엇을 확인해야 하는지가 명시된다. 이 구조는 [[루프-엔지니어링]]의 완료 조건과 직접 연결된다.

### 3. 컨텍스트 예산을 운영 단위로 다룬다

모든 규칙을 항상 넣으면 관련 없는 지시가 주의를 분산시킨다. 메타데이터로 라우팅하고, 선택된 본문과 참조 자료만 불러오는 구조는 [[컨텍스트-엔지니어링]]을 파일 체계로 구현한 셈이다. Agent Skills 규격은 본문을 활성화 시 전부 읽기 때문에, 핵심 절차는 짧게 두고 상세 자료를 별도 파일로 빼는 편이 낫다.

### 4. 팀의 반복되는 예외 논리를 자산으로 만든다

합리화 방지 표는 독특하고 실용적이다. “작은 변경이라 명세가 필요 없다” 같은 문장은 에이전트뿐 아니라 피곤한 사람도 쓴다. 반복되는 핑계와 반박을 기록하면 팀의 암묵적 품질 기준이 리뷰 가능한 텍스트가 된다.

## 검토하며 바로잡을 점

### 숫자는 프로젝트의 현재 상태가 아니다

원문은 2026년 5월 3일 스냅샷에서 20개 스킬, 7개 슬래시 명령, GitHub 별 2만 7천 개를 말한다. 2026년 7월 11일 저장소 README와 GitHub 화면에는 24개 스킬, 8개 명령, 약 7만 7천 개가 표시된다. 구조 설명은 유효하지만 수량은 빠르게 낡으므로 도입 문서에는 고정 사실처럼 옮기지 않는 편이 안전하다.

또 원문은 `80/15/5` 테스트 피라미드를 Google 관행의 예로 든다. Google Testing Blog가 공개한 대표적인 첫 추정치는 단위·통합·종단 간 테스트 `70/20/10`이며, 정확한 혼합은 팀마다 다르다고 명시한다. 따라서 `80/15/5`는 이 프로젝트가 채택한 규칙으로 읽어야지 Google의 보편적 공식 비율로 인용하면 안 된다.

### 스킬은 강제력이 아니라 준수 확률을 높이는 장치다

글의 “건너뛸 수 없게 만든다”는 표현은 목표로는 좋지만 기술적으로는 강하다. Markdown 지시는 모델이 무시하거나 잘못 해석할 수 있다. 테스트 명령을 실제로 실행하는 스크립트, 실패 시 중단하는 훅, 병합을 막는 CI, 쓰기 권한 제한이 있어야 규칙이 결정론적으로 집행된다. 스킬은 [[하네스-엔지니어링]]의 한 층이지 하네스 전체가 아니다.

### 이식 가능성과 동일 동작은 다르다

`SKILL.md`는 사람이 읽을 수 있는 Markdown이라 여러 도구로 옮기기 쉽다. 그러나 자동 발견, 라우팅, 도구 권한, 훅, 명령 체계는 런타임마다 다르다. 같은 파일을 복사했다고 같은 시점에 활성화되고 같은 강도로 집행된다고 보장할 수 없다. 이식 후에는 해당 도구에서 트리거와 종료 조건을 따로 평가해야 한다.

### 모든 일에 전체 SDLC를 씌우면 절차가 비용이 된다

작은 오탈자 수정에 긴 명세와 출시 체크리스트를 요구하면 리드타임과 토큰만 늘 수 있다. 원문도 작업 크기에 따라 활성화할 스킬 수가 달라져야 한다고 말한다. 중요한 것은 “항상 모든 절차”가 아니라 **위험과 변경 규모에 비례하는 최소 워크플로**다.

## 트레이드오프

| 얻는 것 | 치르는 비용 |
|---|---|
| 반복 가능한 개발 절차 | 스킬 작성·유지보수 비용 |
| 리뷰할 수 있는 중간 산출물 | 토큰 사용량과 작업 시간 증가 |
| 모델·도구를 넘나드는 공통 언어 | 런타임별 통합 차이 |
| 예외와 완료 기준의 명문화 | 오래된 규칙이 관성으로 남을 위험 |
| 에이전트의 범위 이탈 감소 | 과도한 규율이 탐색과 실험을 막을 수 있음 |

스킬이 잘 맞는 곳은 반복 작업, 실패 비용이 큰 변경, 여러 사람이 같은 품질 기준을 공유해야 하는 팀이다. 탐색적 프로토타입, 한 번만 하는 저위험 작업, 이미 강력한 자동화 게이트가 있는 단순 변경에는 가벼운 체크리스트가 더 나을 수 있다.

## 실무 적용

처음부터 스킬 묶음 전체를 설치하기보다 팀이 실제로 자주 겪는 실패 하나를 고르는 편이 낫다.

1. 최근 회고와 리뷰에서 반복된 생략을 하나 찾는다. 예: 테스트 없이 버그 수정, 요청 밖 리팩터링.
2. 참고 설명을 5단계 안팎의 행동 순서로 바꾼다.
3. 각 단계가 남길 증거와 최종 종료 조건을 적는다.
4. 자주 나오는 예외 논리와 허용 가능한 진짜 예외를 구분한다.
5. 반드시 지켜야 하는 조건은 스크립트·훅·CI로 옮긴다.
6. 대표 과업과 경계 사례로 스킬 활성화와 준수 여부를 평가한다.
7. 실패가 줄지 않거나 절차 비용이 더 크면 줄이거나 폐기한다.

좋은 스킬은 팀 핸드북의 축약본이 아니다. **작고, 상황이 명확하며, 실행 결과를 검사할 수 있는 운영 코드에 가까운 문서**다. 이 관점에서 `agent-skills`는 정답 묶음이라기보다 팀의 시니어 엔지니어링 습관을 어떻게 파일로 만들지 보여주는 좋은 참조 구현이다. [[github-spec-kit]]이 명세 산출물을 파일과 명령으로 고정한다면, Agent Skills는 SDLC 전반의 행동 순서를 재사용 가능한 모듈로 만든다.
