# wiki-verify — 발행 전 검증 프로토콜

발행 게이트. 4종 검증을 통과해야 위키에 올라간다. fact-checker(§1·§2)와 copy-editor(§3·§4)가 나눠 수행한다.

## §1 최신성 검증 (fact-checker)

초안 메타의 **작성일**을 기준으로:
1. 시간 민감 진술을 추출한다 — 버전 번호, 가격/한도, "최신/현재/최근" 수식, 벤치마크 수치, 제품 상태(beta/GA/deprecated), 날짜 언급.
2. 각각을 웹 검색으로 현재 상태와 대조한다.
3. 판정: 그대로 유효 / **낡음(수정 필수)** / 곧 낡을 위험(각주 권장).

## §2 팩트 검증 (fact-checker)

1. 검증 가능한 주장(수치·인과·비교·인용)을 추출한다. `(검증 필요)` 마커 최우선.
2. **반증 시도** — 주장을 반박하는 검색을 먼저 한다. 반박 실패 시 PASS.
3. 출처 등급: 공식 문서/논문(A) > 기술 블로그(B) > 커뮤니티(C). A·B로 확인 안 되면 UNVERIFIED.
4. 글의 `## 출처` 링크가 실제 주장을 지지하는지 표본 확인한다.
5. **수치는 전수 검사** — 본문의 모든 구체 수치(%, 배수, 개수, 날짜)는 추적 가능한 출처가 있어야 한다. 출처를 못 대는 수치는 **자동 UNVERIFIED**. (근거: 출처 불명 수치는 인터넷 반복 인용으로 살아남는다 — 실제로 CoT "17.7→78.7" 오류가 이 경로로 유입된 바 있음)
6. **경계선 판정 재확인** — 단일 실행 판정은 신뢰성을 과대평가한다(pass@1 문제). FAIL 여부가 애매한 항목은 그 항목만 관점을 바꿔 1회 독립 재확인 후 확정한다.

## §3 오탈자 검증 (copy-editor)

- 맞춤법·띄어쓰기·조사 호응·중복 단어·비문.
- 기술 표기 일관성: 같은 용어가 글 안에서 표기 통일(예: "쿠버네티스" vs "k8s" 혼용 금지, 첫 병기 후 하나로).
- 코드 블록·명령어는 실행 가능성까지는 안 보되, 명백한 오타(따옴표 짝, 백틱 누락)는 잡는다.

## §4 윤문 검증 (copy-editor)

- **humanize-korean 스킬로 AI 티 패턴을 탐지·윤문한다** (번역투, 기계적 병렬, 피동 남용, 접속사 남발, 리듬 균일 등).
- 철칙: 의미·사실·수치·인용 불변. frontmatter·[[위키링크]]·코드 블록 불가침.
- 과윤문 방지: 변경률 30% 초과 시 재검토.

## 리포트 형식 (공통)

`_workspace/02_{factcheck|editing}_report.md`:

```markdown
# 검증 리포트 — {팩트/최신성 | 오탈자/윤문}
검증일: YYYY-MM-DD / 대상: 01_writer_draft.md

## 판정: PASS | FAIL | PASS-WITH-NOTES

## 지적 사항
| # | 위치(인용) | 문제 | 심각도 | 수정 제안 | 근거(URL) |
|---|---|---|---|---|---|

## UNVERIFIED (판단 보류)
- ...
```

- **FAIL 기준**: 낡은 내용 1건 이상, 반박된 팩트 1건 이상, 의미 훼손 윤문 발견.
- **PASS-WITH-NOTES**: 발행 가능하나 각주·보강 권장 사항 있음.
- 심각도: 🔴 발행 차단 / 🟡 수정 권장 / ⚪ 참고.

## 수정 루프 규칙

- FAIL 리포트는 오케스트레이터가 tech-writer에 전달 → 수정 → **FAIL 항목만 재검증**(전체 재검증 아님).
- 루프는 최대 2회. 2회 후에도 FAIL이면 발행 중단하고 사용자에게 쟁점을 보고한다.
- **UNVERIFIED 발행 규칙**: 확인 안 된 **수치·구체 주장**은 그대로 발행하지 않는다 — 출처를 확정해 명기하거나, 수치를 빼고 방향성 서술로 완화하거나, 해당 문장을 삭제한다. UNVERIFIED가 저위험 트리비아(예: 업데이트 횟수)면 오케스트레이터 재량으로 유지 가능하되 리포트에 남긴다.
