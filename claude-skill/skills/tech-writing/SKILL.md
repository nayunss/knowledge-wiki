# tech-writing — 기술 글 작성 가이드

knowledge-wiki에 올라가는 모든 기술 글의 작성 표준. "누가"는 tech-writer 에이전트, "어떻게"가 이 문서다.

## 1. 글의 뼈대 (순서대로)

1. **후킹 도입 (2~4문장)** — 독자가 겪는 문제 또는 뒤집히는 통념으로 시작한다. "X란 무엇인가" 사전식 도입 금지.
2. **핵심 주장 선언 (두괄식)** — 이 글이 말하려는 것 한 문단. 독자가 여기까지만 읽어도 결론을 안다.
3. **본문** — 주장을 지탱하는 섹션들. 각 섹션도 두괄식. 비교는 표, 절차는 번호 목록, 구조는 코드블록/다이어그램.
4. **트레이드오프 섹션 (필수)** — 한계, 안 맞는 경우, 대안. 이 섹션이 없으면 미완성 글이다.
5. **실무 적용 (권장)** — "그래서 내일 뭘 하면 되나". CTO 시각: 도입 판단 기준, 비용, 팀 역량 요구.
6. **관련/출처** — 맨 아래 `관련:` 위키링크 줄 + `## 출처` 섹션.

## 2. 문장 규칙

- 한 문단 = 한 생각. 5문장 넘으면 쪼갠다.
- 문장 길이에 리듬을 준다 — 긴 문장 뒤엔 짧은 문장.
- 수식어보다 구체 수치·사례. "매우 빠르다" 대신 "p99 12ms".
- **네가 직접 센 수치엔 집계 방법을 함께 남겨라.** 개수·바이트·빈도처럼 손으로 잰 값은 재현 명령이나 집계 범위를 본문 표나 `## 출처`에 적는다 — 예: ``grep -o 'data-lang="css"' index.html | wc -l`` → 198. 몇 주 뒤 그 수치가 낡았을 때, 방법이 있으면 같은 자로 다시 재고 없으면 못 잰다. **실측**: kinetics 편은 집계 범위("이펙트 CSS 세 파일의 transition·animation 선언 내")를 적어둔 덕에 사이트가 99 → 117개로 개편된 뒤에도 전 수치를 재집계할 수 있었다. 같은 글에서 방법을 안 적은 수치는 세는 사람마다 값이 갈려(`ease` 82 vs 89 vs 101) 재현에 실패했다.
  - 1차 출처가 발표한 수치(논문의 p값, 벤더의 벤치마크)엔 해당 없다 — 그건 출처 링크가 곧 방법이다. **네가 센 것에만** 붙인다.
- 용어 첫 등장 시 한글(영문) 병기 후 한 줄 설명. 이후 반복 설명 금지.
- 능동태 기본. 번역투("~에 의해", "~되어진다") 금지.

## 3. 설득의 구조

- **주장 → 근거 → 예시** 3박자. 근거 없는 주장은 쓰지 않거나 "추정"으로 명시.
- **두 사실을 잇는 문장은 그 연결의 근거를 대라.** 수치가 각각 맞아도 사이의 **인과·비교·일반화**는 별개 주장이다. 원문이 "A라서 B"라고 했나, 아니면 A와 B를 따로 말했나? 좁은 사실 하나에서 넓은 결론으로 건너뛰지 않았나? 특히 **척도가 다른 수치는 비교하지 마라** — 다른 벤치마크·다른 통계 처리는 변환 정의가 없으면 대소 자체가 성립하지 않는다. 연결의 근거를 원문 문장으로 못 대면 쓰지 말거나 `(검증 필요)`.
  - 이건 몸을 사리라는 말이 **아니다.** 해석은 이 위키의 값이다 — 날 선 주장을 쓰되, 그 주장이 딛는 연결이 원문에 있는지만 확인하라. 겨냥 지점은 "두 사실이 만나는 자리" 하나다.
- 확신이 낮은 대목엔 `(검증 필요)` 마커를 남긴다 — fact-checker가 우선 검증한다.
- 반론을 선제 처리한다: "~라고 생각할 수 있다. 하지만 ...".

## 4. OKF 형식 (필수)

```yaml
---
title: 표시 제목
type: 개념 | 도구 | 설계결정 | 레퍼런스 | 플레이북
description: 한 줄 요약 (index·검색·RAG에 쓰임)
tags: [소문자, 2~4개]
resource: https://…   # 특정 도구/레포/API를 다룰 때만
---
```

- 관련 노트는 본문 문맥 속 `[[위키링크]]`로. 없는 노트 링크 허용(stub).
- 외부 근거는 맨 아래 `## 출처`에 목록으로 — 1차 출처(공식 문서·논문) 우선.
- 파일명: kebab-case + `.md`. 한 파일 = 한 주제.
- AI 엔지니어링 주제는 `content/ai-엔지니어링/` 폴더에, 그 외는 `content/` 루트에.

## 5. 자기 점검 (제출 전)

- [ ] 대상 독자 한 줄이 메타 주석에 있는가
- [ ] 결론이 서두에 있는가
- [ ] 모든 주장에 근거 또는 `(검증 필요)` 마커가 있는가
- [ ] 두 사실을 잇는 문장(인과·비교·일반화)마다 그 **연결**의 근거가 있는가 — 양 끝 수치의 근거 말고. 표에 척도가 다른 값을 한 열에 세우지 않았는가
- [ ] 직접 센 수치마다 집계 명령·범위가 붙어 있는가 — 몇 주 뒤 남이 같은 값을 다시 낼 수 있는가
- [ ] 트레이드오프 섹션이 있는가
- [ ] OKF frontmatter 5필드 점검, `## 출처` 존재
- [ ] 관련 위키 노트 1개 이상 [[링크]]했는가
